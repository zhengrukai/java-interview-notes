# **一、积分系统设计**

关键点：高并发、综合稳定性、性能考虑，结合外部系统接入，是否同时提供同步、异步方式，ticketid串联？

### **组织回答：**

我的整体思路是：同步校验-异步履约-状态机对账，这样的一个思路

首先，（在性能上）我一般为了应对高并发，会在Redis中维护一份热点数据（用户（活跃）积分和奖品库存），利用Lua脚本确保原子性（1ms内），防止超卖。Lua脚本执行：幂等校验、查询用户积分是否足够、权益库存是否足够、扣除用户积分、扣减权益库存、记录防重幂等标识TicketID。Redis会将整个脚本作为一个独立的原子命令执行，执行期间不会被其他请求插入，从而从物理层面彻底杜绝超卖。然后立即返回TicketID给前端，告知用户"兑换处理中"，这样用户体感会好点。此时状态机为\[处理中]。

然后，（在稳定性上）系统通过RocketMQ发送TicketID触发异步履约逻辑，对接积分系统和权益系统（外部？如京东、话费接口），实现解耦。此时前端则拿着TicketID通过轮询或者websocket实时查询最终兑换状态。而MQ的逻辑是使用RocketMQ事务消息，先发送半消息，积分数据库操作成功后再提交消息。这里就要数据库记录状态机为\[已扣分]，TicketID作为主键或者唯一索引。履约服务监听到消息，拿TicketID去数据库查询兑换信息，如果是外部系统，我会用适配器模式，针对不同的供应商封装标准接入层，将异构接口转化为内部标准协议。带着TicketID作为幂等键调用外部接口，根据返回（成功/失败/返回），更新状态机。

如果履约失败，则走自动补偿逻辑：1、通过ticketID查询预扣记录；2、发起反向操作，积分退回Redis和Mysql；3、状态机更新为\[退款]

如果是网络抖动的超时，一般不走立即退款，而是走人工挂起，或者走自动重试链路

针对外部系统的不确定性，我们还可以做一些容错设计。比如离线对账：每天凌晨与供应商进行T+1对账，拉取外部系统的履约流水，与本地数据库进行比对，发现漏单或错单自动挂起人工处理。

然后针对极端场景下，做一些数据安全的措施。比如风控拦截：在积分扣减前接入风控引擎，识别短时间高频兑换、异常账户（灰产），并设置单人单日兑换上限。

根据流水记账，采用“复式记账法”思想，每一笔积分变动必须有对应的业务单据关联，支持全链路溯源。

在我这个系统中呢，我最核心的设计是这个“状态机”，每一个兑换对应多个状态，即使系统中途重启，我也能通过扫描异常状态，自动恢复执行环境，极大降低人工接入的运维成本！

### **扩展提问：如何支撑5000QPS？**

在网关层保护，压力留在缓存层，处理留在异步层

### **组织回答**：

首先，网关限流（sentinel）,设置单用户限流，同一用户1s只能请求1次，超出直接返回“业务繁忙”，然后判断活动时间，以及库存是否为0，设为标志位，后续请求直接拦截不走Redis缓存层。

接下来5000QPS，靠Redis。首先就是上面说的Redis+Lua脚本原子扣减，然后是热点key，使用分段库存，将5000QPS分散到10个不同的Redis Slot，相当于单Key压力降到500QPS，确保Redis响应耗时在1-2ms。Lua脚本执行成功后，并返回给前端。

然后是异步层，消息队列削峰填谷。积分预扣成功后，将兑换请求发入RocketMQ。利用多线程，DB写入控制在500-1000QPS左右平滑写入MySQL，避免数据库连接池耗尽。外部调用线程，根据三方接口的限额，动态调整消费并发度。

本地事务尽可能简单：只记录扣分，状态机，订单。RokectMQ的事务消息，保证积分扣除和发送履约消息是原子操作。

可以做个对账补偿：后台任务定时对比 Redis 预扣数与 MySQL 成功订单数。若发现 Redis 扣了但 MySQL 没记录，根据流水重试或回滚

根据外部系统可以封装一个Adapter模块，使用Sentinel对外部接口进行熔断。如果对方接口响应时间从200ms飙升至2s，立即熔断，状态机设置\[稍后重试]。后续可以用MQ的延迟队列，用退避算法（10s, 30s, 1m）进行三次重试，最终失败，走回退流程。

#### 电商平台（运营中台）：大促期间 Redis 热 Key 导致缓存击穿

*   **场景背景**：在一次对接抖音直播间的促销活动中，某核心商品（爆款秒杀）瞬时并发 QPS 达到 8000+。
*   **故障现象**：监控报警显示 Redis 集群某个分片 CPU 飙升至 90% 以上，该 Key 的响应延迟从毫秒级增加到秒级，部分请求开始穿透到数据库，导致核心数据库连接池满，系统出现大面积 504 错误。
*   **分析过程**：通过 SkyWalking 链路追踪发现，大量请求集中在同一个商品的库存和优惠规则查询上，触发了典型的“热 Key”问题。由于使用了分片集群，单点压力无法通过扩容集群缓解。
*   **解决方案**：

    1.  **紧急处理**：立即通过 Sentinel 开启热点参数限流，对该商品 ID 的访问进行强制排队和降级。
    2.  **多级缓存改造**：引入 **Caffeine 本地缓存**。在应用层对这类热点 Key 设置短期的本地缓存（如 5-10 秒），通过 Redis + Caffeine 形成二级缓存架构，大量请求在应用内存命中，直接卸载 Redis 压力。
    3.  **Key 散列（备份 Key）**：针对该热 Key，在 Redis 中存储多个备份（如 key\_1, key\_2），读取时随机加随机数后缀，将压力分散到不同分片。
*   **最终结果**：改造后成功支撑了后续多场直播带货，DB 压力降低 80%，系统可用性维持在 99.9% 以上。
