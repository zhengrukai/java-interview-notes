在我们的架构设计中，MySQL 是最核心的资产。如果它发生宕机，我会从实时**高可用、业务保护、数据补偿**三个层面来处理：

### 1. 实时高可用：主从切换（硬件/架构层）

首先，我们的 MySQL 并不是单机，而是采用 主从架构（Master-Slave）。

*   **故障转移**： 当主库（Master）宕机时，我们的高可用组件（如 MHA 或中间件）会自动将流量切换到从库（Slave），并将其提升为新主库。
*   **配置更新**： 整个过程大约在 30 秒内完成。在此期间，Skywalking 会监控到大量的数据库连接错误告警。

### 2. 写操作保护：利用 MQ 充当“蓄水池”（核心思路）

这是我们之前做异步化改造最大的优势。

*   **前端不受影响**： 当用户点击‘兑换图书’时，由于我们使用了 Redis 预扣减 + MQ 异步化，请求在 Redis 层面就已经处理完并返回‘兑换成功’了。
*   **MQ 自动积压**： MySQL 宕机后，MQ 的消费端在尝试‘图书订单入库’时会不断报错并触发重试。此时，消息会暂时积压在 MQ 集群中。
*   **意义**： 只要 MQ 没挂，用户的兑换请求就不会丢。对于用户来说，他们看到的只是‘订单处理中’，而不会感知到后端数据库已经崩了。这为我们维修数据库争取了极大的缓冲时间。

### 3. 读操作处理：多级缓存降级（保护）

如果用户想看‘我的订单’或者‘图书列表’，而此时数据库没法查询：

*   **本地缓存兜底**： 像图书详情这类数据，我们有 Caffeine（或 ConcurrentHashMap）本地缓存 挡着，所以用户依然能正常浏览图书。
*   **查询降级**： 针对‘我的订单’查询，如果数据库挂了，我会通过代码逻辑进行降级处理。比如显示一个温馨提示：‘系统维护中，订单稍后同步’。

### 4. 数据恢复与最终一致性（事后）

当 MySQL 恢复后，重头戏是处理积压的数据：

*   **削峰写入**： 数据库重启后，我们不会立刻让 MQ 全速消费，而是通过控制消费端的并发数，让积压的消息平稳地、小批量地写入数据库，避免重启瞬间把数据库再次压垮。
*   **对账校验**： 恢复后，后台脚本会对比 Redis 中的预扣流水和数据库中的实际订单。如果发现某笔 Redis 扣减没有对应的数据库订单，会根据 MQ 里的死信队列进行重补。

面试官可能追问的细节（帮你守住底线）：

1.  问：如果主从切换时，有部分数据没同步到从库，数据丢了怎么办？
    答： “这涉及 MySQL 的半同步复制。如果真的发生了极小概率的数据丢失，我们还有 Redis 的操作流水日志 以及 MQ 的消息持久化。我们可以通过解析 MQ 里的消息，与数据库进行比对，手动或脚本重补这部分丢失的订单。”
2.  问：MQ 消息是有积压上限的，如果数据库修了 2 个小时还没好，MQ 爆了怎么办？
    答： “首先，我们会监控 MQ 的堆积情况。如果堆积量超过安全水位，我们会紧急触发前端限流，暂停新的兑换请求。其次，我们会将 MQ 中的消息转存到日志文件或临时的对象存储中，腾出空间让系统继续运行。总之，原则是：先保住数据，再谈业务处理。”
3.  问：Skywalking 在这个过程中起什么作用？
    答： “非常关键。在 MySQL 挂掉的一瞬间，Skywalking 的拓扑图里指向数据库的线会变红，且 Error 报表 会疯狂报错。它能帮我瞬间判断是‘个别 SQL 慢’还是‘数据库整体宕机’。通过查看 Trace 日志，我能确定消费端是不是在不断重试，从而决定是否需要手动关闭消费端来保护数据库。”
4.  问：为什么不直接让应用连接从库进行写操作？
    答： “不行，从库一般只读。强行写会导致数据冲突和同步链路断裂。在高并发下，保持\*\*‘读写分离’\*\*的规范是保证数据不乱的前提。我们宁可让消息积压在 MQ，也不要在数据一致性上冒险。”
