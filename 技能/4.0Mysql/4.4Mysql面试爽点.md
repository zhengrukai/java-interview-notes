我遇到的最大核心问题是**1. 高并发下的数据库性能瓶颈**、**2. 多业务场景的数据一致性问题**，以及**3. 大表查询与分库分表带来的关联查询难题**，这三类问题在电商、仓储这类写多读多的场景中尤为突出，也是我在项目中重点攻坚并落地解决方案的核心点，接下来我从问题发现、落地解决和实际效果三方面具体说明：

在**问题发现**上，我不会等业务出现卡顿或报错才介入，而是通过**全链路监控** + **数据库自身指标** + **业务日志分析**的三维方式提前预警和精准定位，这也是我做高可用系统的核心思路。

第一，用 SkyWalking 做全链路监控，搭配 Prometheus 监控 MySQL 核心指标，重点关注 QPS/TPS 突增、慢查询数飙升、连接池耗尽、InnoDB 行锁等待时间过长这些信号，比如电商中台订单峰值时，发现行锁等待时间超 200ms，WMS 系统大表查询的 99 分位延迟超 1s；

第二，利用 MySQL 自身工具，开启慢查询日志（设置 long\_query\_time=1s），通过 explain 分析执行计划，发现大量 SQL 未走索引、大表做全表扫描，还有仓储系统多表关联时的笛卡尔积查询；

第三，结合 ELK 技术栈分析业务日志，发现电商下单、仓储库存扣减时出现部分事务回滚、数据写入失败，CRM 系统客户大表分页查询接口频繁超时，这些现象直接定位到性能、一致性、大表查询这三大核心问题。同时我也会关注数据库底层，比如 InnoDB 的缓冲池命中率、磁盘 IO 利用率，判断是内存瓶颈还是 IO 瓶颈。


针对这些问题，我结合电商、仓储、CRM 的不同业务场景，落地了定制化的解决方案，而非单纯的通用优化，核心从SQL 与索引优化、事务与锁机制优化、大表与分库分表设计、架构层面的读写分离四个维度解决，都是项目中实际落地并验证有效的：


第一，针对SQL 与索引性能瓶颈，这是最基础也是最核心的优化。比如 WMS 仓储系统中，货品多单位关联查询的 SQL 因未做合理索引导致全表扫描，我通过 explain 分析后，为联合查询字段建立复合索引，同时删除冗余单值索引，避免索引失效；电商中台的优惠规则计算 SQL 因多表关联过于复杂，我采用子查询优化 + 临时表的方式，拆分复杂查询，将优惠规则的基础数据提前缓存到 Redis，减少数据库查询次数；CRM 系统客户大表的分页查询，用主键 ID 做分页锚点，替代传统的 limit offset，解决大偏移量分页的性能问题，查询效率直接提升 80% 以上。


第二，针对高并发下的事务与数据一致性问题，电商下单、库存扣减这类场景最易出现行锁竞争、超卖、事务回滚。我采用乐观锁替代悲观锁，比如 WMS 库存扣减时，为库存表增加版本号字段，更新时通过版本号做判断，避免行锁竞争导致的阻塞；电商中台的分布式事务场景（比如订单创建 + 库存扣减 + 佣金计算），基于 Seata 实现 TCC 模式的分布式事务，保证跨服务、跨库的数据一致性；同时优化事务范围，将非核心操作（如日志记录、消息推送）移出事务，缩短事务执行时间，减少锁持有时间，解决行锁等待过长的问题。


第三，针对大表查询与分库分表难题，电商中台的订单表、CRM 的客户表随着业务发展数据量超千万，单表查询性能急剧下降。我采用垂直分表 + 水平分库分表的方式，比如将订单表拆分为订单主表（存储核心订单信息）和订单扩展表（存储非高频查询的物流、备注信息），解决单表字段过多的问题；对订单主表按订单创建时间 + 用户 ID做水平分库分表，基于 Sharding-JDBC 实现分库分表的透明化访问，同时为分表键建立索引，保证查询时能精准路由到对应分库分表；对于仓储系统的货品表，因数据量未达分库分表级别，采用分区表设计，按货品类目做范围分区，提升查询和归档效率。


第四，针对高并发读写压力，从架构层面做读写分离，搭建一主多从的 MySQL 集群，主库负责写操作，从库负责读操作（如报表查询、数据统计、历史订单查询），通过 MyCat 实现读写分离的自动路由；同时将高频访问的热点数据（如电商商品信息、仓储库存数据、CRM 客户核心信息）缓存到 Redis，实现 “缓存穿透 + 缓存击穿 + 缓存雪崩” 的三重防护，减少数据库的读请求压力，让数据库专注于核心写操作。


最后说下这些优化方案落地后的实际业务效果：电商中台在峰值 QPS5000 + 的订单场景下，SQL 执行效率提升 90%，事务执行时间从平均 300ms 降至 50ms 内，无任何锁阻塞和数据一致性问题，系统可用性保持 99.9%；WMS 仓储系统的库存表查询和更新效率提升 85%，库存盘点准确率从 75% 提升至 99.8%，磁盘 IO 利用率从 90% 降至 30%；CRM 系统的客户大表分页查询接口响应时间从 1.5s 降至 100ms 内，支撑 5 万 + 客户数据的高频查询和操作。
总结下来，我觉得 MySQL 的问题解决核心是“业务驱动技术优化”，不能脱离业务谈纯技术优化，比如电商的高并发写、仓储的实时库存同步、CRM 的大表查询，场景不同优化思路完全不同。同时要做好“提前预防 + 精准定位 + 分层优化”，从 SQL、索引、锁机制到架构层面的读写分离、分库分表，层层递进解决问题，再配套完善的监控告警体系，才能让 MySQL 在高并发、大数据量的业务场景中稳定支撑系统运行。
