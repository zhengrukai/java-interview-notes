### redo log（重做日志）- 保证「持久性」+ 崩溃恢复

**核心定义**：
redo log 是 InnoDB 自带的「物理日志」，记录的是：数据库「页」的修改内容（比如哪个数据页、哪个位置、修改成了什么值），不是记录 SQL 语句。
redo log 的核心作用：解决「事务提交后，数据还没刷到磁盘，MySQL 崩溃丢失数据」的问题，保证事务的「持久性」。
**为什么需要 redo log？**
MySQL 的数据是存在磁盘的，表数据、索引都是磁盘文件，而磁盘的「写入速度」比内存慢几万倍。如果每次执行 SQL 都直接把数据刷到磁盘，那 MySQL 的并发性能会极低；所以 InnoDB 做了优化：所有数据修改，先写入「内存缓冲池」，再异步刷到磁盘。但这个优化有个致命问题：事务 commit 提交后，数据还在内存，没刷到磁盘，此时 MySQL 崩溃，内存数据丢失，事务的持久性就没了。

**redo log 的工作机制**
redo log 是 「循环写、持久化」的日志文件，大小固定，InnoDB 会保证：事务执行的所有修改，都会先写入 redo log，再写入内存缓冲池，这个规则叫 WAL（Write-Ahead Log）预写日志，是数据库的通用设计，面试说出来绝对加分。

1.  事务执行 SQL，修改数据 → 先写 redo log（标记为「prepare 准备状态」）；
2.  数据写入内存缓冲池，事务执行完成 → 执行 commit 提交；
3.  redo log 标记为「commit 提交状态」，后续 InnoDB 后台线程异步把内存数据刷到磁盘；
4.  极端情况：commit 后 MySQL 崩溃 → 重启时，InnoDB 会读取 redo log，把「已经 commit 但没刷到磁盘」的修改，重新刷到磁盘，数据不会丢失，完美保证「持久性」。

> redo log 保证：事务一旦提交，修改的数据永久有效，即使 MySQL 崩溃也能恢复，核心解决「持久性」问题。
