### InnoDB引擎的索引底层原理：B+树

*   **B + 树的 3 个核心特性**

1.  **树是「平衡」的**：树的高度永远在 3~4 层，哪怕你的表有千万级数据，通过索引查询也只需要「3~4 次磁盘 IO」就能找到数据，磁盘 IO 是数据库查询的最大开销，这是索引快的核心原因
2.  **数据只存在「叶子节点」**：B + 树的非叶子节点，只存「索引字段的值」，不存真实的业务数据；只有最底层的叶子节点，才会存「索引字段 + 整行业务数据」；
3.  **叶子节点是「双向有序链表」**：所有叶子节点按索引字段排好序，还互相串联，支持「范围查询」（比如查你订单表「近 30 天的订单」），这是 MySQL 支持 between/and/order by 高效查询的核心。

*   **面试必问：为什么 MySQL 索引不用二叉树 / 红黑树 / B 树，偏偏用 B + 树？（标准答案，背熟）**

> 答：① 二叉树 / 红黑树是「二叉」的，数据量大时树的高度会变成几十层，磁盘 IO 次数多，查询慢；② B 树的非叶子节点存真实数据，一页磁盘能存的索引数少，树的高度还是高；③ B + 树完美解决以上问题：高度低（3-4 层）、磁盘 IO 少、范围查询快，完全适配 MySQL 的「磁盘存储」特性，是最优解。

### InnoDB 的索引，分「聚簇索引」和「辅助索引」，没有第三种！

*   **聚簇索引（也叫「主键索引」）- 天生自带、最重要、查询最快**

1.  InnoDB 引擎中，每张表必须有且只有 1 个聚簇索引，如果你的表没手动设置主键，MySQL 会自动生成一个隐藏的主键作为聚簇索引；
2.  聚簇索引的 B + 树，叶子节点直接存「整行的业务数据」；
3.  聚簇索引的查询是「直达查询」，不用回表，查询速度是所有索引里最快的。

*   **辅助索引（也叫「二级索引 / 普通索引」）- 手动创建、用的最多、面试必考**

1.  除了主键索引，你手动给任何字段创建的索引，都是「辅助索引」（比如给订单表的user\_id、create\_time、库存表的goods\_id创建的索引）；
2.  辅助索引的 B + 树，非叶子节点 + 叶子节点，都只存「索引字段的值 + 主键的值」，不存整行业务数据；
3.  辅助索引查询，会触发一个关键动作：回表查询（面试必考！！）。

*   **什么是「回表查询」**

> 回表查询 = 辅助索引查询的「两步走」：
> 第一步：通过辅助索引的 B + 树，找到「索引字段对应的主键值」；
> 第二步：拿着这个主键值，去「聚簇索引」的 B + 树里，查找到整行的业务数据。

*   **什么是「覆盖索引」？（解决回表的最优方案）**
    覆盖索引 = 查询的所有字段，都能在「辅助索引」里找到，不用回表
*   **联合索引（复合索引）**
    联合索引 = 给多个字段一起创建的索引。联合索引的底层还是 B + 树，排序规则是：先按第一个字段排序，第一个字段相同，再按第二个字段排序，以此类推。
    **最左前缀匹配原则**
*   **索引失效的 8 个高频场景**

1.  违反最左前缀原则→ 联合索引没带最左字段，比如idx(user\_id,status)，只查status=1；
2.  查询条件对索引字段做了「函数 / 运算」→ 比如where substr(mobile,1,3)='138'、where age+1=30，索引直接失效；
3.  查询条件用了「!= / <> /is null /is not null」→ 索引失效，变成全表扫描；
4.  查询条件用了「like % xxx」（左模糊）→ 比如where name like '%张三'，索引失效；✅ 优化：like '张三%'（右模糊）索引生效；
5.  查询条件用了「or」连接非索引字段→ 比如where user\_id=1001 or name='张三'，如果 name 没索引，整个查询索引失效；
6.  字段类型不匹配 → 比如你的mobile是 varchar 类型，查询用where mobile=13800138000（数字），索引失效；
7.  联合索引中，中间字段用了范围查询（> / < /between）\*\* → 比如idx(a,b,c)，where a=1 and b>2 and c=3，c 字段的索引失效；
8.  MySQL 认为「全表扫描比走索引更快」 → 比如表只有 100 条数据，MySQL 会直接全表扫描，不用索引
