Elasticsearch（简称 ES）是基于 Lucene 内核构建的分布式、近实时、高可用的全文搜索引擎与数据分析引擎，核心围绕分布式分片存储、读写分离、数据可靠性、高性能检索设计。

### 一、核心基础概念

1.  **集群 (Cluster)**：多个 ES 节点组成的分布式服务集群，对外统一提供服务，有唯一的集群名称。
2.  **节点 (Node)**：单个 ES 服务实例，按角色分为核心三类：
    *   **Master 节点**：集群管理者，负责索引元数据维护、分片主副本选举、集群状态管理、节点上下线，默认不处理用户读写请求
    *   **Data 节点**：数据存储节点，承载分片数据，处理文档的读写、检索、聚合计算，图中的 slave 节点本质是 Data 节点。
    *   **协调节点 (Coordinating Node)**：接收客户端请求，负责请求路由、结果汇总与返回，集群中任意节点都可充当协调节
3.  **索引 (Index)**：一类相同结构文档的集合，类比关系型数据库的「数据库」。
4.  **分片 (Shard)**：索引的水平拆分单元，是 ES 分布式的核心。一个索引会拆分为多个分片，均匀分布在不同 Data 节点，解决单节点的存储上限与性能瓶颈。
    *   **主分片 (Primary Shard)**：文档写入的唯一权威节点，所有写入操作必须最终落地到主分片，由主分片确定文档版本号、完成写入后，再同步给副本分片。
    *   **副本分片 (Replica Shard)**：主分片的全量冗余副本，与主分片数据强一致。核心作用：① 高可用容灾，主分片宕机后，副本会被选举为新的主分片；② 读负载分担，读请求可路由到副本，大幅提升读吞吐量。
5.  **Segment**：Lucene 的最小存储单元，是只读的倒排索引文件，ES 的检索、写入优化都围绕 Segment 展开。
6.  **Translog**：事务日志，类比 MySQL 的 Redo Log，核心作用是保证宕机数据不丢失，实现数据持久化。

### 二、ES 整体架构原理

ES 的架构核心是**分布式分片存储 + 中心化元数据管理 + 主副同步容灾**，核心逻辑如下：

1.  **分布式分片分布规则**
    一个索引的多个主分片，会均匀分布在集群不同 Data 节点；同一份分片的主分片和副本分片，绝对不会存放在同一个节点（否则节点宕机，主副同时丢失，失去容灾意义）。
    比如索引有 3 个主分片、1 个副本，共 6 个分片，会均匀分散在集群多个节点，避免单点压力。
2.  **集群元数据管理**
    Master 节点维护全集群的核心元数据：索引列表、每个索引的分片数 / 副本数、分片所在节点、集群节点列表等。集群所有节点都会同步一份元数据副本，保证任意节点都能独立完成请求路由。
3.  **主副数据同步**
    主分片完成写入后，会将数据操作同步给对应的所有副本分片，保证主副数据最终一致，图中 “同步数据” 箭头就是该过程。
4.  **客户端请求路由**
    客户端可连接集群任意节点，该节点自动成为本次请求的协调节点；协调节点根据集群元数据，计算出文档 / 查询对应的分片，将请求精准路由到目标节点，无需客户端感知分片分布。

### 三、ES 写入 / 更新 / 删除流程原理

ES 写入采用**追加写 + 延迟提交 + 异步合并**的设计，平衡写入性能、数据可靠性、近实时搜索能力，同时适配 Lucene Segment 只读的特性。

#### 完整写入全流程

1.  **请求路由**：客户端发起写入请求，协调节点接收后，根据文档 ID 计算出对应的主分片，将请求转发到主分片所在节点。
2.  **内存双写**：主分片节点校验请求合法性后，同时执行两个操作：
    *   将文档写入**内存 Buffer（内存缓冲区）**；
    *   将本次写入操作记录写入 **Translog 的 OS Cache（操作系统页缓存）**。
3.  **Refresh 流程（近实时搜索的核心）**
    *   触发时机：默认**每 1 秒执行一次**，也可手动调用 API 触发。
    *   执行逻辑：将内存 Buffer 里的文档，写入一个新的 Segment 文件，并直接放到 OS Cache 中，同时清空内存 Buffer。
    *   核心特性：Segment 一旦进入 OS Cache，就可以被检索到！这就是 ES\*\* 近实时（NRT）\*\* 的根源：写入后 1 秒即可被搜索，而非实时。手动 refresh 会频繁生成小 Segment，严重影响性能，非必要不使用。
4.  **Translog 刷盘（数据可靠性保障）**
    *   触发时机：默认**每 5 秒执行一次 fsync 操作**。
    *   执行逻辑：将 OS Cache 中的 Translog 数据，强制刷入磁盘的 Translog 日志文件，完成持久化。
    *   核心特性：ES 默认最多丢失 5 秒的写入数据；若需极致数据安全，可配置每次写入都执行 fsync，但写入性能会大幅下降。ES 宕机重启时，会通过 Translog 恢复未持久化到磁盘的数据。
5.  **Flush 流程（Commit 全量持久化）**
    *   触发时机：Translog 文件达到设定阈值，或默认每隔 30 分钟执行一次。
    *   执行逻辑：
        *   执行一次 refresh，将内存 Buffer 所有数据写入 OS Cache 的 Segment，清空内存 Buffer；
        *   创建 commit point 文件，记录当前所有有效的 Segment 文件信息；
        *   将 OS Cache 中所有 Segment 文件，通过 fsync 强制刷入磁盘，完成持久化；
        *   清空当前 Translog 日志文件，完成全量提交。
6.  **Merge 流程（Segment 合并优化）**
    *   触发背景：refresh 每秒生成新的小 Segment，Segment 数量过多会导致文件句柄占用过高、检索性能下降。
    *   执行逻辑：
        *   ES 后台自动将多个小 Segment 合并为一个大 Segment，合并完成后删除原小 Segment；
        *   合并过程中，若发现文档在.del 文件中被标记删除，不会将其写入新 Segment，实现物理删除；
        *   合并完成后，commit point 指向新的大 Segment 文件。

#### 更新与删除的底层逻辑

Lucene 的 Segment 文件是只读的，一旦写入磁盘，无法修改 / 删除内部内容，因此 ES 的更新、删除采用「标记 + 追加」的实现方式：

*   **删除流程**：删除文档时，不会直接从磁盘 Segment 中移除文档，而是在.del文件中记录该文档 ID，标记为已删除。检索时会先匹配文档，再通过.del文件过滤掉已删除的文档，仅在 Segment 合并时，才会真正物理删除标记的文档。
*   **更新流程**：本质是**删除 + 新增**的组合。更新文档时，ES 先将旧版本文档标记为已删除（写入.del 文件），再将新版本文档写入内存 Buffer，走完整的写入流程，既实现了数据更新，又适配了 Segment 只读的特性。

### 四、ES 搜索流程原理

ES 搜索采用经典的**两阶段查询（Query Then Fetch）** 模型，充分利用分布式分片的并行能力，同时最小化网络开销，对应图中的 3 个步骤，完整流程如下：

#### 第一阶段：Query 阶段

核心目标：拿到全局匹配度最高的文档 ID 列表，不获取完整文档内容，最小化数据传输。

1.  客户端发起搜索请求，集群任意节点接收请求，成为本次请求的协调节点。
2.  协调节点解析查询条件，根据索引元数据，确定需要查询的全部分片列表。
3.  协调节点通过负载均衡策略（轮询等），为每个分片选择一个可用副本（主分片 / 副本分片均可），将查询请求转发到对应节点。
4.  每个分片本地执行查询，在 Segment 中匹配符合条件的文档，生成**按相关性排序的文档 ID + 相关性得分列表**，返回给协调节点。
5.  协调节点收到所有分片的结果后，对全量文档 ID 和得分进行全局合并、排序，筛选出 Top N 的文档 ID（比如用户查询前 10 条，就选出全局得分最高的 10 个 ID），完成 Query 阶段。

#### 第二阶段：Fetch 阶段

核心目标：根据 Top N 的文档 ID，获取完整的文档内容，组装最终结果。

1.  协调节点根据文档 ID，计算出每个文档所在的分片，再次向对应分片节点发起 GET 请求，获取文档完整数据。
2.  分片节点返回对应的完整文档给协调节点。
3.  协调节点将所有文档组装为最终的查询结果，返回给客户端。

> 该模型的核心优势：Query 阶段仅传输轻量的 ID 和得分，Fetch 阶段仅拉取需要的 Top N 文档，避免全量数据传输，大幅降低网络开销，提升分布式检索的性能。

### 补充

1.  **倒排索引**：ES 全文检索的核心是 Lucene 的倒排索引。正排索引是「文档 ID → 文档内容」，而倒排索引是「关键词 → 包含该关键词的所有文档 ID 列表」，同时记录词频、位置、偏移量等信息，用于计算相关性得分，实现毫秒级全文匹配。
2.  **高可用保障**
    *   分片级容灾：主分片宕机后，Master 节点会立即将对应副本分片选举为新的主分片，持续提供服务；故障节点恢复后，自动同步数据补齐副本。
    *   集群级容灾：Master 节点有候选节点，当前 Master 宕机后，候选节点会通过 Raft 协议选举出新的 Master，保证集群管理不中断。
3.  **水平扩展能力**：当集群存储 / 性能不足时，只需新增节点加入集群，ES 会自动将分片均衡迁移到新节点，无需人工干预，实现无缝水平扩容。
