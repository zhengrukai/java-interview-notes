### 1.  **事前：高可用架构与监控（预防）**

首先，我们不能让 Redis 成为单点。

*   **部署架构**： 我们生产环境采用的是 Redis 哨兵模式（Sentinel）（或者 Redis Cluster）。一旦主节点宕机，哨兵会自动进行故障转移，选出新的主节点，整个过程对业务代码几乎是透明的。
*   **提前演练**： 在大促前，我会通过 Skywalking 观察 Redis 的连接池健康状况，并设置告警。如果 Redis CPU 占用过高或内存告警，我们会提前感知，不等到它挂。

### 2.  **事中：限流与降级策略（自保）**

如果极端情况下整个 Redis集群都不可用了（比如网络波动或机房故障），系统必须具备自我保护能力，不能让数据库被顶死。

*   **触发熔断降级**： 我在代码里设置了异常捕获。如果 Redis 连接超时或报错达到阈值，会立即触发降级逻辑。
*   **业务降级处理**： 此时我们会暂停‘积分预扣减’这种强依赖 Redis 的抢购逻辑。前端会返回一个通用的‘系统繁忙，请稍后再试’或‘兑换已排队’。虽然牺牲了部分用户的实时性，但保住了数据库，防止了整站崩溃。
*   **本地缓存兜底**： 对于一些非敏感的‘读’请求（比如图书的简介、书名），即便 Redis 挂了，我们还有第一步提到的 Caffeine 本地缓存 挡着，所以用户依然能正常浏览页面，只是点击‘兑换’时会提示排队。

### 3.  **事后：数据一致性与恢复（补救）**

Redis 恢复后，最核心的问题是数据恢复和缓存预热。

*   **缓存预热**： 重新启动服务后，我们不会立刻放开流量，而是先通过脚本将核心的库存、积分数据重新从数据库加载到 Redis 中。
*   **数据对账**： 针对宕机瞬间可能产生的‘Redis 扣了积分但 MQ 没发成功’或者‘数据库没更新’的情况，我们会通过离线对账脚本，比对数据库订单表和 Redis 的剩余库存。如果发现 Redis 里的数不准了，以数据库为准进行订正。

\*\* \*\*

**问：Redis 宕机时，MQ 里的消息会丢吗？**

**答：** “不会。因为 Redis 只是做‘预扣减’。如果 Redis 挂了，请求压根就不会发送到 MQ。只有 Redis 扣减成功的请求才会进 MQ。MQ 本身有持久化机制，所以只要消息发出去了，即便 Redis 挂了，后台的图书入库逻辑依然会慢慢把剩下的活儿干完。”
