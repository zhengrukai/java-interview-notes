# Redisson看门狗原理
Redisson看门狗的核心价值，是**彻底解决分布式锁的「业务未执行完、锁提前过期释放」的并发安全痛点**，是Redisson分布式锁实现高可靠的核心机制，也是Java后端面试的绝对高频考点。

---

## 一、核心痛点：为什么必须要有看门狗？
在分布式场景下，若使用Redis原生的`SET key value NX PX ttl`命令实现分布式锁，会面临一个致命问题：
- 假设锁的TTL设置为30秒，但业务逻辑因网络波动、Full GC、慢SQL等原因，执行了40秒；
- 30秒后锁自动过期释放，此时其他线程可以获取到同一把锁，导致并发安全问题（如数据重复写入、状态不一致）；
- 原业务线程执行完后，还会错误地释放其他线程持有的锁，进一步加剧混乱。

Redisson看门狗就是为了解决这个问题而生：**在业务未执行完时，自动为锁续期，确保锁不会提前释放；业务执行完或客户端宕机时，及时停止续期，让锁自动过期**。

---

## 二、核心原理流程（5步闭环）
Redisson看门狗的实现完全遵循通用看门狗的「阈值约定→喂狗→检测→自愈」闭环，结合Redis特性做了定制化设计：

### 1. 阈值约定：默认锁TTL与续期周期
- **默认锁TTL**：当调用`RLock.lock()`（不带超时时间的方法）时，Redisson会为锁设置一个默认的TTL——**30秒**（可通过`Config.setLockWatchdogTimeout()`全局修改）。
- **续期周期**：行业通用规则，为锁TTL的**1/3**，即默认每10秒续期一次，预留2次重试窗口，避免网络抖动导致误判。

### 2. 启动看门狗：获取锁成功后触发
只有当调用**不带超时时间的锁方法**（如`lock()`、`lockInterruptibly()`）且成功获取锁时，才会启动看门狗；若调用**带超时时间的锁方法**（如`lock(10, TimeUnit.SECONDS)`、`tryLock(10, 5, TimeUnit.SECONDS)`），则**不会启动看门狗**（Redisson认为用户已自行控制锁的持有时间）。

### 3. 喂狗续期：定期刷新锁TTL
看门狗本质是一个**独立的ScheduledExecutorService定时任务线程**，启动后会每10秒（默认）执行一次续期逻辑：
1.  **校验锁持有状态**：通过Redis的`HGET`命令，检查锁的Hash结构中，当前线程ID是否仍为锁的持有者（Redisson的锁是可重入的，用Hash结构存储`{线程ID: 重入次数}`）。
2.  **原子续期**：若确认当前线程仍持有锁，Redisson会通过**Lua脚本**原子性地执行`PEXPIRE`命令，将锁的TTL重新刷新为30秒（默认），确保锁不会过期。
3.  **循环续期**：续期成功后，看门狗会再次调度下一次10秒后的续期任务，形成循环，直到锁被主动释放或客户端宕机。

### 4. 停止看门狗：锁释放时取消任务
当业务线程调用`RLock.unlock()`释放锁时，Redisson会：
1.  先通过Lua脚本原子性地减少重入次数，若重入次数减为0，则删除锁Key；
2.  同时**取消对应的看门狗定时任务**，避免资源泄漏，停止续期。

### 5. 宕机兜底：客户端异常时自动失效
若客户端因宕机、网络断开、进程崩溃等原因无法继续执行续期任务，看门狗会随客户端一起停止：
- 锁的TTL不会再被刷新，30秒（默认）后自动过期释放；
- 其他线程可以正常获取锁，避免死锁，保障系统可用性。

## 三、源码关键节点（面试加分项）
Redisson看门狗的核心源码在`org.redisson.RedissonLock`类中，关键节点如下：

### 1. 锁获取与看门狗启动

// 不带超时时间的lock()方法
public void lock() {
    try {
        lock(-1, null, false);
    } catch (InterruptedException e) {
        Thread.currentThread().interrupt();
    }
}

// 核心锁获取逻辑
private void lock(long leaseTime, TimeUnit unit, boolean interruptibly) throws InterruptedException {
    long threadId = Thread.currentThread().getId();
    // 尝试获取锁，返回剩余TTL（null表示获取成功）
    Long ttl = tryAcquire(leaseTime, unit, threadId);
    if (ttl == null) {
        return; // 获取锁成功，直接返回
    }

    // ... 省略订阅、等待锁的逻辑 ...

    // 获取锁成功后，若leaseTime为-1（即未指定超时时间），启动看门狗

### 2. 看门狗续期逻辑
```java
// 启动看门狗续期任务
private CompletableFuture<Boolean> renewExpirationAsync(long threadId) {
    // 创建定时任务，延迟internalLockLeaseTime/3（默认10秒）后执行
    return commandExecutor.getConnectionManager().getScheduler().schedule(() -> {
        // 执行续期Lua脚本
        CompletableFuture<Boolean> future = renewExpiration(threadId);
        future.whenComplete((res, e) -> {
            if (e != null) {
                // 续期失败，停止看门狗
                return;
            }
            if (res) {
                // 续期成功，递归调度下一次续期任务
                renewExpirationAsync(threadId);
            }
        });
        return future;
    }, internalLockLeaseTime / 3, TimeUnit.MILLISECONDS);
}

// 续期Lua脚本（原子性校验持有者+刷新TTL）
protected CompletableFuture<Boolean> renewExpiration(long threadId) {
    return evalWriteAsync(getRawName(), LongCodec.INSTANCE, RedisCommands.EVAL_BOOLEAN,
            "if (redis.call('hexists', KEYS[1], ARGV[2]) == 1) then " +
                    "redis.call('pexpire', KEYS[1], ARGV[1]); " +
                    "return 1; " +
                    "end; " +
                    "return 0;",
            Collections.singletonList(getRawName()),
            internalLockLeaseTime, getLockName(threadId));
}
```
- Lua脚本的核心逻辑：先通过`hexists`检查当前线程是否仍持有锁，若持有则通过`pexpire`刷新TTL，返回1；否则返回0，停止续期。
- 使用Lua脚本的原因：**保证校验和续期的原子性**，避免在多线程/分布式场景下出现竞态条件。

### 3. 锁释放与看门狗停止

## 四、关键细节与边界情况
1.  **带超时时间的锁不启动看门狗**：若调用`lock(10, TimeUnit.SECONDS)`，Redisson会直接将锁的TTL设置为10秒，**不会启动看门狗**，也不会自动续期——这是面试最常考的区别点。
2.  **可重入锁的续期**：Redisson的锁是可重入的，看门狗续期时会校验重入次数，只要重入次数>0，就会持续续期，直到重入次数减为0、锁被完全释放。
3.  **看门狗线程池管理**：看门狗使用的是Redisson全局的`ScheduledExecutorService`，可通过`Config.setNettyThreads()`或`Config.setLockWatchdogTimeout()`间接调整（默认线程数为CPU核数*2）。
4.  **全局修改默认TTL**：可通过`Config.setLockWatchdogTimeout(60000)`将默认锁TTL修改为60秒，续期周期也会自动变为20秒。
