## key 过期原理

Redis 的过期处理机制是其内存管理中非常核心的一部分。Redis 并不是在 Key 到达过期时间那一刻立即将其删除，而是采用了一种\*\*“定期删除 + 惰性删除”**相结合的策略，辅以**内存淘汰机制\*\*来确保性能与内存空间的平衡。

以下是 Redis 过期原理的详细深度解析：

***

### 1. 过期字典（Expires Dict）

Redis 的数据库结构（`redisDb`）中维护了两个字典：

*   **dict（键值对字典）**：保存了所有的键值对。
*   **expires（过期字典）**：保存了所有设置了过期时间（TTL）的 Key。其 Key 是指向键值对字典中 Key 对象的指针，Value 是一个 `long long` 类型的 UNIX 时间戳（毫秒级）。

当我们要判断一个 Key 是否过期时，Redis 会先去 `expires` 字典中查找，如果当前时间 > 过期时间，则判定为已过期。

***

### 2. 核心删除策略

为了在不影响高性能的前提下清理过期数据，Redis 组合使用了两种策略：

#### (1) 惰性删除 (Lazy Deletion)

*   **原理**：Redis 不会主动监控 Key 是否过期。只有当客户端**访问**（GET、SET、EXISTS 等）某个 Key 时，Redis 才会检查该 Key 是否过期。
*   **流程**：
    1.  接收到读写请求。
    2.  检查 Key 是否在过期字典中。
    3.  如果已过期，删除该 Key，并向客户端返回空（nil）。
*   **优点**：对 CPU 极其友好。只有在必要时才执行删除，不会在不访问时占用 CPU 时间。
*   **缺点**：对内存不友好。如果大量过期 Key 以后再也不被访问，它们会一直占用内存，造成内存泄漏。

#### (2) 定期删除 (Periodic Deletion)

*   **原理**：Redis 内部维护了一个定时任务（默认每秒执行 10 次），通过随机抽取一部分 Key 来检查并删除其中的过期 Key。
*   **执行逻辑（自适应算法）**：
    1.  从过期字典中随机抽取 20 个 Key。
    2.  删除这 20 个 Key 中所有已过期的 Key。
    3.  如果已过期 Key 的占比超过 **25%**，则重复步骤 1（继续抽样删除）。
    4.  为了防止这个过程阻塞主线程太久，Redis 设置了**执行时间上限**（通常默认是 25ms）。
*   **意义**：它是惰性删除的补充，确保那些不再被访问的过期 Key 最终能被清理掉。

***

### 3. 为什么不使用“定时删除”？

**定时删除**是指为每个设置了过期时间的 Key 创建一个定时器，到时间立即删除。

*   **Redis 不采用的原因**：如果 Key 非常多，创建成千上万个定时器会消耗大量的 CPU 资源来处理中断和调度，在高并发下会严重影响 Redis 的吞吐量。

***

### 4. 内存淘汰机制 (Eviction) —— 最后的防线

如果“定期删除”没赶上 Key 过期的速度，或者有大量没有设置过期时间的 Key 占满了内存，触发了 `maxmemory` 限制，Redis 就会触发**内存淘汰机制**（如 `allkeys-lru`、`volatile-lfu` 等）。

*   **过期删除**：针对设置了 TTL 且时间已到的 Key。
*   **内存淘汰**：针对内存已满，根据策略强行剔除一部分 Key（哪怕它们还没过期）。

***

### 5. 持久化与主从复制中的处理

过期 Key 在持久化和集群环境下有特殊的处理逻辑：

#### (1) RDB

*   **生成 RDB**：过期 Key 不会被保存。
*   **载入 RDB**：如果是主库，过期的 Key 会被过滤；如果是从库，会全部载入（但随后主库同步数据时会处理）。

#### (2) AOF

*   **写入 AOF**：当 Key 过期被删除时，Redis 会显式地向 AOF 文件追加一条 `DEL` 命令。
*   **AOF 重写**：过期的 Key 不会被重写进新的 AOF 文件。

#### (3) 主从复制（Consistency）

*   **主库（Master）**：负责所有的过期判断和删除操作。当主库删除一个过期 Key 后，会同步向所有从库发送一条 `DEL` 命令。
*   **从库（Replica）**：**从库不会主动删除过期 Key**。即使从库发现某个 Key 已过期，在没有收到主库的 `DEL` 指令前，它依然会返回该 Key 的值（在 Redis 3.2 之后的版本中，从库在读取时会返回空，但物理上不删除）。
*   **目的**：保证主从数据的一致性。

***

### 总结

Redis 的过期原理是一个**时间与空间的权衡**：

1.  **过期字典**记录时间。
2.  **惰性删除**保障实时性（访问时处理）。
3.  **定期删除**保障内存回收（不访问也处理）。
4.  **主从机制**保障数据统一（主库说了算）。
