Redis 基于Reactor 事件驱动模型

# 核心模块拆解

### 1. IO 多路复用

> Redis 高性能的基石，封装了操作系统的 epoll/kqueue/select 等多路复用 API，统一监听所有 socket 的事件（客户端连接、命令可读、结果可写）。单线程即可监听上万连接，仅当事件就绪时才触发处理，避免阻塞轮询的开销。

### 2. 事件队列

> IO 多路复用监听到就绪事件后，会将事件写入这个队列，实现「事件监听」和「事件处理」的解耦，保证事件按顺序处理。

### 3. 文件事件分派器

> Reactor 模型的分发核心，从事件队列中取出就绪事件，根据事件类型、对应的 socket，分发给对应的处理器执行。

### 4. 连接应答处理器

> 专门处理客户端 TCP 连接建立：客户端发起 connect 时，Redis 监听 socket 产生可读事件，该处理器执行 accept 完成三次握手，创建客户端专属 socket，并绑定后续的命令读事件处理器。

### 5. 命令请求处理器

> 负责处理客户端的命令输入：客户端 socket 有可读事件（发送了命令）时，该处理器读取 RESP 格式的命令、解析校验，交给 Redis 核心执行模块，单线程操作内存数据完成命令执行。

### 6. 命令回复处理器

> 负责给客户端返回结果：命令执行完成后，客户端 socket 有可写事件时，该处理器将执行结果封装为 RESP 格式，发送回客户端，完成一次请求闭环。

# 完整请求流转链路

1.  客户端发起 TCP 连接，IO 多路复用监听到监听 socket 的连接就绪事件，写入事件队列；
2.  文件事件分派器取出事件，分发给连接应答处理器，完成连接建立，为客户端 socket 绑定读事件处理器；
3.  客户端发送 Redis 命令，客户端 socket 产生可读就绪事件，IO 多路复用监听到后写入事件队列；
4.  分派器取出事件，分发给命令请求处理器，完成命令读取、解析、执行（核心命令执行全程单线程）；
5.  命令执行完成，为客户端 socket 绑定可写事件，IO 多路复用监听到可写就绪后写入事件队列；
6.  分派器取出事件，分发给命令回复处理器，将结果返回客户端，完成一次完整请求。

# 补充关键细节

*   Redis 6.0 之前，网络 IO 读写、命令执行全程单线程；6.0 + 引入多线程 IO，仅把「网络数据读写、协议解析」放到多线程，核心命令执行依然是单线程，保证线程安全和执行时序。
*   除了文件事件，Redis 还有时间事件（比如过期 key 清理、AOF 定时刷盘）
