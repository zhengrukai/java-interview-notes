## Redis + Lua 脚本 问题及解决方案

***

### 问题一：脚本执行时间过长（阻塞主线程）—— **最致命**

Redis 的单线程机制决定了在执行 Lua 脚本时，**所有的其他客户端请求都会被阻塞**。如果你的脚本里有一个死循环，或者处理了 10 万条数据，整个 Redis 就会看起来像“死机”了一样，导致线上事故。

*   **现象**：
    *   Redis QPS 归零，所有连接超时。
    *   监控报警显示 CPU 飙升或 Latency 极高。
*   **解决方案**：
    1.  **严禁复杂逻辑**：Lua 脚本必须短小精悍。只做简单的判断和数值计算，禁止在脚本里做耗时的 O(N) 遍历（如遍历大 Set/List）。
    2.  **限制循环次数**：如果必须遍历，必须设置上限（例如每次只处理 100 条），分批次调用。
    3.  **急救措施**：
        *   如果脚本**没执行写操作**（只有读），其他客户端可以发送 `SCRIPT KILL` 强行终止脚本。
        *   如果脚本**已经执行了写操作**（修改了数据），`SCRIPT KILL` 会失效（为了保证数据一致性），此时只能 **`SHUTDOWN NOSAVE`**（重启 Redis 且不持久化），这会丢失数据，是下下策。

***

### 问题二：脚本缓存爆炸（内存泄漏）

*   **原理**：
    Redis 会缓存执行过的 Lua 脚本（以 SHA1 为 Key）。
    如果你在代码里这样写：
    ```java
    // 错误写法：把参数硬编码在脚本字符串里
    String script = "return redis.call('set', 'user:123', 'active')";
    jedis.eval(script);
    ```
    每次 `user_id` 变了（user:456, user:789），Redis 都会把这串代码当做一个**新脚本**缓存起来。如果你有 1000 万个用户，Redis 内存里就会存 1000 万个几乎一样的脚本，导致 **OOM (内存溢出)**。

*   **解决方案**：
    **参数参数化（Parametrization）**。永远使用 `KEYS` 和 `ARGV` 传参，不要拼接字符串。
    ```java
    // 正确写法
    String script = "return redis.call('set', KEYS[1], ARGV[1])";
    jedis.eval(script, Collections.singletonList("user:123"), Collections.singletonList("active"));
    ```
    这样无论用户 ID 怎么变，脚本的 SHA1 都是一样的，Redis 只需要缓存一份代码。

***

### 问题三：Redis Cluster 下的跨 Slot 问题

*   **问题**：
    在 Redis Cluster（集群）模式下，数据被分布在 16384 个 Slot（槽）中。
    Lua 脚本强制要求：**脚本中操作的所有 Key，必须在同一个 Slot 上**。
    如果你的脚本试图操作 `Key_A`（在节点 1）和 `Key_B`（在节点 2），Redis 会直接报错：`CROSSSLOT Keys in request don't hash to the same slot`.

*   **解决方案**：
    1.  **Hash Tag**：这是最通用的解法。
        在 Key 中加入 `{}`，Redis 只会根据 `{}` 内部的字符串计算 Slot。
        例如：`{user:100}:info` 和 `{user:100}:order`。虽然 Key 不同，但大括号内都是 `user:100`，它们会被强制分配到同一个 Slot，脚本就能操作了。
    2.  **业务层拆分**：如果无法使用 Hash Tag，就不要在 Lua 里一次操作多个 Key，改为在 Java/Go 代码里分两次请求执行（虽然失去了原子性，需要业务层做补偿）。

***

### 问题四：缺乏回滚机制（原子性的误区）

*   **问题**：
    很多人认为 Redis Lua 是事务，支持 ACID。**错！**
    Redis Lua 保证的是 **原子性执行（Atomic Execution）**，即“执行过程中不被插入其他命令”，但**不支持回滚（Rollback）**。

    **场景**：脚本有 3 行代码。

    1.  设置 A = 1（成功）
    2.  设置 B = 2（成功）
    3.  抛出一个错误（Error）
        **结果**：脚本停止，报错返回。但在 Redis 里，**A 和 B 的值已经被修改了，不会自动回滚成旧值**。

*   **解决方案**：
    1.  **防御性编程**：把所有检查逻辑（Check）放在脚本最前面。只有所有条件都满足了，才执行最后的修改逻辑（Set）。
        *   *Bad*: 修改A -> 检查B -> 报错。
        *   *Good*: 检查B -> 修改A。
    2.  **幂等性设计**：确保脚本重复执行不会导致数据错乱。

***

### 问题五：随机性导致的主从数据不一致

*   **问题**：
    在 Redis 3.2 之前，如果脚本里使用了 `Time()`（获取系统时间）或 `math.random()`（随机数），并且是基于语句复制（Statement-based replication）到从库的。
    *   主库执行时随机出 5，写入了 5。
    *   从库回放脚本时随机出 8，写入了 8。
    *   **主从数据不一致**。

*   **解决方案**：
    1.  **升级版本**：Redis 3.2+ 引入了 `redis.replicate_commands()`（基于效果复制），即主库执行完脚本后，把生成的**结果命令**（如 `set a 5`）发给从库，而不是发送脚本代码。Redis 5.0+ 默认就是这种模式。
    2.  **参数传入**：如果版本旧，不要在脚本里生成随机数或时间，应该在 Java/Go 代码里生成好，通过 `ARGV` 传进去。

***

### 总结（面试回答策略）

如果面试官问：“Redis Lua 脚本有什么坑？”，你可以这样总结：

> “Redis Lua 脚本主要解决了多条命令执行的原子性问题，但有四个主要风险：
>
> 1.  **阻塞风险（最大痛点）**：因为 Redis 是单线程的，脚本过长会卡死整个服务。解决办法是**脚本必须尽量短小**，严禁在大 key 上做循环。
> 2.  **内存风险**：如果动态拼接脚本字符串，会导致 Script Cache 溢出。解决办法是必须使用 **KEYS 和 ARGV 传参**，利用 `EVALSHA`。
> 3.  **集群限制**：Cluster 模式下脚本涉及的 key 必须在同一个 slot。解决办法是使用 **Hash Tag**（如 `{userId}`）。
> 4.  **无回滚**：脚本执行中间报错不会回滚已修改的数据。解决办法是**把检查逻辑前置**，确保‘先 Check 后 Set’。”
