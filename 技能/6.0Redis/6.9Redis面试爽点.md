在我负责的电商中台、WMS 仓储管理系统、小荷亲子小程序这些高并发项目中，Redis 是核心支撑组件，主要用在热点数据缓存、秒杀分布式锁、库存实时管控、分销佣金规则缓存等场景，日均万级订单、秒杀峰值 QPS8000 + 的业务量下，我碰到的 Redis 最大核心问题是**高并发场景下的热 key 击穿、库存超卖，以及大 key 导致的主线程阻塞**，这三个问题常联动出现，也是高并发业务中 Redis 最易引发的核心痛点，我结合实际项目从发现到解决做了完整的落地优化，接下来具体说说：

首先是**如何发现这些问题**，我主要通过技术监控、业务异常、日志分析三个维度，结合项目中的监控栈落地，不会等故障爆发才发现：第一是技术监控层面，用 SkyWalking 做全链路监控，搭配 Redis 自身的 info、slowlog 命令，监控 keyspace\_misses（缓存未命中）骤升、命令延迟 99 分位超阈值，还有秒杀时 Redis 单节点 CPU、网络带宽直接占满；第二是业务异常层面，比如小荷亲子小程序秒杀初期出现商品超卖、WMS 系统库存数据不一致、电商中台分销佣金查询接口超时；第三是日志分析层面，通过 ELK 技术栈分析业务日志，发现大量请求穿透到 MySQL，导致 DB 的 QPS 和 CPU 突增，同时 Redis 慢日志记录到大 key 的 DEL、全量查询操作耗时过长，这些信号直接定位了 Redis 的核心问题。

然后是**针对性的解决方法**，都是我在项目中实际落地的方案，结合业务场景做了定制化优化，而非单纯的理论调整：

第一，针对小荷亲子小程序秒杀场景的热 key 击穿 + 库存超卖（峰值 QPS8000+），一方面用 Redis 的 SETNX 实现分布式锁，同时给锁设置 5 秒过期时间避免死锁，从根本上解决超卖问题；另一方面做二级缓存设计，核心热 key 比如秒杀商品库存、会员信息，先做应用本地 Caffeine 缓存，再查 Redis，分散单节点压力；同时对热点商品 key 做分片，拆分为多个子 key 分散到不同 Redis 节点，解决单节点 CPU 占满的问题。

第二，针对 WMS 仓储系统的库存缓存一致性问题，我用 Redis 实时缓存库存数据，设置多级预警阈值，结合 RabbitMQ 异步推送库存预警消息；订单扣减库存采用 “先扣 Redis 库存，再通过 MQ 异步同步到 MySQL” 的策略，同时设计了库存回滚机制，避免扣减 Redis 后 DB 写入失败导致的数据不一致；另外用 Redis 的 Hash 结构存储多单位库存数据，避免单 key 过大，查询时用 HSCAN 分批操作，替代全量查询。

第三，针对电商中台分销佣金规则的大 key 阻塞问题，核心做了大 key 拆分，把原有的单一大 Hash 佣金规则 key，按代理等级拆分为多个小 Hash，单个小 key 数据量控制在 10 万以内；删除操作改用 Redis 的 UNLINK 替代 DEL，通过异步删除避免阻塞主线程；佣金规则的更新也采用 RabbitMQ 异步推送，Redis 做缓存更新，避免同步更新的性能损耗。

第四，做了通用的监控和配置优化，结合 ELK+SkyWalking 搭建 Redis 全链路监控，实时监控内存、命中率、命令延迟，设置告警阈值；同时开启 Redis 自动碎片整理，当碎片率超 1.5 时自动触发整理，避免内存浪费导致的 OOM 风险。

最后说下优化后的实际效果，小荷亲子小程序的秒杀场景彻底解决了超卖问题，峰值 QPS5000 + 能稳定运行；WMS 仓储系统的库存盘点准确率从 75% 提升至 99.8%，Redis 缓存命中率稳定在 99% 以上；电商中台分销佣金接口的响应时间从 500ms 降至 50ms 内，Redis 节点无任何阻塞情况，我负责的所有项目 Redis 相关模块支撑系统保持 99.9% 的可用性。

其实从这些实战中我也总结出，Redis 的问题核心不是技术本身，而是要结合业务场景做架构设计，提前规避热 key、大 key，做好缓存一致性和分布式锁的细节设计，再配套完善的监控告警体系，才能在高并发场景下发挥 Redis 的性能优势。
