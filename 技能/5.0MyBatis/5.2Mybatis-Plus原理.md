MyBatis-Plus（简称 MP）是 MyBatis 的「增强工具」，核心宗旨是 “只做增强，不做改变” —— 完全复用 MyBatis 的核心原理（如 SqlSession、动态代理、Executor 等），在此基础上通过自动生成通用 SQL、扩展插件、封装通用 CRUD 等方式，彻底简化 MyBatis 的开发（无需编写基础 CRUD 的 SQL）。

## 一、核心定位：MyBatis 的 “增强补丁”

1. 复用 MyBatis 所有核心组件（SqlSessionFactory、SqlSession、Executor 等）；
2. 扩展 MyBatis 的配置解析、Mapper 代理、SQL 生成逻辑；
3. 提供通用 CRUD、条件构造、分页、主键生成等 “开箱即用” 的能力；
4. 完全兼容 MyBatis 原有用法（自定义 SQL、注解、XML 映射均不受影响）。

## 二、MP 核心组件（增强 MyBatis 的关键）

| 组件 | 核心作用 |
| --- | --- |
| BaseMapper<T> | 通用 Mapper 接口，封装了 17+ 基础 CRUD 方法（selectById、insert、updateById 等），自定义 Mapper 继承即可复用 |
| IService/ServiceImpl<T> | 业务层通用 CRUD 封装，基于 BaseMapper 扩展了批量操作、逻辑删除、条件查询等更丰富的方法 |
| SqlHelper | MP 核心 SQL 构建助手，负责根据实体类、方法参数自动生成通用 CRUD 的 SQL 语句 |
| QueryWrapper/LambdaQueryWrapper | 条件构造器，通过链式调用动态拼接 WHERE 条件（替代手动拼接 SQL 字符串） |
| MybatisPlusInterceptor | MP 插件核心入口，整合分页、乐观锁、防全表更新 / 删除等插件，基于 MyBatis 的 Interceptor 扩展 |
| IdentifierGenerator | 主键生成器，默认实现雪花算法（Snowflake），支持自动生成主键（如 ID 自增、分布式 ID） |
| MetaObjectHandler | 字段自动填充处理器，拦截 insert/update 操作，自动填充 createTime、updateTime 等字段 |
| TableInfoHelper | 表信息助手，通过反射解析实体类的注解（@TableName、@TableId、@TableField），获取表名、字段名、主键等映射关系 |

## 三、MP 核心增强原理（最关键的底层逻辑）
MP 最核心的能力是「自动生成通用 CRUD SQL」和「插件扩展」，下面拆解核心原理：
#### 1. 通用 CRUD 自动生成 SQL 的原理
当你自定义 Mapper 继承 BaseMapper 并调用 selectById/insert 等方法时，MP 的执行逻辑如下：
- **步骤 1：反射解析实体元数据**
    MP 通过 TableInfoHelper 反射获取实体类的信息：
    - 表名：默认取实体类名（驼峰转下划线），或通过 @TableName 指定；
    - 字段名：默认取属性名（驼峰转下划线），或通过 @TableField 指定；
    - 主键：通过 @TableId 识别主键字段，以及主键生成策略（如雪花算法、自增）。
- **步骤 2：根据方法名匹配 SQL 模板**
    MP 为 BaseMapper 的每个方法预设了 SQL 模板（如 selectById 对应 SELECT * FROM 表名 WHERE 主键 = ?）；
- **步骤 3：动态拼接 SQL**
    结合实体元数据和方法参数，填充 SQL 模板的占位符，生成可执行的 SQL（比如 SELECT id, user_name, age FROM user WHERE id = 1）；
- **步骤 4：交给 MyBatis 执行**
    生成的 SQL 最终交给 MyBatis 的 Executor 执行，复用 MyBatis 原有参数处理、结果集转换逻辑。
#### 2. 动态代理增强原理
MP 对 MyBatis 的 Mapper 动态代理做了扩展：
- MyBatis 原生只负责映射 “接口方法 ↔ 自定义 SQL”；
- MP 扩展后，当调用 BaseMapper 的通用方法时，MP 的代理逻辑会接管，先自动生成 SQL，再交给 MyBatis 执行；
- 调用自定义方法（自己写的 SQL）时，仍走 MyBatis 原生逻辑，完全兼容。
#### 3. 分页插件原理
MP 的分页功能是开发中最常用的增强，底层基于 MyBatis 的 Interceptor 拦截器实现：
- **拦截时机**：拦截 Executor.query() 方法；
- **核心逻辑**：
    1. 识别分页参数（Page 对象）；
    2. 先执行 SELECT COUNT(*) FROM 原SQL 获取总条数；
    3. 改写原 SQL，拼接分页语法（如 MySQL 的 LIMIT ?,?、Oracle 的 ROWNUM）；
    4. 执行改写后的 SQL，获取分页数据；
    5. 将总条数和分页数据封装到 Page 对象返回。
#### 4. 主键自动生成原理
MP 默认使用雪花算法生成分布式唯一 ID，核心逻辑：
- 插入数据时，MP 拦截 insert 操作
- 通过 IdentifierGenerator 生成主键 ID；
- 将生成的 ID 填充到实体类的主键字段；
- 再执行插入 SQL（无需手动设置 ID）。
## 四、MP 完整工作流程
```
flowchart TD
    A[加载MP配置] --> B[构建增强版SqlSessionFactory]
    B --> C[创建SqlSession]
    C --> D[获取增强版Mapper代理对象]
    D --> E[调用selectById方法]
    E --> F[反射解析User实体元数据]
    F --> G[自动生成SELECT SQL]
    G --> H[交给MyBatis Executor执行]
    H --> I[结果集转换为User对象]
    I --> J[返回结果]
```
## 总结
1. MP 核心是「增强不替换」：完全复用 MyBatis 原有核心原理，仅在其上扩展通用 CRUD、插件等能力；
2. 通用 CRUD 本质是「反射 + SQL 模板」：通过解析实体注解生成元数据，动态拼接通用 SQL，无需手动编写；
3. 插件扩展基于 MyBatis 拦截器：分页、乐观锁等功能通过拦截 Executor 方法实现，不侵入原有 SQL 逻辑。
