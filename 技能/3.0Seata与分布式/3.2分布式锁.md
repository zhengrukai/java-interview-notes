#### 1. 核心概念
分布式锁是解决**分布式系统中多个节点竞争同一共享资源**的并发控制手段。单机系统中可使用synchronized/ReentrantLock等本地锁，但分布式集群（多 JVM）环境下，本地锁无法跨节点互斥，因此需要分布式锁。
#### 2. 核心要求
- 互斥性：同一时刻仅一个节点持有锁
- 超时释放：防止节点宕机导致死锁
- 可重入：同一节点的同一线程可重复获取锁
- 高可用：锁服务本身无单点故障
- 高性能：加锁 / 解锁操作耗时短
#### 3. Java 实现（Redis + Redisson，主流方案）
Redisson 是 Redis 官方推荐的 Java 客户端，封装了完善的分布式锁实现，避免手动实现的各种坑（如超时续期、误释放锁）

**前置条件**：引入 Maven 依赖
```
<dependency>
    <groupId>org.redisson</groupId>
    <artifactId>redisson-spring-boot-starter</artifactId>
    <version>3.23.0</version>
</dependency>
```
**核心代码：**
```
import org.redisson.api.RLock;
import org.redisson.api.RedissonClient;
import org.redisson.config.Config;
import org.springframework.stereotype.Component;

import javax.annotation.PostConstruct;
import java.util.concurrent.TimeUnit;

@Component
public class RedisDistributedLock {
    private RedissonClient redissonClient;

    // 初始化 Redis 客户端
    @PostConstruct
    public void init() {
        Config config = new Config();
        // 单机 Redis 配置（集群环境需改为 clusterServersConfig）
        config.useSingleServer()
              .setAddress("redis://127.0.0.1:6379")
              .setPassword("your_redis_pwd") // 无密码则省略
              .setDatabase(0);
        redissonClient = org.redisson.Redisson.create(config);
    }

    /**
     * 获取分布式锁
     * @param lockKey 锁的唯一标识（如：order:create:1001）
     * @param leaseTime 自动释放时间（避免死锁）
     * @return 是否获取成功
     */
    public boolean lock(String lockKey, long leaseTime) {
        RLock lock = redissonClient.getLock(lockKey);
        try {
            // 最多等待10秒，上锁后leaseTime秒自动释放
            return lock.tryLock(10, leaseTime, TimeUnit.SECONDS);
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
            return false;
        }
    }

    // 释放锁（仅当前线程持有锁时才释放）
    public void unlock(String lockKey) {
        RLock lock = redissonClient.getLock(lockKey);
        if (lock.isHeldByCurrentThread()) {
            lock.unlock();
        }
    }

    // 业务使用示例
    public void doBusiness(String orderId) {
        String lockKey = "order:create:" + orderId;
        try {
            if (lock(lockKey, 30)) { // 30秒自动释放
                // 执行业务：创建订单、扣减库存等
                System.out.println("获取锁成功，执行业务逻辑");
            } else {
                System.out.println("获取锁失败，请勿重复操作");
            }
        } finally {
            unlock(lockKey); // 最终释放锁
        }
    }
}
```
关键解释：
- RLock：兼容 Java Lock 接口，支持可重入、公平锁；
- tryLock：非阻塞加锁，避免线程无限等待；
- 释放锁时校验isHeldByCurrentThread：防止误释放其他线程的锁。

#### 4. Java 实现（ZooKeeper + Curator，高可靠方案）

ZooKeeper 基于临时节点和 Watcher 机制实现分布式锁，优势是**天然支持可重入、自动释放**（客户端断开会话结束，临时节点自动删除）。

```xml
<!-- Maven 依赖 -->
<dependency>
    <groupId>org.apache.curator</groupId>
    <artifactId>curator-recipes</artifactId>
    <version>5.5.0</version>
</dependency>
```

```java
import org.apache.curator.framework.CuratorFramework;
import org.apache.curator.framework.recipes.locks.InterProcessMutex;
import org.springframework.stereotype.Component;

import javax.annotation.PostConstruct;
import java.util.concurrent.TimeUnit;

@Component
public class ZkDistributedLock {
    private CuratorFramework zkClient;
    private InterProcessMutex mutex;

    @PostConstruct
    public void init() {
        zkClient = CuratorFrameworkFactory.newClient("127.0.0.1:2181", 5000, 3000, null);
        zkClient.start();
    }

    // 获取锁
    public boolean lock(String lockKey, long timeout) {
        mutex = new InterProcessMutex(zkClient, "/locks/" + lockKey);
        try {
            return mutex.acquire(timeout, TimeUnit.SECONDS);
        } catch (Exception e) {
            return false;
        }
    }

    // 释放锁
    public void unlock() {
        try {
            if (mutex != null && mutex.isAcquiredInThisProcess()) {
                mutex.release();
            }
        } catch (Exception e) {
            // 忽略
        }
    }
}
```

#### 5. Redis 锁 vs ZooKeeper 锁 对比

| 维度 | Redis (Redisson) | ZooKeeper (Curator) |
|------|-----------------|---------------------|
| 一致性 | AP（主从异步复制可能丢锁） | CP（ZAB 协议强一致） |
| 性能 | 高（内存操作） | 中（需要创建/删除节点） |
| 可靠性 | 较低（主从切换可能丢锁，Redlock 可缓解） | 高（会话断开自动释放） |
| 实现难度 | 低（Redisson 开箱即用） | 低（Curator 封装完善） |
| 适用场景 | 高性能、允许偶尔丢锁 | 高可靠、强一致性要求 |

> [!IMPORTANT] **面试高频问题：Redlock 算法是什么？**
> - **问题**：单机 Redis 主从切换时可能丢失锁（主写了锁，未同步到从，主挂后从升主，锁消失）
> - **Redlock 方案**：同时向 N 个独立的 Redis 实例申请锁，只要超过半数（N/2+1）成功即认为获取锁
> - **争议**：Martin Kleppmann 论文质疑 Redlock 的时间依赖问题（时钟漂移），实际生产中较少使用 Redlock
> - **实践建议**：大多数场景单机 Redis + Redisson 足够，可靠性要求极高的场景用 ZooKeeper 锁

#### 6. 分布式锁常见面试问题

### Q1：Redis 分布式锁如何保证可重入？
**答：** Redisson 底层使用 Hash 结构存储锁信息：key 为锁名，field 为线程 ID，value 为重入次数。同一线程重复获取锁时，value 递增；释放锁时递减，为 0 时删除 key。

### Q2：业务没执行完锁就过期了怎么办？
**答：** Redisson 内置 **Watch Dog 机制**（看门狗），默认每隔 10 秒（lockWatchdogTimeout/3）检查当前线程是否仍持有锁，如果是则自动续期（重新设置过期时间为 30 秒）。只要线程不宕机，锁就不会过期。

### Q3：主从架构下 Redis 锁丢了怎么解决？
**答：** 三种方案：
1. 用 Redisson 的 Redlock（多实例投票，有争议）
2. 改用 ZooKeeper 实现锁（推荐，强一致）
3. 接受偶尔不一致，用业务幂等兜底（实际生产中最常见）

### Q4：分布式锁的 key 如何设计？
**答：** 根据业务场景设计，原则是**同一业务操作竞争同一个 key**：
- 防止重复提交：`lock:submit:order:{userId}`
- 库存扣减：`lock:stock:{productId}`
- 定时任务防重复执行：`lock:job:{taskName}`
