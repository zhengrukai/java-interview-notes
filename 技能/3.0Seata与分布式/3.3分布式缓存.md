#### 1. 核心概念

分布式缓存是将缓存数据存储在独立的缓存服务（如 Redis）中，供分布式系统所有节点共享访问。解决了单机缓存（如 HashMap）数据不共享、一致性差的问题，核心价值是减轻数据库压力、提升系统响应速度。

#### 2. 核心要求

*   数据一致性：缓存与数据库数据尽可能一致；
*   高可用：缓存服务无单点故障；
*   高性能：基于内存读写，响应快；
*   可选持久化：防止缓存重启后数据丢失。

#### 3. Java 实现（Redis + Spring Cache，简化开发）

Spring Cache 提供注解式缓存操作，结合 Redis 实现分布式缓存，无需手动编写缓存逻辑。
**前置条件：**

    <!-- Maven 依赖 -->
    <dependency>
        <groupId>org.springframework.boot</groupId>
        <artifactId>spring-boot-starter-cache</artifactId>
    </dependency>
    <dependency>
        <groupId>org.springframework.boot</groupId>
        <artifactId>spring-boot-starter-data-redis</artifactId>
    </dependency>

<!---->

    # application.yml 配置
    spring:
      redis:
        host: 127.0.0.1
        port: 6379
        password: your_redis_pwd
      cache:
        type: redis
        redis:
          time-to-live: 3600000 # 缓存默认过期1小时
          key-prefix: "app:cache:" # 缓存key前缀
          cache-null-values: false # 不缓存null值（避免缓存穿透）

**核心代码：**

    import org.springframework.cache.annotation.CacheEvict;
    import org.springframework.cache.annotation.CachePut;
    import org.springframework.cache.annotation.Cacheable;
    import org.springframework.stereotype.Service;

    @Service
    // 统一指定缓存名称
    @CacheConfig(cacheNames = "userCache")
    public class UserCacheService {

        /**
         * 查询缓存：命中则返回缓存，未命中则查库并缓存
         */
        @Cacheable(key = "#userId", unless = "#result == null")
        public User getUserById(Long userId) {
            // 实际项目中替换为数据库查询（MyBatis/DAO）
            System.out.println("从数据库查询用户：" + userId);
            return new User(userId, "张三", 25);
        }

        /**
         * 更新缓存：更新数据库后同步更新缓存
         */
        @CachePut(key = "#user.id")
        public User updateUser(User user) {
            System.out.println("更新数据库用户：" + user);
            return user;
        }

        /**
         * 删除缓存：删除数据库后删除对应缓存
         */
        @CacheEvict(key = "#userId")
        public void deleteUser(Long userId) {
            System.out.println("删除数据库用户：" + userId);
        }

        // 模拟用户实体
        public static class User {
            private Long id;
            private String name;
            private Integer age;

            public User(Long id, String name, Integer age) {
                this.id = id;
                this.name = name;
                this.age = age;
            }
            // getter/setter 省略
        }
    }

**关键解释：**

*   @Cacheable：查询缓存注解，优先读缓存，无则执行方法并缓存；
*   @CachePut：更新缓存注解，执行方法后更新缓存；
*   @CacheEvict：删除缓存注解，删除指定 key 或全部缓存；
*   核心问题处理：
    *   缓存穿透：请求不存在的 key 导致查库 → 解决方案：缓存 null 值（短过期）、布隆过滤器；
    *   缓存击穿：热点 key 过期瞬间大量请求打向数据库 → 解决方案：热点 key 永不过期、分布式锁；
    *   缓存雪崩：大量 key 同时过期 → 解决方案：过期时间加随机值、Redis 集群。

#### 4. Redis 缓存与数据库一致性方案

##### 4.1 更新策略对比

| 策略 | 说明 | 问题 |
|------|------|------|
| **先更新数据库，再删缓存** | Cache Aside Pattern，最常用 | 极端并发场景下可能短暂不一致 |
| 先删缓存，再更新数据库 | 读请求会重建缓存 | 更新期间读请求拿到旧数据写回缓存 |
| 先更新数据库，再更新缓存 | 写后写缓存 | 并发写可能覆盖其他线程的数据 |
| 延迟双删 | 先删缓存 → 更新DB → sleep(N) → 再删缓存 | 复杂且不可靠，不推荐 |

> [!IMPORTANT] **面试标准答案：推荐使用"先更新数据库，再删缓存"方案**
> 理由：并发场景下即使出现不一致，下一次读请求会自动重建正确缓存。对一致性要求极高的场景，可配合 Canal 监听 binlog 异步删除缓存。

##### 4.2 Canal 异步删除缓存方案

```
业务服务：更新数据库 → 返回成功（不操作缓存）
     ↓
MySQL binlog 写入
     ↓
Canal 监听 binlog 变化 → 发送到 MQ
     ↓
缓存服务消费 MQ 消息 → 删除 Redis 中的对应缓存
```

#### 5. Redis 集群模式

| 模式 | 说明 | 优点 | 缺点 |
|------|------|------|------|
| **单机** | 单个 Redis 实例 | 简单 | 单点故障，容量有限 |
| **主从复制** | 一主多从，读写分离 | 提高读性能 | 主节点故障需手动切换 |
| **哨兵模式 (Sentinel)** | 主从 + 自动故障转移 | 高可用，自动选主 | 写容量不变，无法水平扩展 |
| **Cluster 集群** | 多主多从，数据分片 | 高可用 + 水平扩展 | 运维复杂，跨 slot 操作不支持 |

#### 6. 一致性 Hash 算法

**场景：** 分布式缓存节点扩容/缩容时，尽量减少缓存 key 的重新分配。

```
传统 Hash：hash(key) % N  → 节点数 N 变化时，所有 key 都需迁移
一致性 Hash：hash(key) 映射到 2^32 圆环上，顺时针找最近的节点
          → 节点变化时，只有少量 key 需要迁移
```

**虚拟节点：** 在圆环上为每个物理节点创建多个虚拟节点（如 IP#1, IP#2, IP#3），解决数据倾斜问题，使 key 分布更均匀。

#### 7. 缓存高频面试题

### Q5：布隆过滤器如何解决缓存穿透？
**答：** 布隆过滤器（Bloom Filter）是一个位数组 + 多个 Hash 函数，在查询缓存前先判断 key 是否**可能**存在于数据库：
1. 数据写入时，将 key 通过多个 Hash 函数映射到位数组
2. 查询时，如果所有 Hash 位都为 1 → key 可能存在；有一位为 0 → key 一定不存在
3. 不存在则直接返回，避免查询数据库

**误判率：** 布隆过滤器有误判（可能存在但实际不存在），不会漏判。可通过调整位数组大小和 Hash 函数数量降低误判率。

### Q6：Redis 缓存数据量太大怎么解决？
**答：** 
- **数据分片**：使用 Redis Cluster 水平扩展
- **压缩存储**：大 key 拆分（Hash 拆成多个小 Hash）
- **冷热分离**：热点数据放 Redis，冷数据放数据库或其他存储
- **过期策略**：合理设置 TTL，惰性删除 + 定期删除结合

### Q7：Redis 和 Memcached 选哪个？
**答：** 现代项目几乎都选 Redis：

| 对比 | Redis | Memcached |
|------|-------|-----------|
| 数据结构 | String、Hash、List、Set、ZSet 等 | 仅 String（KV） |
| 持久化 | RDB + AOF | 不支持 |
| 高可用 | 哨兵、Cluster | 无（需客户端实现） |
| 单线程/多线程 | 单线程（6.0 后 I/O 多线程） | 多线程 |
| 适用场景 | 缓存、消息队列、分布式锁等 | 纯缓存场景 |
