# 3.4 分布式事务理论

## 概述

分布式事务是指一次业务操作跨越多个微服务/数据库，需要保证这些数据操作要么全部成功、要么全部失败，满足 ACID 特性。在微服务架构中是面试高频考点。

## 核心概念

### 本地事务的局限

单机环境下，Spring 的 `@Transactional` 基于数据库本地事务即可保证一致性。但在微服务架构中，每个服务有独立数据库，本地事务无法跨库回滚，因此需要分布式事务方案。

### 分布式事务四大模式

| 模式 | 一致性 | 性能 | 实现复杂度 | 典型场景 |
|------|--------|------|-----------|---------|
| 2PC/XA | 强一致 | 低 | 低 | 传统金融、银行系统 |
| TCC | 最终一致 | 高 | 高 | 支付、转账等核心业务 |
| SAGA | 最终一致 | 中 | 中 | 长流程业务（如订单履约） |
| 本地消息表 | 最终一致 | 高 | 低 | 对实时性要求不高的场景 |

## 常见面试题

### Q1：什么是 2PC（两阶段提交）？

**答：** 2PC（Two-Phase Commit）是最经典的分布式事务协议，分为两个阶段：

**第一阶段（准备阶段/Vote 阶段）：**
- 协调者（TC）向所有参与者（RM）发送事务请求
- 各参与者执行事务但不提交，锁定资源
- 参与者向协调者反馈"同意提交"或"中止"

**第二阶段（提交/回滚阶段）：**
- 若所有参与者都同意 → 协调者发送 Commit 指令，各参与者正式提交
- 若有任一参与者拒绝 → 协调者发送 Rollback 指令，各参与者回滚

```
协调者(TC)          参与者A(RM)         参与者B(RM)
    |                    |                  |
    |--- PREPARE ------->|                  |
    |--- PREPARE ------------------------->|
    |                    |                  |
    |<-- VOTE: YES ------|                  |
    |<-- VOTE: YES ------------------------|
    |                    |                  |
    |--- COMMIT -------->|                  |
    |--- COMMIT -------------------------->|
    |                    |                  |
    |<-- ACK ------------|                  |
    |<-- ACK ------------------------------|
```

**2PC 的问题：**
1. **同步阻塞**：参与者在准备阶段锁定资源，全程阻塞等待
2. **单点故障**：协调者宕机，参与者资源一直被锁定
3. **数据不一致风险**：第二阶段网络异常导致部分参与者未收到 Commit
4. **性能差**：两次网络往返，不适合高并发场景

### Q2：3PC 解决了 2PC 什么问题？

**答：** 3PC（Three-Phase Commit）在 2PC 基础上增加了 **CanCommit** 阶段，并引入超时机制：

- **CanCommit**：协调者询问参与者是否可执行事务（不执行、不锁资源）
- **PreCommit**：类似 2PC 的 Prepare，执行事务并锁资源
- **DoCommit**：正式提交

**改进点：**
- 超时自动提交：参与者超时会自动提交（而非一直阻塞），降低了单点故障影响
- 提前探测：CanCommit 阶段提前判断可行性，减少无效锁定

**仍存在的问题：** 网络分区下仍可能数据不一致，且增加了一轮通信，实际生产中很少使用 3PC。

### Q3：什么是 TCC 模式？

**答：** TCC（Try-Confirm-Cancel）是**业务层面**的两阶段提交，需要开发者手动实现三个接口：

| 阶段 | 说明 |
|------|------|
| **Try** | 预留/检查资源，如冻结资金 |
| **Confirm** | 确认执行，使用 Try 阶段预留的资源完成业务 |
| **Cancel** | 取消执行，释放 Try 阶段预留的资源 |

**TCC 执行流程（以转账为例）：**

```
Try 阶段：
  - 服务A：冻结转出账户 100 元（余额 200 → 可用 100，冻结 100）
  - 服务B：冻结转入账户 100 元（记录待入账 100）

Confirm 阶段（成功时）：
  - 服务A：扣除冻结金额（可用 100，冻结 0）
  - 服务B：入账（余额增加 100，待入账清除）

Cancel 阶段（失败时）：
  - 服务A：解冻（余额恢复 200，冻结 0）
  - 服务B：清除待入账记录
```

> [!IMPORTANT] **TCC 的核心难点**
> - 业务侵入性强，每个操作都要写 Try/Confirm/Cancel 三个方法
> - **空回滚**：Try 未执行却收到 Cancel（网络丢包）→ 需记录事务状态，Try 未执行则 Cancel 直接返回
> - **幂等性**：Confirm/Cancel 可能重复调用 → 需通过事务 ID 去重
> - **悬挂**：Cancel 比 Try 先到（网络延迟）→ 需判断 Try 是否已执行

### Q4：什么是 SAGA 模式？

**答：** SAGA 是**长事务**的最终一致性方案，将一个大事务拆分为多个本地事务的有序执行，每个子事务都有对应的**补偿操作**：

- **正常流程**：T1 → T2 → T3 → ... → Tn（全部成功）
- **异常流程**：T1 → T2 → T3（失败）→ 补偿 T2 → 补偿 T1（**逆向回滚**已执行的子事务）

**适用场景：** 业务流程长、步骤多（如电商下单 → 扣库存 → 创建订单 → 扣优惠券 → 发送通知），其中某一步失败后逐个补偿回滚。

**SAGA vs TCC 对比：**

| 维度 | SAGA | TCC |
|------|------|-----|
| 资源预留 | 不预留，直接执行 | Try 阶段预留 |
| 补偿方式 | 逆向补偿（已执行的业务需要反操作） | 释放预留资源 |
| 侵入性 | 中（只需写补偿方法） | 高（需写三个方法） |
| 隔离性 | 弱（中间状态对其他事务可见） | 强（Try 阶段预留后其他事务不可用） |

### Q5：本地消息表方案是什么？

**答：** 本地消息表是一种**轻量级**的分布式事务实现，基于**消息队列 + 本地数据库表**实现最终一致性：

```
核心流程：
1. 业务操作和消息写入同一本地事务表（保证原子性）
2. 定时任务扫描本地消息表，发送到 MQ
3. 下游服务消费 MQ 消息，执行对应业务
4. 消费失败则重试，超过阈值进入死信队列人工处理
```

**优点：** 实现简单，不依赖额外组件，性能好。
**缺点：** 只适用于异步场景，无法实现强一致；需要处理消息重复消费（幂等性）。

> [!IMPORTANT] **面试常问：分布式事务方案怎么选？**
> - 需要强一致性 → 2PC/XA（但性能差，不推荐）
> - 核心业务（支付/转账）→ TCC 或 Seata AT 模式
> - 长流程业务（订单履约）→ SAGA
> - 对实时性要求不高 → 本地消息表 / MQ 事务消息
> - **最佳实践：优先用最终一致性，避免强一致性方案**
