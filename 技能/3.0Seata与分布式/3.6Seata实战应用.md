# 3.6 Seata 实战应用

## 概述

本笔记记录 Seata AT 模式在 Spring Cloud 微服务中的完整集成流程和常见问题。

## 环境准备

### Seata Server 部署

```bash
# 1. 下载 Seata Server（以 1.7.0 为例）
# 2. 修改 conf/registry.conf，配置注册中心（以 Nacos 为例）
registry {
  type = "nacos"
  nacos {
    application = "seata-server"
    serverAddr = "127.0.0.1:8848"
    group = "SEATA_GROUP"
    namespace = ""
  }
}

config {
  type = "nacos"
  nacos {
    serverAddr = "127.0.0.1:8848"
    group = "SEATA_GROUP"
    namespace = ""
  }
}

# 3. 启动 seata-server
sh bin/seata-server.sh
```

## Spring Cloud 集成

### Maven 依赖

```xml
<!-- Seata 分布式事务 -->
<dependency>
    <groupId>io.seata</groupId>
    <artifactId>seata-spring-boot-starter</artifactId>
    <version>1.7.0</version>
</dependency>
<!-- Seata 与 Spring Cloud 集成（数据源代理） -->
<dependency>
    <groupId>com.alibaba.cloud</groupId>
    <artifactId>spring-cloud-starter-alibaba-seata</artifactId>
</dependency>
```

### 配置文件（application.yml）

```yaml
seata:
  enabled: true
  application-id: order-service         # 当前服务名称
  tx-service-group: my_test_tx_group    # 事务分组，需与 Seata Server 配置一致
  service:
    vgroup-mapping:
      my_test_tx_group: default         # 事务分组映射到 Seata Server 集群
    grouplist:
      default: 127.0.0.1:8091          # Seata Server 地址
  config:
    type: file                          # 简化版用 file，生产环境用 nacos
  registry:
    type: file
  data-source-proxy-mode: AT            # 使用 AT 模式
```

### 每个业务库创建 undo_log 表

```sql
-- AT 模式必须，每个涉及分布式事务的库都要有这张表
CREATE TABLE `undo_log` (
  `id`            BIGINT(20)   NOT NULL AUTO_INCREMENT,
  `branch_id`     BIGINT(20)   NOT NULL,
  `xid`           VARCHAR(100) NOT NULL,
  `context`       VARCHAR(128) NOT NULL,
  `rollback_info` LONGBLOB     NOT NULL,
  `log_status`    INT(11)      NOT NULL,
  `log_created`   DATETIME     NOT NULL,
  `log_modified`  DATETIME     NOT NULL,
  PRIMARY KEY (`id`),
  UNIQUE KEY `ux_undo_log` (`xid`, `branch_id`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

## 核心代码示例

### 场景：电商下单（订单服务 → 库存服务 → 账户服务）

**订单服务（TM，事务发起方）：**

```java
@Service
public class OrderService {

    @Autowired
    private StockFeignClient stockFeignClient;
    @Autowired
    private AccountFeignClient accountFeignClient;
    @Autowired
    private OrderMapper orderMapper;

    /**
     * 全局事务入口：一个注解搞定分布式事务
     */
    @GlobalTransactional(name = "createOrder", rollbackFor = Exception.class)
    public void createOrder(OrderDTO orderDTO) {
        // 1. 创建订单（状态：待支付）
        Order order = new Order();
        order.setUserId(orderDTO.getUserId());
        order.setProductId(orderDTO.getProductId());
        order.setCount(orderDTO.getCount());
        order.setAmount(orderDTO.getAmount());
        order.setStatus(0);
        orderMapper.insert(order);

        // 2. 远程调用扣减库存
        stockFeignClient.deduct(orderDTO.getProductId(), orderDTO.getCount());

        // 3. 远程调用扣减账户余额
        accountFeignClient.deduct(orderDTO.getUserId(), orderDTO.getAmount());

        // 4. 更新订单状态为已支付
        order.setStatus(1);
        orderMapper.updateById(order);

        // 如果第3步抛出异常，所有操作（1、2、3）自动回滚
    }
}
```

**库存服务（RM）：**

```java
@Service
public class StockService {

    @Autowired
    private StockMapper stockMapper;

    /**
     * RM 不需要加任何 Seata 注解
     * Seata 通过 MyBatis 拦截器自动代理数据源
     */
    public void deduct(Long productId, Integer count) {
        int rows = stockMapper.deduct(productId, count);
        if (rows <= 0) {
            throw new RuntimeException("库存不足");
        }
    }
}
```

**账户服务（RM）：**

```java
@Service
public class AccountService {

    @Autowired
    private AccountMapper accountMapper;

    public void deduct(Long userId, BigDecimal amount) {
        int rows = accountMapper.deduct(userId, amount);
        if (rows <= 0) {
            throw new RuntimeException("余额不足");
        }
    }
}
```

## 常见问题与解决方案

### Q1：Seata 的 AT 模式如何保证隔离性？

**答：** AT 模式通过**全局锁**实现写隔离：

1. 一阶段提交前，RM 向 TC 申请该行数据的全局锁
2. 如果全局锁被其他全局事务占用，则进入重试（默认重试 30 次，间隔 10ms）
3. 超过重试次数则抛出 LockConflictException，触发当前事务回滚

**读隔离：** 默认不加读锁，可能存在脏读。需要强隔离时可升级为 TCC 模式。

### Q2：全局事务超时怎么处理？

**答：** Seata 默认全局事务超时时间为 60 秒，可配置：

```yaml
seata:
  client:
    tm:
      default-global-transaction-timeout: 60000  # 60秒
```

**超时后的行为：**
- TC 主动回滚所有已注册的分支事务
- 已提交但 undo_log 未删除的分支，由 Seata Server 异步处理

**避免超时的措施：**
- 合理设置超时时间（根据业务链路最长耗时）
- 远程调用加超时控制（Feign 配置 connectTimeout / readTimeout）
- 避免在事务中执行耗时操作（如大查询、外部 HTTP 调用）

### Q3：Seata 与本地事务 @Transactional 的关系？

**答：** 两者可以共存，职责不同：

| 注解 | 作用范围 | 层级 |
|------|---------|------|
| `@Transactional` | 单个服务的本地数据库事务 | 本地事务 |
| `@GlobalTransactional` | 跨多个服务的分布式事务 | 全局事务 |

**最佳实践：** 在 `@GlobalTransactional` 方法内部调用本地 Service 时，本地 Service 的 `@Transactional` 仍然生效（作为分支事务的一部分）。

### Q4：Seata AT 模式有哪些限制？

**答：**
1. **不支持跨库 JOIN 查询**（每个库独立）
2. **DDL 语句不受 Seata 管理**（建表、改字段无法回滚）
3. **嵌套全局事务**：一个 `@GlobalTransactional` 方法调用另一个 `@GlobalTransactional` 方法时，后者会加入前者的全局事务
4. **异步调用不支持**：`@Async` 或 MQ 异步场景无法直接使用 Seata
5. **大数据量批量操作**：undo_log 体积大，影响性能，建议分批处理

### Q5：Seata 高可用部署方案？

**答：** 生产环境 Seata Server 必须集群部署：

```yaml
# seata-server 集群配置（以 Nacos 为例）
# 多实例启动 seata-server，自动注册到 Nacos
# application.yml 中指向 Nacos 集群
seata:
  registry:
    type: nacos
    nacos:
      server-addr: nacos-cluster:8848  # Nacos 集群地址
  store:
    mode: db  # 集群模式必须用 db 存储事务日志，不能用 file
    db:
      datasource: druid
      url: jdbc:mysql://mysql-cluster:3306/seata
      user: root
      password: root
```
