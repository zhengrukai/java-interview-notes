# 3.5 Seata 核心原理

## 概述

Seata（Simple Extensible Autonomous Transaction Architecture）是阿里开源的**分布式事务解决方案**，2019 年捐赠给 Apache 基金会。它提供了一站式分布式事务服务，是目前 Java 微服务架构中**最主流**的分布式事务框架。

> [!IMPORTANT] **Seata 的核心设计目标**
> 让分布式事务的使用像本地事务一样简单：一个 `@GlobalTransactional` 注解搞定。

## Seata 三大组件

| 组件 | 全称 | 职责 |
|------|------|------|
| **TC** | Transaction Coordinator | **事务协调器**，独立部署的服务（Seata Server），维护全局事务的运行状态 |
| **TM** | Transaction Manager | **事务管理器**，开启全局事务、提交或回滚全局事务（通常是发起方） |
| **RM** | Resource Manager | **资源管理器**，管理分支事务资源，向 TC 注册分支事务并报告状态 |

```
       TM（发起方）
         |
         | @GlobalTransactional
         v
    +----TC (Seata Server)----+
    |          |              |
    v          v              v
  RM(A)      RM(B)          RM(C)
 (订单服务) (库存服务)     (支付服务)
```

## Seata 四种事务模式

### 1. AT 模式（默认，最常用）

**核心思想：** 无侵入的自动两阶段提交，对业务代码**零侵入**，只需加 `@GlobalTransactional` 注解。

#### 两阶段流程

**第一阶段（一阶段提交）：**
1. 解析 SQL，生成**前置镜像**（before image）
2. 执行业务 SQL
3. 生成**后置镜像**（after image）
4. 将 undo_log（回滚日志）和业务数据在**同一个本地事务中提交**
5. 向 TC 注册分支事务

**第二阶段（二阶段提交/回滚）：**
- **全局提交：** TC 通知各 RM 异步删除 undo_log（性能高）
- **全局回滚：** TC 通知各 RM 根据 undo_log **反向补偿**，恢复数据

```
一阶段（同步执行）：
  1. BEGIN 本地事务
  2. 查询生成 before image
  3. 执行 UPDATE/INSERT/DELETE
  4. 查询生成 after image
  5. 组装 undo_log 记录
  6. 提交本地事务（业务数据 + undo_log 一起提交）
  7. 向 TC 注册分支事务

二阶段（TC 协调）：
  全局提交 → RM 异步删除 undo_log（快速）
  全局回滚 → RM 对比 after image，如果数据被修改则人工介入，否则用 before image 恢复
```

#### undo_log 表结构

AT 模式需要每个业务库都有一张 `undo_log` 表：

```sql
CREATE TABLE `undo_log` (
  `id`            BIGINT(20)   NOT NULL AUTO_INCREMENT,
  `branch_id`     BIGINT(20)   NOT NULL COMMENT '分支事务ID',
  `xid`           VARCHAR(100) NOT NULL COMMENT '全局事务ID',
  `context`       VARCHAR(128) NOT NULL COMMENT '上下文',
  `rollback_info` LONGBLOB     NOT NULL COMMENT '回滚信息（JSON）',
  `log_status`    INT(11)      NOT NULL COMMENT '0正常, 1防御',
  `log_created`   DATETIME     NOT NULL,
  `log_modified`  DATETIME     NOT NULL,
  PRIMARY KEY (`id`),
  UNIQUE KEY `ux_undo_log` (`xid`, `branch_id`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8;
```

#### 写隔离问题（全局锁）

AT 模式下，如果两个全局事务同时修改同一行数据：
- 第一个事务获取**全局锁**，正常执行
- 第二个事务获取全局锁失败，进入重试，超过重试次数则回滚

> [!IMPORTANT] **AT 模式的核心考点**
> 1. **如何保证 undo_log 和业务数据的一致性？** → 在同一个本地事务中提交
> 2. **二阶段回滚时数据被改了怎么办？** → 对比 after image 与当前数据，不一致则需人工介入（AT 模式的局限性）
> 3. **AT vs TCC 的区别？** → AT 无侵入但依赖 undo_log 和全局锁；TCC 性能更好但侵入性高
> 4. **适用场景：** 适合大多数业务场景，但不适合高并发竞争写同一行数据的场景

### 2. TCC 模式

Seata 的 TCC 模式是**手动实现**的 Try/Confirm/Cancel 逻辑，与理论上的 TCC 一致：

```java
@LocalTCC
public interface AccountTCCService {

    // Try：冻结资金
    @TwoPhaseBusinessAction(name = "deductTCC", commitMethod = "confirm", rollbackMethod = "cancel")
    boolean deduct(@BusinessActionContextParameter(paramName = "userId") String userId,
                   @BusinessActionContextParameter(paramName = "amount") BigDecimal amount);

    // Confirm：确认扣款
    boolean confirm(BusinessActionContext context);

    // Cancel：解冻资金（回滚）
    boolean cancel(BusinessActionContext context);
}
```

**适用场景：** 性能要求高、能接受业务侵入的核心交易场景。

### 3. SAGA 模式

Seata 的 SAGA 模式通过状态机定义长事务流程：

```json
{
  "Name": "orderSaga",
  "StartState": "CreateOrder",
  "States": {
    "CreateOrder": {
      "Type": "ServiceTask",
      "ServiceName": "orderService",
      "ServiceMethod": "create",
      "CompensateState": "compensateCreateOrder",
      "Next": "DeductStock"
    },
    "DeductStock": {
      "Type": "ServiceTask",
      "ServiceName": "stockService",
      "ServiceMethod": "deduct",
      "CompensateState": "compensateDeductStock",
      "Next": "Payment"
    },
    "Payment": {
      "Type": "ServiceTask",
      "ServiceName": "paymentService",
      "ServiceMethod": "pay",
      "Next": "Finish"
    }
  }
}
```

**适用场景：** 长流程、多系统编排的业务（如订单履约、供应链流程）。

### 4. XA 模式

Seata 对标准 XA 协议的封装，利用数据库自身的 XA 事务能力：

```java
// 使用方式与 AT 模式相同，只需指定模式
@GlobalTransactional
public void orderCreate() {
    // 业务代码
}
```

```
XA 模式流程：
  TM 向 TC 发起全局事务
  ↓
  TC 通知 RM 执行 XA START → 执行 SQL → XA END → XA PREPARE
  ↓
  所有 RM PREPARE 成功后，TC 通知 XA COMMIT / XA ROLLBACK
```

> **XA vs AT：** XA 是强一致性（阻塞），AT 是最终一致性（非阻塞）。XA 性能差，一般不推荐使用。

## 四种模式对比

| 维度 | AT 模式 | TCC 模式 | SAGA 模式 | XA 模式 |
|------|---------|---------|----------|---------|
| 业务侵入 | 无侵入 | 高（三方法） | 中（补偿方法） | 无侵入 |
| 一致性 | 最终一致 | 最终一致 | 最终一致 | 强一致 |
| 性能 | 中 | 高 | 中 | 低 |
| 隔离性 | 全局锁保证 | 业务预留保证 | 无隔离 | XA 协议保证 |
| 实现难度 | 低 | 高 | 中 | 低 |
| 推荐场景 | 通用业务 | 核心交易 | 长流程 | 传统金融 |

> [!IMPORTANT] **面试必问：Seata 的 AT 模式和 TCC 模式怎么选？**
> - 80% 场景用 AT 模式（零侵入、简单）
> - 高并发核心场景用 TCC（性能好、可控性强）
> - 不要用 XA（性能太差）
> - 长流程用 SAGA（编排方便）
