### 定义

##### 1. 一致性(Consistency)

所有节点在同一时刻看到的数据完全相同，读操作必须返回最新写入值

##### 2. 可用性(Availability)

非故障节点总能在有限时间内返回非错误响应（不保证数据最新）

##### 3. 分区容错性(Partition Tolerance)

网络分区（节点通信中断）时系统仍能继续运行

### 核心结论
- 分布式系统**必须选择 P**（网络不可靠，分区一定会发生，无法避免）。
- 所以实际只有两种选择:
    - **CP**：保证一致性 + 分区容错，牺牲可用性
    - **AP**：保证可用性 + 分区容错，牺牲一致性

---

## BASE 理论（CAP 的延伸）

BASE 是对 AP 系统的实践指导，核心思想是**即使无法做到强一致性，每个应用可以根据自身业务特点采用适当的方式达到最终一致性**。

| 概念 | 说明 |
|------|------|
| **BA** (Basically Available) | 基本可用：分布式系统部分节点故障时，仍能保证核心功能可用（如电商大促时降级非核心功能） |
| **S** (Soft State) | 软状态：允许系统存在中间状态（如订单"支付中"），该状态不影响系统整体可用性 |
| **E** (Eventually Consistency) | 最终一致性：经过一段时间后，所有节点的数据会达到一致状态 |

## 常见面试题

### Q1：CP 和 AP 各有哪些代表？

**答：**
- **CP 系统：** ZooKeeper、Etcd、HBase、MongoDB（默认配置）、Seata（XA 模式）
  - 分区发生时拒绝写入或返回旧数据，保证数据一致性
- **AP 系统：** Eureka、Cassandra、DynamoDB、Redis（集群）
  - 分区发生时仍接受读写，但可能出现数据不一致

### Q2：MySQL 和 Redis 在 CAP 中属于哪种？

**答：**
- **MySQL 主从复制**：偏向 CP。主库写入后同步到从库，网络分区时可能拒绝写入或产生数据延迟
- **Redis**：偏向 AP。集群模式下各节点独立响应请求，允许短暂数据不一致

### Q3：BASE 理论在分布式事务中如何落地？

**答：** 所有最终一致性的分布式事务方案都基于 BASE 理论：
- Seata AT 模式：通过 undo_log 补偿实现最终一致性
- MQ 事务消息：消息重试 + 消费者幂等实现最终一致性
- TCC 模式：Confirm/Cancel 保证最终状态正确
