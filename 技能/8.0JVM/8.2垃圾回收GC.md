#### 一、对象存活判断方法

| 方法    | 原理                                                   | 缺陷 / 优势                         |
| ----- | ---------------------------------------------------- | ------------------------------- |
| 引用计数法 | 对象被引用 + 1，引用失效 - 1，计数器 = 0 则可回收                      | 缺陷：无法解决循环引用（A→B，B→A，无外部引用仍无法回收） |
| 可达性分析 | 以 GC Roots（虚拟机栈引用、静态变量、Native 方法引用等）为起点，遍历引用链，无链则可回收 | 优势：解决循环引用，JVM 实际使用的方法           |

#### 二、垃圾收集算法

| 算法      | 核心步骤                     | 优缺点                    | 应用场景                         |
| ------- | ------------------------ | ---------------------- | ---------------------------- |
| 标记 - 清除 | 标记可回收对象 → 直接清除           | 优点：简单；缺点：效率低、内存碎片      | CMS 收集器（老年代）                 |
| 标记 - 复制 | 内存分两块，复制存活对象到另一块 → 清空原块  | 优点：无碎片、效率高；缺点：内存利用率低   | 新生代（优化为 8:1:1 比例，仅牺牲 10% 内存） |
| 标记 - 整理 | 标记存活对象 → 向内存一端移动 → 清空另一端 | 优点：无碎片、利用率 100%；缺点：效率低 | 老年代（Serial Old/Parallel Old） |

#### 三、常见垃圾收集器

*   Parallel GC (并行 GC)

> 使用区域：新生代 + 老年代
> 优点：超高吞吐量、支持量化的调优目标（↓）、新生代回收效率极高（复制算法）
>
> *   \-XX\:MaxGCPauseMillis=&lt;N&gt;：指定 GC 最大停顿时间（毫秒），JVM 会优先保证停顿不超过该值；
> *   \-XX\:GCTimeRatio=&lt;N&gt;：指定吞吐量目标（取值 1\~99），例如 GCTimeRatio=99 表示 GC 耗时占比不超过 1%。
>
> 缺点：全程 STW，无并发回收阶段、Full GC 停顿时间随堆内存增大而显著增加
> 核心特点：多线程、关注吞吐量（用户代码运行时间占比），JDK8 默认
> 适用场景：后台批处理、数据计算

*   CMS (Concurrent Mark Sweep, 并发标记清除)

> 使用区域：老年代
> 核心特点：并发收集（低 STW）、标记 - 清除算法，步骤：初始标记（STW）→并发标记→重新标记（STW）→并发清除
> 优点：极致低停顿，响应速度快
> 适用场景：Web 应用（响应时间优先）
> 缺点：① 标记 - 清除算法产生内存碎片；② 并发阶段占用 CPU，影响用户线程；③ 无法处理 “浮动垃圾”（并发标记时产生的新垃圾，需下次 GC 回收）。

*   G1

> 使用区域：全堆（新生代 + 老年代）
> 优点：
>
> *   通过-XX\:MaxGCPauseMillis=&lt;N&gt;参数支持精确控制停顿目标（默认 200ms）
> *   大堆内存高效管理（Region 分区优势）
> *   内存碎片少，无 Concurrent Mode Failure 风险, 采用复制算法进行 Region 回收
> *   混合回收（Mixed GC）机制
> *   分代与并发结合
>     核心特点：分区式、优先回收垃圾多的区域，支持可预测 STW 时间，JDK9 默认
>     适用场景：大内存、低延迟场景（如电商核心接口）

#### 四、STW(Stop-The-World) 全局停顿

是 JVM 在执行垃圾回收（GC）等关键操作时，暂停所有应用程序线程（User Thread），仅保留 GC 线程运行的一种机制。此时，Java 应用程序会短暂 "冻结"，无法响应任何用户请求，直到 GC 操作完成后才恢复应用线程执行。

##### 关键特性

*   全局影响：所有 Java 代码停止执行，native 代码可继续运行但不能与 JVM 交互
*   自动触发：由 JVM 后台自动发起和完成，用户无感知但会影响系统性能
*   时间差异：不同 GC 算法的 STW 时长差异巨大（从毫秒级到秒级不等）

##### 核心原因

STW 的根本目的是**保证内存状态的一致性**，确保 GC 过程中对象可达性分析的准确性

#### 五、GC触发时机

*   **Minor GC(新生代垃圾回收)**：新生代 Eden 区满时触发，频率高、STW 时间短；
*   **Major GC(老年代垃圾回收)**：老年代空间不足时触发，常伴随 Minor GC，STW 时间较长；
*   **Full GC**：整堆回收（新生代 + 老年代 + 元空间），STW 最长，需尽量避免（触发场景：老年代满、元空间满、System.gc () 等）。

#### 高频问答

1.  JDK8 移除永久代的原因
    ① 永久代内存固定，易出现 OOM；② 元空间使用本地内存，默认不限制，更灵活；③ 永久代与 GC 耦合度高，元空间简化了 JVM 内存管理。
