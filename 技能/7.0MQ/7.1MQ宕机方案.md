思路分为**即时自保**和**数据补偿**两个层面

### 1.  第一时间：拦截流量，防止数据丢失（自保）

当 MQ 宕机时，生产者（发送消息的代码）会抛出异常。

*   **同步回滚 Redis**： 在代码的 catch 块中，如果捕获到 MQ 发送失败，我必须立即执行 Redis 的反向补偿（即把扣掉的库存和积分加回去）。然后给用户返回‘系统繁忙’。
*   **为什么要回滚？** 因为如果 MQ 挂了我们还让 Redis 继续扣，就会造成 Redis 和数据库的数据严重脱节，后续人工修复的成本极高。
*   **熔断机制**： 同样利用我们在处理 Redis 宕机时的逻辑，如果 MQ 连续发送失败，说明中间件集群整体不可用，系统会自动触发熔断，短时间内不再尝试兑换请求，保护系统不卡死在等待 MQ 响应上。

### 2.  第二时间：利用‘本地消息表’兜底（高可用方案）

为了防止‘Redis 回滚也失败’或者‘想在 MQ 宕机时依然让业务继续运行’，我们设计了一个本地消息表的方案：

*   **落库备份**： 当 MQ 发送异常时，我们会将这笔‘待处理的订单消息’直接持久化到数据库的一个临时表（或记录到本地日志文件）中。
*   **状态标记**： 这些记录会被标记为‘待补偿’。
*   **这样做的意义**： 虽然 MQ 挂了，但我们的核心业务数据已经安全地存在了数据库里，只是还没处理而已。

### 3.  事后：自动对账与数据修复（补偿）

等 MQ 恢复后，我们需要解决数据一致性问题：

*   **自动扫描补偿**： 我们有一个后台定时任务（Worker），它会定期扫描刚才提到的‘本地消息表’或者‘异常日志’。
*   **重发机制**： 定时任务会将这些还没发出去的消息重新投递到 MQ，直到发送成功并收到确认（ACK）后，才删除本地的临时记录。
*   **最终一致性**： 通过这种‘预扣减 + 异常持久化 + 定时任务重试’的模式，即使 MQ 宕机一小时，我们也能够保证在它恢复后，所有的兑换订单都能正确落库，不会漏掉一个用户。

**面试官可能追问的细节：**

1.  问：**如果你把错误消息写到数据库，数据库也挂了呢？**
    答： “如果 Redis、MQ、数据库全挂了，那说明机房断电或整个基础设施瘫痪了。这种极端情况下，我会利用 Logback 的告警日志 将请求参数打印到本地磁盘文件。磁盘文件是最后的防线。等环境恢复后，我们可以通过写个简单的 Python 脚本解析日志，人工恢复数据。作为架构设计，我们要考虑的是‘单点故障’，而不是‘全球毁灭’。”（这个回答带一点幽默，但也体现了你的严谨）
2.  问：**MQ 恢复后，重发消息导致“重复消费”怎么办？**
    答： “这非常关键。我们必须保证消费端的幂等性。在消费端（落库代码）执行前，我们会先查一下数据库里是否已经存在该‘业务流水号’（ID）。如果已经存在，说明这条消息处理过了，直接丢弃并返回成功。这样即便 MQ 因为网络原因重发了多次，也不会导致用户兑换了两次图书。”
3.  问：**为什么不直接在 MQ 宕机时同步写库？**
    答： “不能这样做。因为我们之前的压测结论是：数据库只能抗 500 QPS。如果 MQ 挂了我们改用同步写库，那 5000 QPS 的流量会瞬间把数据库压垮。弃车保帅，宁可让业务处理变慢或者报错，也要保护数据库不宕机。”
4.  问：**Skywalking 能发现 MQ 挂了吗？**
    答： “可以。在 Skywalking 的拓扑图中，如果 MQ 集群挂了，指向 MQ 的连线会变成红色，且报错率（Error Rate）会飙升。我会配置 Skywalking 的 Alarm（告警），通过钉钉或邮件第一时间通知我，通常在用户还没大规模感知前，我们就已经介入了。”
