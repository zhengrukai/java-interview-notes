# RabbitMQ

### 一、 核心概念与架构 (Big Picture)

首先用一句话概括：**RabbitMQ 是一个基于 Erlang 语言开发，实现了 AMQP（高级消息队列协议）的开源消息代理软件。**

你需要脑补一张图：`Producer -> Exchange -> Queue -> Consumer`

#### 1. 核心组件

*   **Producer (生产者)**：发送消息的应用。
*   **Consumer (消费者)**：接收并处理消息的应用。
*   **Broker (消息代理)**：RabbitMQ 服务节点本身。
*   **Virtual Host (虚拟主机)**：逻辑隔离概念（类似数据库中的“库”），不同VHost下的Exchange和Queue互相不可见，权限控制的基本单位。
*   **Channel (信道)**：**面试重点**。它是建立在TCP连接上的虚拟连接。
    *   *为什么要用信道？* TCP连接的创建和销毁开销很大。RabbitMQ通过复用一条TCP连接来实现多线程并发（多路复用），提高性能。

#### 2. 消息流转模型

*   **Exchange (交换机)**：接收生产者发送的消息，根据**Routing Key (路由键)** 和 **Binding Key (绑定键)** 的规则，将消息路由到指定的队列中。**（注意：生产者从不直接把消息发给队列）**
*   **Queue (队列)**：存储消息的容器。
*   **Binding (绑定)**：Exchange 和 Queue 之间的关联规则。

***

### 二、 四大交换机类型 (Exchange Types)

这是面试必问的基础，需要清晰区分：

1.  **Direct Exchange (直连模式)**：
    *   **原理**：完全匹配。Routing Key 必须与 Binding Key 完全一致。
    *   *场景*：点对点传输，明确的业务路由。
2.  **Fanout Exchange (扇形/广播模式)**：
    *   **原理**：忽略 Routing Key，将消息发送给所有绑定到该 Exchange 的队列。
    *   *场景*：广播消息，如配置更新、群发通知。速度最快。
3.  **Topic Exchange (主题模式)**：
    *   **原理**：模糊匹配。支持通配符 `*` (匹配一个单词) 和 `#` (匹配零个或多个单词)。
    *   *场景*：复杂的发布/订阅路由，如 `log.error` 和 `log.info`。
4.  **Headers Exchange (头模式)**：
    *   **原理**：根据消息的 Header 属性进行匹配，不看 Routing Key。
    *   *场景*：很少使用，性能较差。

***

### 三、 消息可靠性 (如何保证消息不丢失？)

这是**最高频**的面试题，必须分三个阶段回答：

#### 1. 生产者端 (发送可靠性)

*   **Confirm 机制 (发布确认)**：生产者开启 Confirm 模式。消息成功到达 Broker 后，RabbitMQ 会回传一个 ACK 给生产者。如果超时或失败，生产者可以重试。
*   *(备选方案：事务机制，但性能太差，不推荐)*

#### 2. Broker 端 (存储可靠性)

*   **持久化 (Persistence)**：需要做三层持久化：
    1.  **Exchange 持久化** (`durable=true`)
    2.  **Queue 持久化** (`durable=true`)
    3.  **Message 持久化** (发送消息时设置 `delivery_mode=2`)

#### 3. 消费者端 (消费可靠性)

*   **手动 ACK (Manual Acknowledgement)**：关闭自动 ACK。消费者处理完业务逻辑后，手动调用 `channel.basicAck()`。
*   如果处理失败，可以调用 `basicNack` 并设置 `requeue=true` 让消息重回队列，或者进入死信队列。

***

### 四、 高级特性与常见场景

#### 1. 死信队列 (DLX - Dead Letter Exchange)

消息在什么情况下会变成“死信”？

1.  消息被拒绝 (Reject/Nack) 且 `requeue=false`。
2.  消息过期 (TTL - Time To Live)。
3.  队列达到最大长度。

*   **应用场景**：异常数据处理、**延时队列**（利用 TTL + DLX 实现，RabbitMQ 原生不支持延时队列，这是经典实现方式）。

#### 2. 消费端限流 (QoS)

*   **原理**：避免瞬时大流量把消费者压垮。
*   **实现**：`channel.basicQos(prefetchCount)`。比如设置为 1，代表消费者处理完一条并 ACK 之前，Broker 不会再推新消息给它。

#### 3. 消息积压怎么办？

*   **紧急方案**：
    1.  修复消费者 bug。
    2.  扩容消费者实例。
    3.  如果积压过于严重，建立新的临时 Queue，写一个临时转发程序，把积压消息转到新 Queue，开启多倍的消费者快速处理（不处理业务，只存库或简单处理）。

***

### 五、 集群与高可用 (Clustering)

#### 1. 普通集群模式

*   **原理**：元数据（Metadata，如队列结构、交换机配置）在所有节点同步，但**消息实体只存储在主节点**。
*   **缺点**：如果主节点挂了，消息就丢了或暂时不可访问。

#### 2. 镜像队列模式 (Mirror Queue) - *高可用方案*

*   **原理**：将队列的消息数据同步镜像到集群中的其他节点。如果 Master 挂了，自动推举最老的 Slave 成为新的 Master。
*   *面试加分项：可以提一下 Quorum Queues (仲裁队列)，这是 RabbitMQ 3.8+ 引入的基于 Raft 协议的队列，比镜像队列性能更好，数据更安全。*

***

### 六、 面试速记口诀 (总结)

如果面试官问：“请简单介绍一下 RabbitMQ 的原理”，你可以这样组织语言：

> RabbitMQ 是一个基于 AMQP 协议的消息中间件。
>
> 1.  **架构上**，它主要由生产者、Exchange（交换机）、Queue（队列）和消费者组成。生产者发消息给 Exchange，Exchange 根据 Routing Key 路由到 Queue，消费者从 Queue 消费。
> 2.  **路由机制上**，主要有 Direct（精准）、Fanout（广播）、Topic（通配符）三种模式。
> 3.  **可靠性方面**，通过生产者的 Confirm 机制、Broker 的持久化（交换机、队列、消息）、以及消费者的手动 ACK 来保证消息不丢失。
> 4.  **业务场景上**，常用于解耦、异步处理和流量削峰。对于延时任务，我们通常结合 TTL 和死信队列来实现。

# RocketMQ

面试 RocketMQ 时，重点要转移到 **“高性能存储设计”、“分布式架构”** 以及它独有的 **“事务消息”** 上。RocketMQ 参考了 Kafka 的设计，但在功能丰富度和低延迟上做了针对业务场景（特别是电商）的优化。

以下是为你整理的 RocketMQ 面试核心原理：

***

### 一、 核心架构 (The Big Picture)

RocketMQ 是一个 **纯 Java 开发**、**低延迟、高吞吐** 的分布式消息中间件。它的架构主要分为四部分：

1.  **NameServer (注册中心)**：
    *   *作用*：类似于 Zookeeper，但更轻量。Broker 启动时向 NameServer 注册，Producer/Consumer 从 NameServer 获取 Broker 的路由信息。
    *   *特点*：**无状态节点**，节点之间不通信。这与 Zookeeper 的强一致性不同，NameServer 是 AP 模型（可用性优先），性能极高。
2.  **Broker (核心节点)**：
    *   *作用*：负责消息存储、投递和查询。是核心负重组件。
    *   *结构*：支持 Master-Slave 模式。Master 可读写，Slave 只读（做备份和容灾）。
3.  **Producer (生产者)**：支持同步发送、异步发送、单向发送。
4.  **Consumer (消费者)**：支持推（Push）和拉（Pull）模式。
    *   *注意*：RocketMQ 的 Push 本质上是 **长轮询 (Long Polling)** 实现的。

***

### 二、 存储模型 (最核心原理 - 面试必问)

这是 RocketMQ 与 Kafka 最大的区别，也是它支持“海量 Topic”的关键。

#### 1. 文件结构

*   **CommitLog (消息主体)**：
    *   **原理**：**所有 Topic 的消息全部顺序写入同一个文件（CommitLog）**。
    *   *优势*：**顺序写盘**，速度极快（利用 OS 的 PageCache），避免了磁盘随机 I/O。
*   **ConsumeQueue (逻辑队列/索引)**：
    *   **原理**：因为消息混杂在 CommitLog 里，消费者怎么找自己 Topic 的消息？靠 ConsumeQueue。它只存消息在 CommitLog 中的 **偏移量(Offset)、大小和 Tag Hash**。
    *   *优势*：极轻量级，方便消费者快速定位。
*   **IndexFile (索引文件)**：
    *   **原理**：提供通过 Key 或时间区间查询消息的能力（支持 Hash 索引）。

#### 2. 读写分离机制

*   **写消息**：直接顺序写入 CommitLog。
*   **读消息**：先读 ConsumeQueue 拿到 Offset，再去读 CommitLog 拿真实数据。
*   **零拷贝 (Zero Copy)**：底层使用 `mmap` (内存映射文件) 技术，减少 CPU 拷贝次数，提高读写性能。

***

### 三、 高级特性 (杀手锏)

面试官问 RocketMQ 原理，通常是在期待你回答 **事务消息**。

#### 1. 事务消息 (Transactional Message)

这是 RocketMQ 在金融/电商场景的看家本领，用于实现 **最终一致性**。

*   **流程**：
    1.  **发送半消息 (Half Message)**：Producer 发送一条消息给 Broker，但标记为“暂不能投递”。
    2.  **执行本地事务**：Producer 执行自己的数据库操作（如订单入库）。
    3.  **提交/回滚**：
        *   本地事务成功 -> 发送 Commit -> Broker 把消息变为可见。
        *   本地事务失败 -> 发送 Rollback -> Broker 删除/忽略该消息。
    4.  **回查机制 (补偿)**：如果 Broker 迟迟没收到 Commit/Rollback（比如网络断了），Broker 会主动询问 Producer：“刚才那个事务到底成功没？” Producer 检查数据库后反馈。

#### 2. 延时消息 (Scheduled Message)

*   **原理**：不像 RabbitMQ 需要死信队列，RocketMQ 自带延时功能。
*   **限制**：不支持任意时间，只支持预设的 18 个等级（1s, 5s, 10s... 2h）。
*   **实现**：Broker 收到延时消息后，先把消息暂存在一个名为 `SCHEDULE_TOPIC_XXXX` 的内部 Topic 中，等时间到了，再把消息“还原”到真实的 Topic 给消费者。

#### 3. 消息有序性

*   **全局有序**：一个 Topic 只有一个 Queue（性能低，少用）。
*   **分区有序 (Partition Order)**：通过 Sharding Key（如 OrderId），把同一笔订单的消息都发到同一个 Queue，保证局部有序（常用）。

***

### 四、 高可用与刷盘机制

#### 1. 刷盘策略 (数据怎么存磁盘)

*   **同步刷盘 (Sync Flush)**：消息写入内存后，强制刷入磁盘才返回 ACK。**数据最安全，但慢**。
*   **异步刷盘 (Async Flush)**：写入内存就返回 ACK，后台线程慢慢刷盘。**吞吐量高，断电可能丢少量数据**。

#### 2. 复制策略 (Master-Slave 同步)

*   **同步复制**：Master 和 Slave 都写成功才返回成功。
*   **异步复制**：Master 写成功就返回，异步传给 Slave。

#### 3. Dledger (Raft 协议) - *加分项*

*   在 RocketMQ 4.5 以后，引入了 Dledger 组件，基于 **Raft 协议** 实现了自动选主（Auto Failover）。解决了老版本 Master 挂了需要人工切换的问题。

***

### 五、 面试对比：RocketMQ vs Kafka vs RabbitMQ

如果在面试中能主动对比，说明你理解深刻：

| 特性       | RabbitMQ         | RocketMQ                   | Kafka            |
| :------- | :--------------- | :------------------------- | :--------------- |
| **开发语言** | Erlang (难改源码)    | Java (容易二次开发)              | Scala/Java       |
| **吞吐量**  | 万级               | **十万级**                    | 百万级              |
| **时效性**  | 微秒级 (最快)         | 毫秒级                        | 毫秒级              |
| **可用性**  | 主从架构             | 主从/Dledger                 | 分布式分区            |
| **核心优势** | **灵活路由**、成熟、社区活跃 | **事务消息**、**支持海量Topic**、低延迟 | **超高吞吐**、大数据日志采集 |
| **缺点**   | 吞吐量稍低，Erlang 难维护 | 生态不如 Kafka/RabbitMQ 广泛     | 实时性稍差，Topic 太多会慢 |

***

### 六、 面试速记口诀 (总结)

如果面试官问：“请说说 RocketMQ 的原理”，你可以这样回答：

> RocketMQ 是阿里开源的、基于 Java 的分布式消息中间件，主要为了解决电商业务中的高并发和分布式事务问题。
>
> 1.  **架构上**，它是“NameServer + Broker + 生产者 + 消费者”的模型。NameServer 轻量级且无状态，Broker 负责存储。
> 2.  **存储设计上（核心）**，它采用了 **CommitLog 顺序写** 的方式，所有 Topic 的消息都存在一个大文件里，极大提高了写入速度；同时通过 ConsumeQueue 做逻辑索引来提高读取效率。这让它能支持海量的 Topic 而不掉速。
> 3.  **特性上**，它最大的亮点是 **事务消息**，通过“半消息+回查机制”实现了分布式系统的最终一致性。此外还支持延时消息和顺序消息。
> 4.  **可靠性上**，支持同步/异步刷盘和主从复制，配合 Dledger 可以实现自动故障切换。
