# RabbitMQ 集群突然打满 — 不重启快速扩容方案

## 🔍 第一步：快速定位问题根因

在操作前先判断是哪种"打满"：

| 类型 | 现象 | 优先处理 |
|------|------|----------|
| **消息堆积** | Queue depth 暴增，Consumer 跟不上 | 扩 Consumer |
| **内存打满** | `mem_alarm` 触发，Producer 被 block | 临时调高内存水位 |
| **磁盘打满** | `disk_free_alarm` 触发 | 清理 / 扩磁盘 |
| **连接数打满** | Connection refused | 扩节点 / 调限制 |

```bash
# 快速查看集群状态
rabbitmqctl cluster_status
rabbitmqctl list_queues name messages consumers memory
rabbitmqctl list_alarms
```

## 🚀 核心扩容操作（不重启）

### 1. 动态扩展 Consumer（最快见效）

**手动拉起更多 Consumer 实例**（K8s 环境）：
```bash
kubectl scale deployment your-consumer --replicas=20
```

**调大单 Consumer 的 prefetch（提升单机消费能力）**：
```python
# 运行时通过管理 API 无法直接改，需在 Consumer 代码中设置
channel.basic_qos(prefetch_count=500)  # 默认通常是 1，严重拖慢吞吐
```

---

### 2. 动态调整内存/磁盘水位线（解除 flow control）

```bash
# 临时调高内存水位（默认 0.4，即 40% 物理内存）
rabbitmqctl set_vm_memory_high_watermark 0.6

# 或设置绝对值
rabbitmqctl set_vm_memory_high_watermark absolute 4GB

# 调整磁盘空闲阈值（默认 50MB，调低解除磁盘告警）
rabbitmqctl set_disk_free_limit 1GB
```

> ⚠️ 这是临时手段，解除告警后要同步处理根本问题

---

### 3. 热加入新节点扩容集群

RabbitMQ 支持**不停服加入新节点**：

```bash
# 在新节点上执行
rabbitmqctl stop_app
rabbitmqctl reset
rabbitmqctl join_cluster rabbit@existing-node
rabbitmqctl start_app

# 验证
rabbitmqctl cluster_status
```

新节点加入后，通过 Policy 将队列的 Consumer 引流过去，或让 Producer 的 Load Balancer 感知新节点。

---

### 4. 对堆积队列开启临时消费加速

**方案A：通过 Management API 动态转移消息**
```bash
# 将堆积队列的消息 shovel 到多个子队列并行消费
rabbitmqctl set_parameter shovel bulk-move \
  '{"src-uri":"amqp://","src-queue":"overloaded_queue",
    "dest-uri":"amqp://","dest-queue":"overflow_queue_1"}'
```

**方案B：启用 Federation / Shovel 插件跨节点均衡**
```bash
rabbitmq-plugins enable rabbitmq_shovel
rabbitmq-plugins enable rabbitmq_shovel_management
```

---

### 5. 临时限流 Producer（保护消费端追上）

```bash
# 对某个 vhost 设置 per-vhost 消息速率限制
# 通过 Management HTTP API
curl -u admin:password -X PUT \
  http://localhost:15672/api/vhosts/your_vhost \
  -H "Content-Type: application/json" \
  -d '{"description":"","tags":"","tracing":false}'

# 或在连接层面通过 Policy 限制 Publisher confirm 频率
```

---

### 6. 清理无效/过期消息（快速降低 Queue Depth）

```bash
# 通过 HTTP API 清空某个非核心队列
curl -u admin:pass -X DELETE \
  http://localhost:15672/api/queues/%2F/queue_name/contents

# 或设置 TTL Policy 让旧消息自动过期（不影响新消息）
rabbitmqctl set_policy TTL ".*" \
  '{"message-ttl":60000}' --apply-to queues
```

## 📋 完整处置 SOP

```
1. 查告警类型（内存/磁盘/消息堆积）
       ↓
2. 内存/磁盘告警 → 调高水位线，争取时间
       ↓
3. 消息堆积 → 立即 scale consumer + 调大 prefetch
       ↓
4. 单集群处理不过来 → 热加新节点 + shovel 分流
       ↓
5. 同步限流 producer，让消费端追上
       ↓
6. 稳定后复盘：扩容 consumer 常驻实例 / 拆分队列
```

## 🛡️ 事后防御建议

- **Consumer 侧**：基于队列深度配置 HPA（K8s），自动弹性伸缩
- **队列侧**：设置 `x-max-length` + `overflow: reject-publish` 或 `drop-head`，防止无限堆积
- **监控侧**：队列深度 > 阈值时提前告警，不要等到打满再处理
- **架构侧**：核心业务队列与非核心队列隔离在不同 vhost / 节点
