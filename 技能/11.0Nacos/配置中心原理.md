Nacos（Dynamic Naming and Configuration Service）作为配置中心，核心原理可以分为几个关键机制。

## 整体架构## 核心机制一：长轮询（Long Polling）

这是 Nacos 感知配置变更的关键机制，兼顾实时性与效率。

image

长轮询的精妙之处在于：客户端发送请求时携带当前配置的 **MD5 摘要**，服务端将请求挂起最多 29.5 秒——如果期间配置有变更，立刻响应；否则超时后客户端立即重新发起轮询。这样既避免了短轮询的频繁请求，又接近推送的实时性。

## 其他关键机制

**配置隔离（命名空间 + Group + dataId）**

Nacos 用三级坐标定位一条配置：`namespace`（环境隔离，如 dev/prod）→ `group`（业务分组）→ `dataId`（具体配置文件名）。完整唯一键是这三者的组合。

**客户端本地容灾快照**

客户端第一次从服务端拉到配置后，会同步写一份到本地磁盘（`~/nacos/config/` 下的文件快照）。当 Nacos Server 完全不可用时，客户端会读取快照继续工作，保证服务不因配置中心故障而中断。

**集群一致性协议**

Nacos 2.x 对不同数据类型采用不同协议：配置数据（需要强一致）走 **Raft**，服务实例临时数据走 **Distro**（AP 协议）。配置写请求必须打到 Leader，由 Leader 写 MySQL 并广播给 Follower 节点后才算成功。

**变更通知链路总结**

> 控制台发布 → Leader 写 MySQL & 更新内存缓存 → 同步 Follower → 挂起的长轮询被唤醒 → 返回变更的 dataId → 客户端拉取新值 → 触发 Listener 回调更新本地状态

这一链路通常在 **1 秒内**完成端到端的配置生效。
