Collection 是单列集合的根接口，下分三个核心子接口：List、Set、Queue。

---

#### 1. List（有序、可重复）
List 保证元素的插入顺序，允许重复元素，支持通过索引访问。

| 实现类 | 底层结构 | 核心特点 | 适用场景 |
| --- | --- | --- | --- |
| ArrayList | 动态数组 | 查询快（O(1)），增删慢（需移动元素）；默认初始容量 10，扩容 1.5 倍 | 频繁查询、数据量稳定 |
| LinkedList | 双向链表 | 首尾增删快（O(1)），随机查询慢（O(n)）；同时实现 Deque 接口 | 频繁增删、需要队列/栈 |
| Vector | 动态数组 | 线程安全（方法加 synchronized），效率低；扩容 2 倍 | 已过时，用 CopyOnWriteArrayList 替代 |

### ArrayList 深入（高频⭐⭐⭐⭐⭐）

**扩容机制**：
```java
// 核心扩容逻辑（JDK 8）
private void grow(int minCapacity) {
    int oldCapacity = elementData.length;
    int newCapacity = oldCapacity + (oldCapacity >> 1); // 1.5 倍
    if (newCapacity < minCapacity)
        newCapacity = minCapacity;
    elementData = Arrays.copyOf(elementData, newCapacity); // 数组拷贝
}
```

**关键数字**：
- 默认初始容量：**10**（首次 add 时才真正分配，无参构造时为空数组）
- 扩容倍数：**1.5 倍**（oldCapacity + oldCapacity >> 1）
- 最大容量：`Integer.MAX_VALUE - 8`

**add(E e) 流程**：
1. `ensureCapacityInternal(size + 1)` → 检查是否需要扩容
2. 需要扩容 → `grow()` → `Arrays.copyOf()` 创建新数组并拷贝
3. `elementData[size++] = e` → 将元素放到数组末尾

**add(int index, E e) 流程**：
1. 范围检查 → 扩容检查
2. `System.arraycopy()` → 将 index 后的元素整体后移一位
3. `elementData[index] = e` → 插入元素

> 中间插入的时间复杂度为 **O(n)**（需要移动元素），所以频繁中间插入不适合 ArrayList

### ArrayList vs LinkedList（高频⭐⭐⭐⭐⭐）

| 维度 | ArrayList | LinkedList |
|------|----------|------------|
| 底层结构 | **动态数组** | **双向链表** |
| 随机访问 | **O(1)**（下标直接定位） | O(n)（需从头/尾遍历） |
| 尾部添加 | **均摊 O(1)**（偶尔扩容） | O(1) |
| 中间插入/删除 | O(n)（移动元素） | O(1)（已知节点时），但查找节点 O(n) |
| 内存占用 | 紧凑（连续内存，CPU 缓存友好） | 每个节点额外存 prev + next 指针 |
| 实现接口 | List, RandomAccess | List, **Deque** |

**结论**：**绝大多数场景选 ArrayList**。LinkedList 的理论优势在实际中被 CPU 缓存命中率碾压，只有在大量首尾操作（队列/栈）时 LinkedList 才有优势。

---

#### 2. Set（无序、不可重复）
Set 不保证元素顺序，禁止重复元素（通过 equals() 和 hashCode() 判断）。

| 实现类 | 底层结构 | 核心特点 | 适用场景 |
| --- | --- | --- | --- |
| HashSet | **HashMap**（仅存 key，value 固定为 PRESENT） | 无序，O(1) 查询/插入；元素需重写 equals + hashCode | 快速去重 |
| LinkedHashSet | LinkedHashMap | 维护**插入顺序**；性能略低于 HashSet | 有序去重 |
| TreeSet | **TreeMap**（红黑树） | **自然排序**或自定义 Comparator；O(log n) | 排序去重 |

**关键点**：HashSet 本质是 HashMap，add(e) 等价于 map.put(e, PRESENT)，利用 HashMap 的 key 不重复特性实现去重。

---

#### 3. Queue（队列，先进先出）

| 实现类 | 底层结构 | 核心特点 |
|--------|---------|---------|
| LinkedList | 双向链表 | 同时实现 List 和 Deque，可作双端队列 |
| PriorityQueue | **最小堆**（数组实现） | 按优先级出队，非 FIFO；不允许 null |
| ArrayDeque | 循环数组 | 双端队列，比 LinkedList 快（CPU 缓存友好），**推荐替代 Stack** |

**ArrayDeque vs LinkedList 作为队列/栈**：
- ArrayDeque 基于循环数组，**无需创建节点对象**，GC 压力小
- 性能优于 LinkedList，是 Java 官方推荐的栈和队列实现

### fail-fast vs fail-safe（高频⭐⭐⭐⭐）

| 维度 | fail-fast | fail-safe |
|------|----------|-----------|
| 代表 | ArrayList、HashMap、HashSet | CopyOnWriteArrayList、ConcurrentHashMap |
| 机制 | 遍历时修改结构 → 抛 **ConcurrentModificationException** | 遍历时操作**副本或分段**，不抛异常 |
| 原理 | 内部维护 `modCount`，迭代器检测到不一致即快速失败 | 写时复制 / 分段锁 |
| 性能 | 无额外开销 | 写操作有拷贝/锁开销 |

```java
// fail-fast 示例（会抛异常）
List<String> list = new ArrayList<>(Arrays.asList("a", "b", "c"));
for (String s : list) {
    if ("b".equals(s)) {
        list.remove(s); // ConcurrentModificationException!
    }
}

// 正确做法1：使用 Iterator.remove()
Iterator<String> it = list.iterator();
while (it.hasNext()) {
    if ("b".equals(it.next())) {
        it.remove(); // 安全
    }
}

// 正确做法2：使用 removeIf（JDK 8+）
list.removeIf("b"::equals);
```

### 线程安全集合选型

| 需求 | 推荐 | 不推荐 |
|------|------|--------|
| 线程安全 List | **CopyOnWriteArrayList**（读多写少） | Vector / Collections.synchronizedList |
| 线程安全 Set | CopyOnWriteArraySet / ConcurrentHashMap.newKeySet() | Collections.synchronizedSet |
| 线程安全 Queue | **ConcurrentLinkedQueue**（无锁）/ **LinkedBlockingQueue** | — |

### 高频追问

1. **ArrayList 的 elementData 为什么用 transient 修饰？** 因为数组通常未满，序列化时只序列化 `size` 个有效元素（通过自定义 writeObject），避免浪费空间
2. **Arrays.asList() 返回的 List 能 add 吗？** 不能。返回的是 `Arrays$ArrayList`（内部类），固定大小，调用 add/remove 会抛 UnsupportedOperationException
3. **Collections.unmodifiableList 和 List.of 的区别？** `unmodifiableList` 是原 List 的视图，原 List 变它也变；`List.of`（JDK 9+）创建真正不可变的 List
4. **为什么不推荐 LinkedList？** 实际场景中，ArrayList 的 CPU 缓存局部性（内存连续）远超 LinkedList 链表跳转的理论优势
5. **CopyOnWriteArrayList 适合什么场景？** **读多写极少**（如事件监听器列表），写时复制整个数组，写性能差
