## 一句话结论

Spring 通过**三级缓存**解决 **setter/字段注入**的循环依赖：一级缓存存成品 Bean，二级缓存存半成品（早期引用），三级缓存存 ObjectFactory（延迟生成代理对象）。**构造器注入的循环依赖无法解决**（Bean 还没实例化，无法提前暴露）。

---

### 什么是循环依赖

```java
@Service
public class A {
    @Autowired
    private B b;  // A 依赖 B
}

@Service
public class B {
    @Autowired
    private A a;  // B 依赖 A
}
```

创建 A → 需要注入 B → 创建 B → 需要注入 A → A 还没创建完 → **死循环**

---

### 三级缓存结构

```java
// DefaultSingletonBeanRegistry 中的三个 Map
public class DefaultSingletonBeanRegistry {

    /** 一级缓存：存放完整的、初始化完成的 Bean（成品） */
    private final Map<String, Object> singletonObjects = new ConcurrentHashMap<>(256);

    /** 二级缓存：存放早期暴露的 Bean（半成品，已实例化但未完成属性填充） */
    private final Map<String, Object> earlySingletonObjects = new ConcurrentHashMap<>(16);

    /** 三级缓存：存放 Bean 的工厂（ObjectFactory，用于生成早期引用） */
    private final Map<String, ObjectFactory<?>> singletonFactories = new HashMap<>(16);
}
```

| 缓存级别 | Map 名称 | 存储内容 | 何时放入 |
|---------|---------|---------|---------|
| 一级 | singletonObjects | **完整 Bean**（可直接使用） | Bean 完全初始化后 |
| 二级 | earlySingletonObjects | **早期引用**（已实例化，未填充属性） | 从三级缓存获取并升级时 |
| 三级 | singletonFactories | **ObjectFactory**（工厂 Lambda） | Bean 实例化后、属性填充前 |

---

### 解决流程（A 依赖 B，B 依赖 A）

```
Step 1: 创建 A
├── 实例化 A（调用构造函数，得到原始对象 a）
├── 将 A 的 ObjectFactory 放入三级缓存
│     singletonFactories.put("a", () -> getEarlyBeanReference("a", a))
├── 属性填充：发现需要 B → 去获取 B
│
Step 2: 创建 B（A 的创建暂停，先创建 B）
├── 实例化 B（得到原始对象 b）
├── 将 B 的 ObjectFactory 放入三级缓存
├── 属性填充：发现需要 A → 去获取 A
│
Step 3: B 获取 A（关键步骤！）
├── 一级缓存找 A → 没有（A 还没创建完）
├── 二级缓存找 A → 没有
├── 三级缓存找 A → ✅ 找到 ObjectFactory
│     执行 ObjectFactory.getObject()
│     → 如果 A 需要 AOP 代理，这里生成代理对象
│     → 如果不需要代理，返回原始对象
├── 将结果放入二级缓存，删除三级缓存中的 A
├── B 拿到 A 的早期引用 → B 属性填充完成
├── B 完成初始化 → 放入一级缓存
│
Step 4: 回到 A 的创建
├── A 拿到完整的 B → A 属性填充完成
├── A 完成初始化 → 放入一级缓存
├── 删除二级缓存中的 A
│
✅ 完成！A 和 B 都在一级缓存中
```

---

### 为什么需要三级缓存？两级不行吗？（高频⭐⭐⭐⭐⭐）

**如果只有两级缓存**：Bean 实例化后直接放入二级缓存。但问题是：

- 如果 A 被 AOP 代理，此时放入二级缓存的是**原始对象**还是**代理对象**？
- 如果所有 Bean 都提前创建代理 → **违反 Spring 设计原则**（代理应在 BPP 后置处理阶段生成）
- 如果不提前创建代理 → B 拿到的是原始 A，而容器中最终存的是代理 A → **不是同一个对象！**

**三级缓存的解决方案**：

```
三级缓存存的是 ObjectFactory（工厂 Lambda），不是 Bean 本身

这个工厂的逻辑是：
  如果 A 需要 AOP 代理 → 生成并返回代理对象
  如果 A 不需要代理 → 返回原始对象

只有在真正被其他 Bean 依赖时（B 获取 A 时），才执行这个工厂
→ 延迟代理创建，只在必要时提前生成
→ 保证 B 拿到的 A 和容器中最终的 A 是同一个对象
```

**一句话总结**：三级缓存的核心目的是**延迟代理对象的创建**，在有循环依赖时才提前生成代理，无循环依赖时仍在正常流程（BPP 后置处理）生成代理。

### 哪些场景无法解决？

#### 1. 构造器注入循环依赖

```java
@Service
public class A {
    public A(B b) { ... }  // 构造器注入 B
}

@Service
public class B {
    public B(A a) { ... }  // 构造器注入 A
}
// ❌ 报错：BeanCurrentlyInCreationException
```

**原因**：构造器注入发生在**实例化阶段**，此时对象还未创建，无法放入任何缓存。

**解决方案**：
```java
// 方案1：@Lazy 延迟加载（最简单）
@Service
public class A {
    public A(@Lazy B b) { ... }  // 注入的是 B 的代理，首次使用时才真正创建
}

// 方案2：改为 Setter 注入（打破构造器依赖）
@Service
public class A {
    @Autowired
    private B b;
}
```

#### 2. prototype 作用域循环依赖

```java
@Scope("prototype")
@Service
public class A {
    @Autowired
    private B b;
}
// ❌ 每次都创建新对象，无法缓存
```

**原因**：prototype 不走单例缓存，每次 getBean 都新建 → 无限递归。

#### 3. @Async 导致的循环依赖

```java
@Service
public class A {
    @Autowired
    private B b;

    @Async
    public void asyncMethod() { ... }
}
// 可能报错：BeanCurrentlyInCreationException
```

**原因**：@Async 也通过 BPP 生成代理，但在早期引用阶段和最终代理阶段可能产生两个不同的代理对象。

**解决**：`@Lazy` 注解打破循环。

### 源码关键方法

```java
// DefaultSingletonBeanRegistry.getSingleton()
protected Object getSingleton(String beanName, boolean allowEarlyReference) {
    // 1. 先查一级缓存
    Object singletonObject = this.singletonObjects.get(beanName);
    if (singletonObject == null && isSingletonCurrentlyInCreation(beanName)) {
        // 2. 再查二级缓存
        singletonObject = this.earlySingletonObjects.get(beanName);
        if (singletonObject == null && allowEarlyReference) {
            synchronized (this.singletonObjects) {
                // 3. 最后查三级缓存
                ObjectFactory<?> singletonFactory = this.singletonFactories.get(beanName);
                if (singletonFactory != null) {
                    singletonObject = singletonFactory.getObject(); // 执行工厂方法
                    this.earlySingletonObjects.put(beanName, singletonObject); // 升级到二级
                    this.singletonFactories.remove(beanName); // 删除三级
                }
            }
        }
    }
    return singletonObject;
}
```

### 全局对照表

| 注入方式 | 循环依赖 | 能否解决 | 原因 |
|---------|---------|---------|------|
| **setter / 字段注入** | A ↔ B | ✅ 能 | 三级缓存提前暴露 |
| **构造器注入** | A ↔ B | ❌ 不能 | 实例化阶段无法暴露 |
| **setter + prototype** | A ↔ B | ❌ 不能 | prototype 不走缓存 |
| **构造器 + @Lazy** | A ↔ B | ✅ 能 | 注入代理，延迟加载 |

### 高频追问

1. **为什么 Spring 默认不解决构造器注入的循环依赖？** 构造器注入时对象还不存在，无法放入缓存提前暴露。Spring 认为构造器循环依赖是**设计问题**，应该重构代码而非框架兜底
2. **如果没有 AOP，只需要两级缓存吗？** 理论上是的。没有代理的情况下，实例化后直接放二级缓存就够了。三级缓存是为了兼容 AOP 场景
3. **@Lazy 是怎么解决循环依赖的？** 注入的不是真实对象，而是一个**代理对象**。真实对象在首次方法调用时才创建，打破了创建时的循环
4. **Spring Boot 2.6+ 对循环依赖有什么变化？** 默认**禁止循环依赖**（`spring.main.allow-circular-references=false`），鼓励开发者从设计上避免循环依赖
5. **如何从设计上避免循环依赖？** 引入第三方类解耦（A→C←B）、使用事件机制（ApplicationEvent）、将共同逻辑抽取到新 Service、使用 @Lazy 打破
