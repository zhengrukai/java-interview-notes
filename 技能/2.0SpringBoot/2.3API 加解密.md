### API 加解密过滤器，用于对请求参数解密、响应结果加密

算法支持：AES（对称加密）、RSA（非对称加密）
注解驱动：通过 @ApiEncrypt 注解控制是否加解密

#### 1. 请求解密流程

    1. 获取 @ApiEncrypt 注解
    2. 检查是否需要解密（request=true 且有加密请求头）
    3. 创建 ApiDecryptRequestWrapper 包装原始请求
    4. 解密 request body，替换为解密后的内容
    5. 放行 chain.doFilter(request, 包装后的请求)

#### 2. 响应加密流程

    1. 检查是否需要加密（response=true）
    2. 创建 ApiEncryptResponseWrapper 包装原始响应
    3. 放行 chain.doFilter(request, 包装后的响应)
    4. 获取原始响应内容
    5. 加密响应内容并写回

> 1.如何保证加密/解密的安全性

*   使用 HTTPS 传输
*   密钥通过配置中心管理，不硬编码
*   支持 AES + RSA 混合加密（RSA 加密 AES 密钥）

> 2.如果加密失败怎么办？

    try {
        // 解密/加密逻辑
    } catch (Exception ex) {
        // 交给全局异常处理器处理
        CommonResult<?> result = globalExceptionHandler.allExceptionHandler(request, ex);
        ServletUtils.writeJSON(response, result);
        return;
    }

> 3.为什么 Response 只能写一次？

    HttpServletResponse.getWriter() / getOutputStream()
        ↓
    返回的是**底层流**（PrintWriter / ServletOutputStream）
        ↓
    一旦调用 flush()，数据就会发送到客户端
        ↓
    无法回退或重写



    # 具体流程
    客户端请求
        ↓
    服务器处理业务逻辑
        ↓
    controller return 后
        ↓
    Spring MVC 开始写入 Response
        ↓
    ① response.getWriter().write("原始数据")
        ↓
    ② 数据写入到 **ServletOutputStream**（底层是 TCP 流）
        ↓
    【关键】此时数据已经发送到客户端缓冲区，无法撤销！
        ↓
    如果再写一次 → 报错或覆盖

> 4.为什么需要 Wrapper 模式？

    正常流程（无加密）：
    Controller → response.getWriter().write("原始数据") → 发送到客户端

    加密流程（需要Wrapper）：
    1. 创建 ApiEncryptResponseWrapper
    2. 包装后，write() 不发送到客户端，而是写入 ByteArrayOutputStream（内存缓冲区）
    3. 业务处理完成
    4. 从缓冲区读取原始数据 → 加密 → 写入真正的 response

> 5.Request 为什么只能读一次

    # InputStream 是消耗性流
    // Request 的 body 通过 InputStream 读取
    ServletInputStream inputStream = request.getInputStream();
    // InputStream 只能读取一次，读完就"空了"



    # 流程图
    客户端发送请求（包含 body）
        ↓
    服务器接收，封装为 InputStream
        ↓
    第一次读取：inputStream.read() → 获取字节数据
        ↓
    【关键】读取后，流中的数据被"消费"掉了
        ↓
    第二次读取：inputStream.read() → 返回 -1（流已结束）
        ↓
    无法再次读取！
