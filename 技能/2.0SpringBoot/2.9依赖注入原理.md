# Spring Boot 依赖注入原理及特点

## 一、依赖注入（DI）原理

### 1. 核心思想：控制反转（IoC）

传统方式中对象**主动 new** 依赖，IoC 将控制权交给 **Spring 容器**，对象**被动接收**依赖。

    传统：  A ──new──▶ B ──new──▶ C    （强耦合）

    IoC：       ┌─────────────┐
                │  Spring 容器  │
                └──┬────┬────┬─┘
              注入↓  注入↓  注入↓
                 A     B     C          （松耦合）

### 2. 容器启动与 Bean 注册流程

    @SpringBootApplication 启动
            │
            ▼
    @ComponentScan 扫描包路径
            │
            ▼
    发现 @Component / @Service / @Repository / @Controller
            │
            ▼
    解析为 BeanDefinition（类名、作用域、依赖等元信息）
            │
            ▼
    注册到 BeanDefinitionRegistry
            │
            ▼
    实例化 Bean → 依赖注入 → 初始化回调（@PostConstruct）
            │
            ▼
    存入 IoC 容器，应用就绪

### 3. 两大容器接口

| 接口                     | 特点                                |
| ---------------------- | --------------------------------- |
| **BeanFactory**        | 底层接口，延迟加载，轻量                      |
| **ApplicationContext** | BeanFactory 子接口，预加载、支持 AOP、事件、国际化 |

### 4. 三种注入方式

#### ① 构造器注入（✅ 推荐）

```java
@Service
public class OrderService {
    private final UserService userService;

    // Spring 4.3+ 单构造器可省略 @Autowired
    public OrderService(UserService userService) {
        this.userService = userService;
    }
}
```

**优点**：依赖不可变（`final`）、启动时校验、便于测试

#### ② Setter 注入（可选依赖时用）

```java
@Service
public class OrderService {
    private UserService userService;

    @Autowired(required = false)
    public void setUserService(UserService userService) {
        this.userService = userService;
    }
}
```

#### ③ 字段注入（❌ 不推荐）

```java
@Service
public class OrderService {
    @Autowired
    private UserService userService; // 反射注入，无法 final，难测试
}
```

### 5. 自动装配匹配规则

    @Autowired 流程：

    按类型匹配 → 找到几个？
      ├─ 0个 → 抛 NoSuchBeanDefinitionException
      ├─ 1个 → 注入成功 ✅
      └─ 多个 → 按名称匹配（@Qualifier 或字段名）
                  ├─ 匹配到 → 注入成功 ✅
                  └─ 未匹配 → 抛 NoUniqueBeanDefinitionException

| 注解           | 来源      | 策略                 |
| ------------ | ------- | ------------------ |
| `@Autowired` | Spring  | **先类型**后名称         |
| `@Resource`  | Java 标准 | **先名称**后类型         |
| `@Qualifier` | Spring  | 配合 @Autowired 指定名称 |

### 6. Bean 作用域

| 作用域               | 说明             |
| ----------------- | -------------- |
| **singleton**（默认） | 容器中只有一个实例      |
| **prototype**     | 每次获取创建新实例      |
| **request**       | 每个 HTTP 请求一个实例 |
| **session**       | 每个会话一个实例       |

### 7. 循环依赖解决（三级缓存）

    A 依赖 B，B 依赖 A：

    一级缓存 singletonObjects       ── 完整 Bean
    二级缓存 earlySingletonObjects  ── 半成品 Bean（提前暴露）
    三级缓存 singletonFactories     ── Bean 工厂

    1. 实例化 A → A 的工厂放入三级缓存
    2. A 属性注入 → 发现需要 B → 去创建 B
    3. 实例化 B → B 属性注入 → 发现需要 A
    4. 从三级缓存取 A 的工厂 → 生成 A 的早期引用 → 放入二级缓存
    5. B 完成初始化 → 放入一级缓存
    6. A 拿到 B → A 完成初始化 → 放入一级缓存

> ⚠️ 构造器注入的循环依赖**无法解决**，可用 `@Lazy` 打破。

## 二、Spring Boot 的核心特点

### 1. 七大特点一览

| 特点                | 说明                              |
| ----------------- | ------------------------------- |
| **约定优于配置**        | 遵循默认约定，减少配置量                    |
| **自动配置**          | 根据引入的依赖自动配置 Bean                |
| **起步依赖（Starter）** | 一个依赖搞定一组技术栈                     |
| **内嵌服务器**         | 内置 Tomcat/Jetty/Undertow，无需外部部署 |
| **外部化配置**         | 支持多环境配置灵活切换                     |
| **Actuator 监控**   | 生产级健康检查、指标监控                    |
| **无 XML 配置**      | 纯注解 + Java 配置                   |

### 2. 自动配置原理（最核心）

    @SpringBootApplication
        ├── @SpringBootConfiguration    → 本质是 @Configuration
        ├── @ComponentScan              → 扫描当前包及子包
        └── @EnableAutoConfiguration    → 🔑 自动配置入口
                  │
                  ▼
            加载 META-INF/spring.factories（2.x）
            或 META-INF/spring/xxx.imports（3.x）
                  │
                  ▼
            加载所有 XxxAutoConfiguration 类
                  │
                  ▼
            条件注解过滤，决定哪些生效：
            @ConditionalOnClass       ── classpath 有某个类才生效
            @ConditionalOnMissingBean ── 用户没自定义才生效（用户优先）
            @ConditionalOnProperty    ── 配置文件有某属性才生效

**核心原则：用户自定义的 Bean 永远优先于自动配置。**

### 3. 起步依赖

```xml
<!-- 引入一个 starter，自动管理所有子依赖及版本 -->
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-web</artifactId>
</dependency>
<!-- 自动包含：spring-web, spring-webmvc, tomcat, jackson 等 -->
```

### 4. 内嵌服务器

```bash
# 传统：打 WAR → 部署到外部 Tomcat
# Spring Boot：打 JAR → 直接运行
java -jar my-app.jar   # 内嵌 Tomcat 自动启动
```

### 5. 外部化配置优先级

    命令行参数           (最高)
      ↓
    系统环境变量
      ↓
    application-{profile}.yml
      ↓
    application.yml
      ↓
    代码中默认值          (最低)

### 6. Spring Boot vs Spring

| 维度 | Spring           | Spring Boot   |
| -- | ---------------- | ------------- |
| 配置 | 大量 XML / Java 配置 | 自动配置，极少配置     |
| 依赖 | 手动管理版本兼容         | Starter 一键引入  |
| 部署 | WAR 包 + 外部容器     | JAR 包 + 内嵌服务器 |
| 上手 | 学习曲线陡            | 快速上手          |

***

## 三、总结

    Spring Boot = Spring + 自动配置 + Starter + 内嵌服务器

    DI 核心流程：扫描 → 注册 → 实例化 → 注入 → 初始化

    最佳实践：
      ✅ 构造器注入（不可变、安全、可测试）
      ✅ 面向接口编程（解耦、可替换）
      ✅ 善用自动配置 + 条件注解
      ❌ 避免字段注入
      ❌ 避免构造器循环依赖
