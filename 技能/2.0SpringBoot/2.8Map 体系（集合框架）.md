Map 存储 key-value 键值对，key 不可重复，value 可重复。

### HashMap（高频⭐⭐⭐⭐⭐）

#### 底层结构

- **JDK 7**：数组 + 链表（头插法）
- **JDK 8**：数组 + 链表 + **红黑树**（尾插法）

```
数组（Node[] table）
┌───┬───┬───┬───┬───┬───┬───┬───┐
│ 0 │ 1 │ 2 │ 3 │ 4 │ 5 │ 6 │ 7 │ ...
└─┬─┴───┴─┬─┴───┴───┴─┬─┴───┴───┘
  │       │           │
  ▼       ▼           ▼
 [A]     [C]         [E]
  │       │           │
  ▼       ▼           ▼
 [B]     [D]       红黑树
                  (链表≥8 且 数组≥64)
```

#### 核心参数

| 参数 | 默认值 | 说明 |
|------|--------|------|
| 初始容量 | **16** | 必须是 2 的幂 |
| 负载因子 | **0.75** | `size > capacity × 0.75` 时扩容 |
| 树化阈值 | **8** | 链表长度 ≥ 8 且数组长度 ≥ 64 → 转红黑树 |
| 退化阈值 | **6** | 红黑树节点 ≤ 6 → 退化回链表 |
| 最小树化容量 | **64** | 数组长度 < 64 时，链表超 8 也只扩容不树化 |

#### put 流程（JDK 8，面试必背）

```
1. 计算 hash：(h = key.hashCode()) ^ (h >>> 16)  ← 高位扰动
2. 定位桶下标：(n - 1) & hash  ← 等价取模（n 必须是 2 的幂）
3. 桶为空 → 直接插入 Node
4. 桶不为空：
   a. 首节点 key 相同 → 覆盖 value
   b. 首节点是 TreeNode → 红黑树插入
   c. 首节点是链表 → 尾插法遍历：
      - 找到相同 key → 覆盖
      - 到末尾 → 插入新节点 → 检查链表长度 ≥ 8？
        - 数组长度 ≥ 64 → 树化
        - 数组长度 < 64 → 扩容
5. size > threshold → resize() 扩容（容量翻倍）
```

#### 扩容机制（resize）

1. 容量翻倍：`newCap = oldCap << 1`
2. 重新计算每个元素的桶位置
3. **JDK 8 优化**：不重新计算 hash，只看 `hash & oldCap` 是 0 还是 1
   - 0 → 位置不变
   - 1 → 新位置 = 原位置 + oldCap

#### 为什么容量必须是 2 的幂？

- `(n - 1) & hash` 等价于 `hash % n`，但**位运算比取模快**
- 保证 hash 分布均匀，减少碰撞
- 扩容时利用 `hash & oldCap` 快速分流，效率更高

#### JDK 7 vs JDK 8 关键差异

| 维度 | JDK 7 | JDK 8 |
|------|-------|-------|
| 数据结构 | 数组 + 链表 | 数组 + 链表 + **红黑树** |
| 插入方式 | **头插法** | **尾插法** |
| 扩容 | 先扩容再插入 | 先插入再扩容 |
| hash 计算 | 4 次扰动 | 1 次扰动（h ^ h>>>16） |
| 并发问题 | **环形链表**（死循环） | 数据覆盖丢失（无环） |

> JDK 7 头插法在并发 rehash 时，两个线程同时扩容可能形成环形链表，导致 get() 死循环。JDK 8 改为尾插法解决了此问题，但 HashMap **仍然不是线程安全的**。

### ConcurrentHashMap（高频⭐⭐⭐⭐⭐）

#### JDK 7：分段锁（Segment）

```
ConcurrentHashMap
├── Segment[0] (继承 ReentrantLock)
│   └── HashEntry[] table
├── Segment[1]
│   └── HashEntry[] table
├── ...
└── Segment[15]  (默认 16 个段)
    └── HashEntry[] table
```

- 每个 Segment 独立加锁，**并发度 = Segment 数量（默认 16）**
- 读操作无锁（volatile 保证可见性）
- 缺点：Segment 数量初始化后不可变，锁粒度偏大

#### JDK 8：Node 数组 + CAS + synchronized

```
Node[] table
┌───┬───┬───┬───┬───┐
│ 0 │ 1 │ 2 │ 3 │ 4 │ ...
└─┬─┴───┴─┬─┴───┴───┘
  │       │
  ▼       ▼
 [A]     [C]   ← 对头节点加 synchronized
  │       │
  ▼       ▼
 [B]     [D]
```

- **锁粒度细化到单个桶**（synchronized 锁桶的头节点）
- 空桶插入用 **CAS**（无锁）
- 读操作完全无锁（Node 的 val 和 next 用 volatile 修饰）
- `size()` 使用 **baseCount + CounterCell[]** 分段计数（类似 LongAdder）

#### JDK 7 vs JDK 8 对比

| 维度 | JDK 7 | JDK 8 |
|------|-------|-------|
| 锁机制 | 分段锁（Segment，ReentrantLock） | **CAS + synchronized（锁桶头节点）** |
| 锁粒度 | 每个 Segment（多个桶共享） | **每个桶** |
| 数据结构 | 数组 + 链表 | 数组 + 链表 + **红黑树** |
| 读操作 | 无锁（volatile） | 无锁（volatile） |
| 并发度 | 固定（默认 16） | **等于桶数量**（动态扩展） |
| 计数方式 | 每个 Segment 独立计数 | baseCount + CounterCell[] |

### 其他 Map 实现

| 实现类 | 底层结构 | 核心特点 | 适用场景 |
| --- | --- | --- | --- |
| LinkedHashMap | HashMap + 双向链表 | 维护**插入顺序**或**访问顺序**（LRU 缓存基础） | 有序 Map、简易 LRU |
| TreeMap | **红黑树** | key 有序（自然排序/Comparator）；O(log n) | 排序 Map |
| Hashtable | 数组 + 链表 | 全表锁（synchronized），不允许 null key/value | **已过时**，用 ConcurrentHashMap |
| ConcurrentHashMap | 数组 + 链表/红黑树 | 线程安全，高并发性能优 | **高并发首选** |

### LinkedHashMap 实现 LRU 缓存

```java
// accessOrder=true：按访问顺序排列（最近访问的放末尾，最久未访问的在头部）
LinkedHashMap<String, String> lru = new LinkedHashMap<>(16, 0.75f, true) {
    @Override
    protected boolean removeEldestEntry(Map.Entry<String, String> eldest) {
        return size() > 100; // 超过 100 个元素时，自动移除最久未访问的
    }
};
```

### HashMap vs Hashtable vs ConcurrentHashMap

| 维度 | HashMap | Hashtable | ConcurrentHashMap |
|------|---------|-----------|-------------------|
| 线程安全 | 否 | 是（全表 synchronized） | 是（CAS + 分段/桶锁） |
| null key/value | 允许 1 个 null key | **不允许** | **不允许** |
| 性能 | 最高 | 最低 | 高 |
| 继承体系 | AbstractMap | **Dictionary**（已过时） | AbstractMap |
| 迭代器 | fail-fast | fail-fast | **弱一致性**（不抛异常） |

### 高频追问

1. **HashMap 为什么线程不安全？** JDK 7 并发 rehash 环形链表 + 数据丢失；JDK 8 并发 put 数据覆盖 + size 不准确
2. **为什么树化阈值是 8？** 泊松分布下，链表长度达到 8 的概率约为千万分之六，是极小概率事件。设为 8 在时间和空间之间取得平衡
3. **为什么退化阈值是 6 而不是 8？** 防止频繁树化/退化（抖动）。如果都是 8，刚好在 8 附近增删会反复转换
4. **HashMap 的 key 可以是可变对象吗？** 技术上可以，但**极其危险**。修改 key 后 hashCode 变了，原来的 entry 就找不到了
5. **ConcurrentHashMap 的 size() 是精确的吗？** 不是严格精确的（弱一致性），统计时可能有并发修改。但 `mappingCount()` 返回 long，推荐使用
6. **ConcurrentHashMap 的 key 和 value 为什么不能为 null？** 消除二义性：get 返回 null 时无法区分是 "key 不存在" 还是 "value 为 null"。多线程环境下这种歧义会导致 bug
