## 一句话结论

Spring 声明式事务的本质是 **AOP + 事务管理器**：通过动态代理在目标方法前后织入事务逻辑（开启事务 → 执行业务 → 提交/回滚）。`@Transactional` 失效是面试最高频问题，核心原因是**代理机制被绕过**或**异常未被正确传播**。

---

### 事务实现原理

```
调用方
  ↓
代理对象（CGLIB/JDK 动态代理）
  ↓
TransactionInterceptor（事务拦截器）
  ├─ 1. 获取事务属性（传播行为、隔离级别、回滚规则）
  ├─ 2. 获取 PlatformTransactionManager
  ├─ 3. 开启事务（getTransaction）
  │     └─ 从 DataSource 获取 Connection → setAutoCommit(false)
  │     └─ 将 Connection 绑定到当前线程（TransactionSynchronizationManager）
  ├─ 4. 执行目标方法
  ├─ 5a. 正常返回 → commitTransactionAfterReturning（提交）
  └─ 5b. 抛出异常 → completeTransactionAfterThrowing
        ├─ 匹配回滚规则 → rollback（回滚）
        └─ 不匹配 → commit（仍然提交！）
```

**关键**：事务的 Connection 通过 `ThreadLocal` 绑定到当前线程，同一线程内的多次数据库操作共用同一个 Connection，从而保证事务性。

### 七种传播行为（高频⭐⭐⭐⭐）

| 传播行为 | 含义 | 常用场景 |
|---------|------|---------|
| **REQUIRED**（默认） | 有事务就加入，没有就新建 | 大多数业务方法 |
| **REQUIRES_NEW** | **总是新建事务**，挂起外部事务 | 日志记录（不受外部事务回滚影响） |
| SUPPORTS | 有事务就加入，没有就非事务执行 | 查询方法 |
| NOT_SUPPORTED | 非事务执行，挂起外部事务 | 不需要事务的操作 |
| MANDATORY | 必须在事务中调用，否则抛异常 | 必须被其他事务方法调用的方法 |
| NEVER | 非事务执行，如果存在事务则抛异常 | 禁止在事务中调用的方法 |
| **NESTED** | 外部事务内创建**嵌套事务（保存点）** | 批量处理中部分失败可回滚 |

**REQUIRED vs REQUIRES_NEW vs NESTED 对比**：

```
REQUIRED：A 调 B，共用同一个事务。B 异常 → A 和 B 都回滚

REQUIRES_NEW：A 调 B，B 开新事务。B 异常 → 只回滚 B；A 不受影响
             （但如果 A catch 了 B 的异常，A 可以继续提交）

NESTED：A 调 B，B 使用保存点。B 异常 → 回滚到保存点（只回滚 B）
        A 异常 → A 和 B 都回滚（B 是 A 的子事务）
```

### 四种隔离级别

| 隔离级别 | 脏读 | 不可重复读 | 幻读 | 说明 |
|---------|------|----------|------|------|
| READ_UNCOMMITTED | 有 | 有 | 有 | 最低隔离，几乎不用 |
| READ_COMMITTED | **无** | 有 | 有 | Oracle 默认 |
| **REPEATABLE_READ** | 无 | **无** | 有 | **MySQL InnoDB 默认** |
| SERIALIZABLE | 无 | 无 | **无** | 串行化，性能最差 |

> Spring 默认 `Isolation.DEFAULT`，即使用底层数据库的默认隔离级别

### @Transactional 六大失效场景（高频⭐⭐⭐⭐⭐）

#### 1. 方法非 public

```java
@Transactional
private void doSave() { ... }  // ❌ 事务不生效！
```

**原因**：Spring AOP 基于代理，JDK 动态代理只能代理接口方法（必须 public），CGLIB 虽然能代理非 public 方法，但 Spring 的 `TransactionInterceptor` **显式检查方法是否为 public**，非 public 直接跳过。

#### 2. 同类内部方法调用（最常见！）

```java
@Service
public class OrderService {
    public void createOrder() {
        this.saveOrder();  // ❌ this 调用，不经过代理，事务不生效！
    }

    @Transactional
    public void saveOrder() { ... }
}
```

**原因**：`this.saveOrder()` 调用的是原始对象而非代理对象，AOP 无法介入。

**解决方案**：
```java
// 方案1：注入自身代理（推荐）
@Autowired
private OrderService self; // Spring 注入的是代理对象
public void createOrder() {
    self.saveOrder(); // ✅ 经过代理
}

// 方案2：从 ApplicationContext 获取代理
@Autowired
private ApplicationContext context;
public void createOrder() {
    context.getBean(OrderService.class).saveOrder(); // ✅
}

// 方案3：AopContext（需开启 exposeProxy）
((OrderService) AopContext.currentProxy()).saveOrder(); // ✅
```

#### 3. 异常被 catch 吞掉

```java
@Transactional
public void save() {
    try {
        userDao.insert(user);
        int i = 1 / 0;  // 异常
    } catch (Exception e) {
        log.error("出错了", e);  // ❌ 异常被吞，事务感知不到，不会回滚！
    }
}
```

**解决**：catch 后手动标记回滚：
```java
catch (Exception e) {
    log.error("出错了", e);
    TransactionAspectSupport.currentTransactionStatus().setRollbackOnly(); // ✅
}
```

#### 4. 异常类型不匹配

```java
@Transactional  // 默认只回滚 RuntimeException 和 Error
public void save() throws IOException {
    userDao.insert(user);
    throw new IOException("文件异常");  // ❌ 受检异常，默认不回滚！
}
```

**解决**：
```java
@Transactional(rollbackFor = Exception.class)  // ✅ 指定回滚所有异常
public void save() throws Exception { ... }
```

#### 5. 数据库引擎不支持事务

```sql
-- MyISAM 不支持事务！
CREATE TABLE orders (...) ENGINE=MyISAM;  -- ❌
-- 应使用 InnoDB
CREATE TABLE orders (...) ENGINE=InnoDB;  -- ✅
```

#### 6. Bean 未被 Spring 管理

```java
// 手动 new 的对象，不在 Spring 容器中，事务注解无效
OrderService service = new OrderService();  // ❌ 不是代理对象
service.save();
```

### @Transactional 常用属性

```java
@Transactional(
    propagation = Propagation.REQUIRED,    // 传播行为
    isolation = Isolation.DEFAULT,          // 隔离级别
    timeout = 30,                           // 超时秒数（-1 不超时）
    readOnly = false,                       // 是否只读
    rollbackFor = Exception.class,          // 回滚异常类型
    noRollbackFor = BusinessException.class // 不回滚的异常
)
```

**readOnly = true 的作用**：
- 提示数据库优化（如 MySQL 跳过加锁、跳过 MVCC 版本判断）
- 某些框架会路由到从库
- 方法内如果有写操作会抛异常

### 编程式事务（对比）

```java
@Autowired
private TransactionTemplate transactionTemplate;

public void save() {
    transactionTemplate.execute(status -> {
        userDao.insert(user);
        orderDao.insert(order);
        // 异常自动回滚，正常自动提交
        return null;
    });
}
```

| 维度 | 声明式（@Transactional） | 编程式（TransactionTemplate） |
|------|----------------------|---------------------------|
| 使用方式 | 注解 | 代码 |
| 侵入性 | 低 | 高 |
| 粒度 | 方法级 | **代码块级**（更灵活） |
| 适用 | 大多数场景 | 需要细粒度控制的场景 |

### 高频追问

1. **@Transactional 加在接口上还是实现类上？** 加在**实现类**上。如果使用 CGLIB 代理（Spring Boot 默认），接口上的注解不会被继承
2. **同一个方法中调用两个带 @Transactional 的方法，事务如何传播？** 取决于传播行为。默认 REQUIRED → 加入外层事务；REQUIRES_NEW → 各自独立事务
3. **@Transactional 方法调用另一个 @Transactional 方法，内层异常被 catch，外层会回滚吗？** REQUIRED 下会！因为共用事务，内层异常已标记 rollback-only，外层提交时发现标记 → 抛 UnexpectedRollbackException
4. **长事务有什么风险？** 数据库连接长时间被占用（连接池耗尽）、锁持有时间长（阻塞其他事务）、undo log 膨胀
5. **如何避免长事务？** 减小事务范围、将查询操作移到事务外、使用编程式事务精确控制、设置 timeout
