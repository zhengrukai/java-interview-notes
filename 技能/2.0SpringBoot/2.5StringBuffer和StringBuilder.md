## 一句话结论

String 是**不可变**的字符序列（final char[]），每次修改都会创建新对象；StringBuilder 和 StringBuffer 都是**可变**的（内部 char[] 可扩容），区别在于 StringBuffer 线程安全（方法加 synchronized），StringBuilder 非线程安全但**性能更高**。

---

### String 不可变性（高频⭐⭐⭐⭐⭐）

```java
public final class String implements java.io.Serializable, Comparable<String>, CharSequence {
    /** JDK 8: final char[] value */
    /** JDK 9+: final byte[] value + byte coder（Latin1/UTF16 压缩） */
    private final char value[];  // final 修饰，引用不可变
}
```

**不可变的三重保证**：
1. **类用 final 修饰** → 不可被继承，防止子类破坏不可变性
2. **char[] 用 private final 修饰** → 引用不可变，且外部无法直接访问
3. **没有提供修改 char[] 内容的方法** → 所有 "修改" 操作（concat、replace、substring）都返回**新 String 对象**

**为什么设计成不可变？**
- **字符串常量池**：不可变才能安全复用，节省内存
- **线程安全**：天然线程安全，无需加锁
- **HashMap key 安全**：hashCode 可以缓存，保证 key 不变
- **安全性**：网络连接、数据库 URL、文件路径等敏感信息不可被篡改

### 字符串常量池（String Pool）

```java
String s1 = "abc";              // 字面量 → 直接放入常量池
String s2 = "abc";              // 复用常量池中的同一个对象
String s3 = new String("abc");  // 强制在堆中创建新对象
String s4 = s3.intern();        // 手动入池，返回常量池中的引用

System.out.println(s1 == s2);   // true（常量池同一对象）
System.out.println(s1 == s3);   // false（堆 vs 常量池）
System.out.println(s1 == s4);   // true（intern 返回池中引用）
```

**常量池位置变迁**：
- JDK 6：永久代（PermGen）
- JDK 7+：**堆内存**（方便 GC 回收）

**经典面试题**：`new String("abc")` 创建了几个对象？
- **1 或 2 个**：如果常量池中已有 "abc" → 只在堆中创建 1 个；如果没有 → 常量池 1 个 + 堆 1 个 = 2 个

### 三者核心对比

| 维度 | String | StringBuilder | StringBuffer |
|------|--------|--------------|-------------|
| 可变性 | **不可变** | 可变 | 可变 |
| 线程安全 | 是（不可变天然安全） | **否** | 是（synchronized） |
| 性能 | 拼接产生大量临时对象 | **最高** | 略低于 StringBuilder |
| 底层实现 | `final char[]` | `char[]`（可扩容） | `char[]`（可扩容） |
| 初始容量 | — | **16**（无参构造） | **16**（无参构造） |
| 扩容策略 | — | 原容量 × 2 + 2 | 原容量 × 2 + 2 |
| 适用场景 | 少量字符串、常量 | **单线程**大量拼接 | **多线程**大量拼接 |

### 底层扩容原理

```java
// StringBuilder / StringBuffer 共用父类 AbstractStringBuilder
abstract class AbstractStringBuilder {
    char[] value;  // 内部数组（非 final，可变）
    int count;     // 实际使用长度

    // 扩容核心逻辑
    private int newCapacity(int minCapacity) {
        int newCapacity = (value.length << 1) + 2;  // 原容量 × 2 + 2
        if (newCapacity < minCapacity) {
            newCapacity = minCapacity;               // 不够就直接用最小需求
        }
        // 最终通过 Arrays.copyOf 创建新数组并拷贝
        return newCapacity;
    }
}
```

**关键点**：每次扩容涉及 **数组拷贝**，如果能预估最终长度，建议通过构造器指定初始容量避免多次扩容。

```java
// 推荐：预估容量，减少扩容次数
StringBuilder sb = new StringBuilder(256);
```

### String 拼接的编译器优化

```java
// 编译前
String s = "a" + "b" + "c";
// 编译后（编译期常量折叠）
String s = "abc";  // 直接合并，零开销

// 编译前（变量拼接）
String s = a + b + c;
// JDK 8 编译后：等价于 new StringBuilder().append(a).append(b).append(c).toString()
// JDK 9+ 编译后：使用 invokedynamic + StringConcatFactory（更高效）
```

**面试关键点**：循环内用 `+` 拼接字符串，每次循环都会创建新 StringBuilder → **性能灾难**，应手动使用 StringBuilder。

```java
// 错误：循环内 + 拼接
String result = "";
for (int i = 0; i < 10000; i++) {
    result += i;  // 每次创建新 StringBuilder + toString → 大量临时对象
}

// 正确：手动 StringBuilder
StringBuilder sb = new StringBuilder();
for (int i = 0; i < 10000; i++) {
    sb.append(i);
}
String result = sb.toString();
```

### StringBuffer 的线程安全

```java
// StringBuffer 的 append 方法
@Override
public synchronized StringBuffer append(String str) {
    toStringCache = null;
    super.append(str);  // 调用父类 AbstractStringBuilder 的逻辑
    return this;
}
```

**为什么现在很少用 StringBuffer？**
- 绝大多数字符串拼接发生在**单线程**环境（方法内局部变量）
- 多线程共享可变字符串的场景极少
- 即使需要线程安全，也可以用外部同步或其他方案

### 高频追问

1. **String s = new String("xyz") 创建了几个对象？** 1 或 2 个（取决于常量池中是否已有 "xyz"）
2. **String 的 hashCode 为什么可以缓存？** 不可变性保证 hashCode 不变，首次计算后存入 `hash` 字段，后续直接返回
3. **StringBuilder 和 StringBuffer 的初始容量是多少？** 16。传入字符串的构造器初始容量为 str.length() + 16
4. **为什么 JDK 9 把 char[] 改成 byte[]？** Compact Strings 优化：Latin1 字符用 1 字节存储（LATIN1 编码），中文等用 2 字节（UTF16），节省约 50% 内存
5. **intern() 方法有什么用？什么时候该用？** 将字符串放入常量池并返回池中引用。适合大量重复字符串的场景（如 XML 解析），但滥用会导致常量池膨胀
