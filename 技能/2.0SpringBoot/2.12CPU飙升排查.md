# CPU 飙升如何排查

## 问题
CPU飙升如何排查

## 回答
CPU 飙升的排查，核心就一句话：

> **先确认是谁把 CPU 打满，再确认它为什么打满。**

不要一上来就猜：

- 是代码死循环？
- 是 Full GC？
- 是 SQL 慢？
- 是线程太多？
- 是流量突增？

CPU 高只是**现象**，真正要找的是 **进程、线程、代码、SQL、流量、GC、锁竞争** 里的根因。

---

## 一、先建立排查思路

建议按这 4 层来查：

1. **机器层**：到底是不是整个机器 CPU 高
2. **进程层**：哪个进程占 CPU
3. **线程层**：哪个线程占 CPU
4. **代码层**：线程到底卡在哪段代码

如果是 Java 服务，最终通常会落到：

- 某个线程热点方法
- GC 太频繁
- 死循环 / 空转
- 锁竞争
- SQL / 序列化 / JSON / 正则 / 加解密特别重
- 线程池堆积后疯狂争抢 CPU

---

## 二、第一步：先看是不是系统整体 CPU 高

先确认到底是：

- 单个 Java 进程 CPU 高
- 还是整台机器都高
- 还是内核态 CPU 高
- 还是用户态 CPU 高

常用命令：

```bash
top
htop
uptime
vmstat 1
mpstat -P ALL 1
sar -u 1 5
```

重点关注：

- `us`：用户态 CPU
- `sy`：内核态 CPU
- `id`：空闲
- `wa`：IO wait
- `load average`
- 每个核是否均衡

### 怎么判断方向

#### 1. `us` 很高
通常说明：

- 应用代码计算重
- Java 线程在跑业务逻辑
- JSON/序列化/加解密/正则/循环等耗 CPU

#### 2. `sy` 很高
通常说明：

- 系统调用多
- 网络包处理重
- 上下文切换频繁
- 锁竞争严重
- 内核开销大

#### 3. `wa` 很高
这更偏 IO，不一定是纯 CPU 问题。

---

## 三、第二步：找出哪个进程占 CPU

用：

```bash
top -c
ps -eo pid,ppid,cmd,%mem,%cpu --sort=-%cpu | head
pidstat -u 1
```

如果是 Java 服务，重点找到 Java 进程 PID。

比如：

```bash
ps -ef | grep java
top -Hp <pid>
```

---

## 四、第三步：定位到哪个线程最耗 CPU

这是 Java 排查最关键的一步。

```bash
top -Hp <pid>
```

你会看到线程级 CPU 占用，例如：

- 某个线程占 300%
- 某几个线程持续高

记下高 CPU 线程的 **线程 ID（十进制）**。

然后转成十六进制：

```bash
printf "%x\n" <tid>
```

接着抓线程栈：

```bash
jstack <pid> > jstack.log
```

在 `jstack.log` 里搜索这个十六进制线程 ID，例如：

```text
nid=0x1a2b
```

这样就能定位到：

- 哪个线程
- 正在执行什么方法
- 卡在什么代码路径

---

## 五、第四步：根据线程栈判断根因

拿到线程栈后，通常就能分几类看。

### 场景 1：业务代码死循环 / 空转

典型特征：

- 线程一直 RUNNABLE
- 栈里反复是某个 while 循环、重试循环、轮询逻辑

比如：

```java
while(true) {
    if (queue.isEmpty()) {
        continue;
    }
}
```

这种会把 CPU 直接打满。

#### 处理
- 检查是否无阻塞空轮询
- 增加 sleep / wait / blocking queue
- 修正退出条件
- 限制重试频率

---

### 场景 2：频繁 Full GC / GC 线程占 CPU

现象：

- Java 进程 CPU 高
- `jstat -gcutil <pid> 1s`
- GC 次数很频繁
- GC 线程吃掉很多 CPU

常用命令：

```bash
jstat -gcutil <pid> 1s
jcmd <pid> GC.heap_info
jcmd <pid> VM.flags
```

如果是 GC 问题，通常原因是：

- 对象创建太快
- 内存太小
- 大对象多
- 缓存失控
- 泄漏导致老年代压力大

#### 处理
- 看 GC 日志
- 看堆内存使用
- 做 heap dump
- 分析对象分配热点
- 调整堆大小和 GC 参数
- 减少短命对象创建

---

### 场景 3：线程竞争 / 锁竞争严重

现象：

- CPU 高
- 线程很多
- 上下文切换频繁
- 有些线程 RUNNABLE，但实际在抢锁
- `sy` 也可能高

常见原因：

- `synchronized` 锁粒度太大
- `ReentrantLock` 热点锁
- 多线程争同一个 Map / 缓存 / 队列
- 自旋过久

可以结合看：

```bash
vmstat 1
pidstat -wt 1 -p <pid>
jstack <pid>
```

#### 处理
- 缩小锁粒度
- 减少共享资源争用
- 使用分段锁 / 无锁结构 / Concurrent 容器
- 避免在锁内做耗时操作

---

### 场景 4：线程池配置不合理

比如：

- 线程数太多
- 大量线程同时争抢 CPU
- 队列爆满
- 拒绝重试形成风暴

现象：

- 线程非常多
- `top -Hp` 看到很多线程都在跑
- 机器上下文切换很高

排查方向：

- 核心线程数是否远超 CPU 核数
- 是否创建了太多业务线程池
- 是否每个请求都异步拆分很多子任务

#### 处理
- 收敛线程池数量
- 线程池参数与 CPU 核数匹配
- 限流
- 减少无意义并发

---

### 场景 5：SQL 问题间接引发 CPU 高

这里分两类。

#### 类 A：数据库 CPU 高
如果是 MySQL 本身 CPU 高，去看：

- 慢 SQL
- explain
- 锁等待
- 全表扫描
- 排序 / join / 临时表

#### 类 B：应用 CPU 高
比如应用层拿到大量数据后做：

- 内存排序
- JSON 转换
- 聚合计算
- 大对象拼接

也会导致 Java CPU 高。

#### 处理
- SQL 端建索引、减数据量
- 应用端避免拉太多数据
- 分页、流式处理、异步化

---

### 场景 6：序列化 / JSON / 正则 / 加解密过重

这些都是很常见的 CPU 杀手。

比如：

- Jackson/Fastjson 大对象频繁序列化
- 正则表达式灾难性回溯
- RSA/AES 高频加解密
- Base64 大量编解码
- Excel / PDF 导出

线程栈里通常会出现：

- `ObjectMapper`
- `Pattern`
- `Cipher`
- `MessageDigest`
- `GZIPOutputStream`

#### 处理
- 降低对象规模
- 优化正则
- 避免重复序列化
- 热点数据缓存
- 重 CPU 操作异步化

---

### 场景 7：流量突增导致 CPU 飙升

这类不是“程序坏了”，而是：

- 请求数激增
- 突发流量
- 重试风暴
- 爬虫 / 攻击
- 某个接口被打爆

排查：

- 看监控平台 QPS
- 看 Nginx / 网关日志
- 看接口维度 RT / TPS
- 看调用链 tracing

#### 处理
- 限流
- 熔断
- 降级
- 缓存
- 黑白名单
- WAF / 防刷

---

## 六、Java 服务 CPU 高的标准排查流程

这是最常用的一套。

### 第 1 步：看系统资源

```bash
top
vmstat 1
mpstat -P ALL 1
```

确认：

- CPU 是否真的高
- 是 us 高还是 sy 高
- 是否单核打满还是多核平均高

### 第 2 步：找高 CPU 进程

```bash
top -c
ps -eo pid,cmd,%cpu --sort=-%cpu | head
```

### 第 3 步：找高 CPU 线程

```bash
top -Hp <pid>
```

记下高 CPU 的线程 ID。

### 第 4 步：导出线程栈

```bash
jstack <pid> > jstack.log
```

### 第 5 步：十进制线程 ID 转十六进制

```bash
printf "%x\n" <tid>
```

然后在线程栈里搜：

```text
nid=0x...
```

### 第 6 步：判断线程在干嘛
看它是：

- 死循环
- GC
- 锁竞争
- 网络处理
- JSON
- SQL 结果处理
- 线程池任务风暴

### 第 7 步：结合监控验证
再配合：

- APM（SkyWalking / Pinpoint / Arthas / Prometheus）
- GC 日志
- 慢 SQL
- 接口 RT / QPS
- 调用链

确认不是误判。

---

## 七、数据库 CPU 高怎么查

如果你问的是 **MySQL 服务器 CPU 飙升**，排查方法不同。

重点看：

- 哪些 SQL 最耗 CPU
- 是否全表扫描
- 是否排序/临时表
- 是否热点更新
- 是否锁竞争

常用思路：

```sql
show processlist;
show engine innodb status;
```

再看：

- 慢查询日志
- `EXPLAIN`
- 是否缺索引
- 是否执行了大批量更新 / 大 join

---

## 八、线上排查时最容易犯的错

### 1. 只看 CPU，不看线程
你只知道“Java 进程 CPU 400%”，没用，必须下钻到线程。

### 2. 只抓一次 jstack
线程状态会变化，最好连续抓几次：

```bash
jstack <pid> > jstack1.log
sleep 3
jstack <pid> > jstack2.log
sleep 3
jstack <pid> > jstack3.log
```

如果同一个线程连续都在同一热点方法，基本就坐实了。

### 3. 把 IO 等待当 CPU 问题
有时候不是 CPU 算力不够，而是：

- IO 慢
- 锁卡住
- 网络抖动

### 4. 一上来就重启
重启能缓解，但会丢失现场。

正确做法通常是：

1. 先保留现场
2. 抓线程栈 / GC / top
3. 再决定是否摘流量或重启

---

## 九、如果是面试，该怎么回答

你可以这样答：

> CPU 飙升排查一般分四层：机器层、进程层、线程层、代码层。先用 top、vmstat 判断是用户态还是内核态 CPU 高，再用 top -Hp 定位高 CPU 线程。对于 Java 进程，记下线程 ID 后通过 jstack 导出线程栈，并把线程 ID 转成十六进制匹配 nid，定位具体热点代码。常见原因包括死循环、频繁 GC、锁竞争、线程池配置不合理、SQL 结果处理过重、JSON/正则/加解密计算重、流量突增等。排查时要结合 GC、慢 SQL、APM 和调用链一起看，最终定位到具体线程和具体代码路径，而不是只看 CPU 指标本身。

---

## 十、一句话总结

**CPU 飙升的排查本质是：先定位高 CPU 进程，再定位高 CPU 线程，最后用线程栈和监控把问题落到具体代码。**
