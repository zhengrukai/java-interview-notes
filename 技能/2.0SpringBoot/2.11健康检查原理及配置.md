# Spring Boot 健康检查原理及配置（面试回答）

## 问题
spring boot 健康检查原理及配置，应对面试

## 回答
面试里说 Spring Boot 健康检查，最好从 **是什么、原理、怎么配置、实际用途、注意点** 这 5 个层次来答。

---

# 一、先给面试版结论

> Spring Boot 健康检查通常是基于 **Actuator** 实现的。
> 它通过暴露 `/actuator/health` 端点，统一汇总应用以及各依赖组件的状态，比如数据库、Redis、磁盘空间、消息队列等。
> 底层原理是 Spring Boot 会收集所有 `HealthIndicator` 或 `ReactiveHealthIndicator` 的检测结果，再通过 `HealthEndpoint` 聚合成一个总状态返回。
> 在配置上可以通过 `management.endpoint.health.show-details`、`management.endpoints.web.exposure.include` 以及各组件的健康检查开关来控制。
> 在实际生产中，健康检查通常配合 Kubernetes、Nginx、注册中心、负载均衡做存活探测和就绪探测。

如果是面试，这段先说出来，已经很像样了。

---

# 二、什么是健康检查

健康检查本质上是：

> **让外部系统知道当前应用是不是“活着”、能不能接流量、依赖是不是正常”。**

常见问题：

- 服务进程还在，但数据库已经连不上
- JVM 没挂，但线程池满了
- 应用启动了，但核心依赖还没准备好
- Redis 挂了，接口其实已经不可用了

所以不能只看“进程有没有启动”，还要看“服务是否健康”。

---

# 三、Spring Boot 健康检查依赖什么

核心依赖一般是：

```xml
spring-boot-starter-actuator
```

加上这个 starter 后，Spring Boot 会提供很多监控和管理端点，比如：

- `/actuator/health`
- `/actuator/info`
- `/actuator/metrics`
- `/actuator/env`
- `/actuator/beans`

其中健康检查最核心的就是：

```text
/actuator/health
```

# 四、健康检查原理

这个是面试重点。

---

## 1. 核心组件

Spring Boot 健康检查相关核心类可以理解为几层：

1. **HealthIndicator**
2. **HealthContributor**
3. **HealthEndpoint**
4. **StatusAggregator**
5. **HttpCodeStatusMapper**

你面试不用全背类名，但最好理解职责。

---

## 2. `HealthIndicator` 是最核心的检测单元

每个组件都可以有自己的健康检查实现。

比如：

- 数据库一个 `DataSourceHealthIndicator`
- Redis 一个 `RedisHealthIndicator`
- 磁盘空间一个 `DiskSpaceHealthIndicator`
- 自定义业务一个 `CustomHealthIndicator`

它们一般返回：

- `UP`
- `DOWN`
- `OUT_OF_SERVICE`
- `UNKNOWN`

例如数据库健康检查，本质就是：

- 能不能拿到连接
- 能不能执行检测 SQL

---

## 3. Spring Boot 会自动装配很多健康检查器

如果你的项目里引入了某些依赖，Spring Boot 会自动注册对应的健康检查器。

例如：

- 有 `DataSource` -> 自动有数据库健康检查
- 有 Redis -> 自动有 Redis 健康检查
- 有 Mongo -> 自动有 Mongo 健康检查
- 有磁盘空间检查 -> 自动有 DiskSpace 健康检查

所以很多时候你什么都没写，访问：

```text
/actuator/health
```

就已经能看到一些基础状态。

---

## 4. `HealthEndpoint` 负责聚合结果

每个 `HealthIndicator` 只负责自己的状态。

Spring Boot 最终会通过 `HealthEndpoint` 把这些结果汇总起来。

例如：

- db = UP
- redis = UP
- diskSpace = UP

最终整体状态就是：

```json
{
  "status": "UP"
}
```

如果其中一个核心组件挂了，例如：

- db = DOWN
- redis = UP
- diskSpace = UP

最终总状态一般就会变成：

```json
{
  "status": "DOWN"
}
```

---

## 5. 聚合规则是谁决定的

是 **状态聚合器** 决定的。

默认情况下大致优先级是：

```text
DOWN > OUT_OF_SERVICE > UP > UNKNOWN
```

也就是说：

- 只要有关键组件是 `DOWN`
- 整体通常就判定为 `DOWN`

---

## 6. 为什么 HTTP 状态码会变

健康检查不仅有业务状态，还有 HTTP 状态码。

例如：

- `UP` -> 200
- `DOWN` -> 503
- `OUT_OF_SERVICE` -> 503

这样外部系统，比如：

- Kubernetes
- Nginx
- 注册中心
- 负载均衡器

就可以根据 HTTP 状态码决定：

- 是否摘流
- 是否重启
- 是否继续路由请求

---

```json
{
```


这就是把每个组件的子状态展开了。

# 六、怎么配置

这个是面试常问点。

## 1. 暴露 health 端点

```yaml
management:
  endpoints:
    web:
      exposure:
        include: health,info
```

说明：

- 默认不是所有端点都对外暴露
- 这里表示对外开放 `health` 和 `info`

如果你想全开：

```yaml
management:
  endpoints:
    web:
      exposure:
        include: "*"
```

但生产一般不建议全开。

---

## 2. 控制健康详情是否显示

```yaml
management:
  endpoint:
    health:
      show-details: always
```

可选值通常有：

- `never`
- `when_authorized`
- `always`

### 含义
- `never`：只显示总体状态
- `always`：显示每个组件详细信息
- `when_authorized`：登录或授权后才显示

生产环境一般不建议公开太多细节，避免泄露内部组件信息。

---

## 3. 控制组件启用/禁用

例如禁用数据库健康检查：

```yaml
management:
  health:
    db:
      enabled: false
```

禁用 Redis 健康检查：

```yaml
management:
  health:
    redis:
      enabled: false
```

原因可能是：

- 某些依赖不属于核心路径
- 某个健康检查本身代价高
- 不希望某个次要依赖导致整体变 `DOWN`

---

## 4. 磁盘空间阈值配置

```yaml
management:
  health:
    diskspace:
      enabled: true
      threshold: 10MB
```

意思是：

- 当剩余磁盘空间低于阈值时
- 健康状态变差

---

## 5. 健康探针配置（K8s 常见）

Spring Boot 2.3+ 常见配置：

```yaml
management:
  endpoint:
    health:
      probes:
        enabled: true
```

开启后通常会提供：

- `liveness`
- `readiness`

比如：

- `/actuator/health/liveness`
- `/actuator/health/readiness`

这在 Kubernetes 场景非常常见。

---

# 七、自定义健康检查原理

这是很容易被问到的延伸点。

如果默认的数据库、Redis、磁盘检查不够，你可以自己写。

## 1. 实现 `HealthIndicator`

例如：

```java
@Component
public class OrderSystemHealthIndicator implements HealthIndicator {

    @Override
    public Health health() {
        boolean ok = checkOrderSystem();
        if (ok) {
            return Health.up()
                    .withDetail("service", "order-system")
                    .withDetail("message", "ok")
                    .build();
        }
        return Health.down()
                .withDetail("service", "order-system")
                .withDetail("message", "unavailable")
                .build();
    }

    private boolean checkOrderSystem() {
        return true;
    }
}
```

---

## 2. 生效原理

Spring Boot 启动时会把这个 Bean 识别成一个健康检查器。

访问 `/actuator/health` 时，它的结果也会一起被汇总。

---

## 3. 自定义健康检查适合什么场景

适合检查：

- 某个核心第三方接口是否可用
- 某个业务线程池是否满载
- 某个本地缓存是否异常
- 某个业务开关是否准备完成
- 某个关键下游依赖是否准备好

---

# 八、健康检查在生产中的实际用途

这部分很重要，能体现你不是只会背注解。

## 1. 配合负载均衡摘流
比如 Nginx、SLB、网关会探测：

```text
/actuator/health
```

如果返回 503，就把这个节点摘掉。

---

## 2. 配合 Kubernetes 存活与就绪探针

### 存活探针 liveness
判断：

- 这个 Pod 还活着吗？
- 要不要重启？

### 就绪探针 readiness
判断：

- 这个 Pod 已经准备好接流量了吗？

例如：

- 应用启动了
- 但数据库连接池还没准备好
- 缓存预热还没完成

这时 liveness 可能是活着的，但 readiness 可能还没 ready。

---

## 3. 配合注册中心上下线
某些服务治理体系会根据健康状态来决定：

- 是否从注册中心摘除实例
- 是否继续转发请求

---

## 4. 运维监控
监控平台会定时抓：

- `health`
- `metrics`
- `prometheus`

来感知系统整体可用性。

---

# 九、健康检查和心跳的区别

这个面试偶尔也会问。

### 健康检查
判断的是：

- 服务当前是否健康
- 是否能提供服务
- 依赖是否正常

### 心跳
更多是：

- 服务还活着吗
- 与注册中心的连接还在吗

所以：

**心跳更偏“活着”，健康检查更偏“能不能正常服务”。**

---

# 十、面试时容易忽略的点

## 1. 健康检查不等于业务可用
比如：

- JVM 活着
- 数据库连得上
- 但线程池已经满了
- 或核心下游接口超时

这时候基础健康检查可能仍然是 `UP`，但业务已经不可用了。

所以生产中常常需要**自定义健康检查**。

---

## 2. 不要把所有依赖都当成致命依赖
例如：

- 某个次要统计服务挂了
- 不应该让整个应用 health 变成 `DOWN`

所以要区分：

- 核心依赖
- 非核心依赖

---

## 3. 不能随便暴露太多细节
如果 `show-details: always` 对公网开放，可能暴露：

- 数据库类型
- Redis 信息
- 磁盘情况
- 组件名称

生产环境需要谨慎。

---

## 4. readiness 和 liveness 不要混用
这是 K8s 场景的大坑。

- **liveness** 失败 -> 容器可能被重启
- **readiness** 失败 -> 容器只是摘流，不一定重启

如果配错，可能导致服务频繁重启。

---

# 十一、面试推荐答法

你可以这样答：

> Spring Boot 的健康检查通常基于 Actuator 实现，核心端点是 `/actuator/health`。底层会把各个 `HealthIndicator` 的结果统一聚合，比如数据库、Redis、磁盘空间等，再返回整体状态 `UP`、`DOWN`、`OUT_OF_SERVICE` 或 `UNKNOWN`。如果引入了相关依赖，Spring Boot 会自动装配对应的健康检查器；也可以通过实现 `HealthIndicator` 自定义业务健康检查。配置上常见的是通过 `management.endpoints.web.exposure.include` 暴露 health 端点，通过 `management.endpoint.health.show-details` 控制是否展示详情，通过 `management.health.xxx.enabled` 控制某类检查是否启用。在生产环境里，它通常配合 Kubernetes 的 liveness/readiness、负载均衡摘流和注册中心做服务治理。

---

# 十二、如果问“原理”怎么再深一点

你可以再补一句：

> 其底层本质是 Spring Boot 在容器中收集所有 `HealthContributor`，由 `HealthEndpoint` 统一执行并聚合状态，然后再通过状态映射决定 HTTP 返回码，比如 `UP` 返回 200，`DOWN` 返回 503。这样外部系统就能根据健康端点的返回决定是否继续给该实例分配流量。

---

# 十三、一句话总结

**Spring Boot 健康检查 = Actuator 提供 health 端点 + 各类 HealthIndicator 检查结果聚合 + 对外暴露统一可用状态。**
