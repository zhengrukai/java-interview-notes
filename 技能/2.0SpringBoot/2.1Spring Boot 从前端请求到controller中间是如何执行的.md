### 第三阶段：Spring MVC 核心调度（DispatcherServlet）

这是最关键的部分。请求通过 Filter 后，会进入 Spring MVC 的**中央调度器** —— `DispatcherServlet`。

#### 1. `DispatcherServlet` 接收请求

它是前端控制器（Front Controller），所有进入 Spring 上下文的请求都由它统一分发。

#### 2. 查找处理器（HandlerMapping）

*   **原理**：`HandlerMapping` 扫描了所有的 `@Controller` 和 `@RequestMapping` 注解，维护了一个映射表。
*   **结果**：返回一个 **HandlerExecutionChain**（处理器执行链）。这个链包含两部分：
    *   目标 Handler（也就是你的 Controller 方法）。
    *   一组拦截器（HandlerInterceptor）。

#### 3. 执行 HandlerInterceptor 拦截器

*   **先执行所有拦截器的 preHandle**
    顺序：

    **只要有一个返回 false，请求直接中断，不走 Controller**

    Interceptor1.preHandle → Interceptor2.preHandle → ...

#### 4. 查找适配器（HandlerAdapter）（真正执行 Controller 的工具人）

Spring 不直接调用 Controller.method ()，
而是通过 HandlerAdapter 统一调用。

*   **动作**：它找一个适配器 `HandlerAdapter`。
*   **结果**：对于 `@RequestMapping` 定义的方法，通常使用 `RequestMappingHandlerAdapter`。

#### 5. 执行Controller方法

    handlerAdapter.handle(request, response, handler)

内部干这些事：

1.  解析请求参数（JSON、表单、路径变量…）
    *   **ArgumentResolvers（参数解析器）**：负责处理 `@RequestParam`、`@PathVariable`、`@RequestHeader`。

    *   **DataBinder**：处理数据转换（String 转 Integer）和 JSR-303 参数校验（`@Valid`）。
2.  消息转换器 HttpMessageConverter
    *   **HttpMessageConverters（消息转换器）**：如果你用了 `@RequestBody`，这里会调用 **Jackson** 等库，将输入流中的 JSON 字符串反序列化为 Java 对象。
3.  反射调用你的 @Controller 方法
4.  得到返回值（String / ModelAndView / 对象）

#### 6. 执行完 Controller 之后：往回走

1.  执行拦截器的 postHandle
2.  处理返回值
3.  执行拦截器最后：afterCompletion
4.  最后回到 Filter

    chain.doFilter 之后的代码

## 过滤器（Filter）和拦截器（Interceptor）

#### 一、核心概念

*   **过滤器（Filter）**：属于Servlet 规范（Java EE 标准），由 Web 容器（如 Tomcat、Jetty）管理，是请求进入 Servlet 之前的 “全局拦截器”，作用于所有 Web 请求（包括静态资源、HTML、JS、Controller 请求等），更偏向 “容器级” 的请求处理。
*   **拦截器（Interceptor）**：属于Spring 框架特有，由 Spring 容器管理，仅作用于 Spring MVC 的 Controller 请求（不处理静态资源），更偏向 “业务级” 的请求拦截和处理。

| 对比维度           | 过滤器（Filter）                  | 拦截器（Interceptor）                 |
| -------------- | ---------------------------- | -------------------------------- |
| 所属规范           | Servlet 规范（Java EE）          | Spring 框架自定义                     |
| 作用范围           | 所有 Web 请求（静态资源、Controller 等） | 仅 Spring MVC 的 Controller 请求     |
| 触发时机           | 请求进入 Servlet 之前 / 之后         | 请求进入 Controller 方法之前 / 之后、视图渲染前后 |
| 依赖容器           | 依赖 Web 容器（Tomcat/Jetty）      | 依赖 Spring 容器，可脱离 Web 容器（如单元测试）   |
| 注入 Spring Bean | 原生 Filter 无法直接注入（需特殊配置）      | 可直接注入 Service/Repository 等 Bean  |
| 控制方式           | 基于函数回调                       | 基于 AOP 思想（动态代理）                  |
| 异常处理           | 无法捕获 Controller 内的业务异常       | 可捕获 Controller 内的异常              |

#### 二、使用场景选择

##### 用过滤器的场景（容器级、全局处理）

1.  **统一编码处理**：如所有请求 / 响应设置 UTF-8 编码（最经典场景）。
2.  **跨域请求处理**：通过 Filter 设置 CORS 响应头（Access-Control-Allow-Origin 等）。
    3.\*\* XSS/SQL 注入过滤\*\*：拦截所有请求，清洗参数中的危险字符。
3.  **静态资源拦截**：如限制未登录用户访问静态资源（图片、文件）。
4.  **请求参数预处理**：如统一解析请求体中的加密参数。

##### 用拦截器的场景（业务级、Spring 上下文内）

1.  **登录状态校验**：如未登录用户禁止访问后台接口（示例中的场景）。
2.  **权限校验**：如根据用户角色拦截特定接口（如管理员才能访问 /admin/\*\*）。
3.  **接口日志记录**：记录请求的 URL、参数、响应时间、IP 等（业务相关）。
4.  **接口耗时统计**：统计 Controller 方法的执行时间，做性能监控。
5.  **业务逻辑前置 / 后置处理**：如请求前初始化上下文，请求后清理资源。
6.  **异常统一处理**：拦截 Controller 抛出的异常，返回统一格式的响应。
