# 多数据源原理，以及 @DSTransactional 如何保证多数据源数据库的数据一致性

## 问题
多数据源原理，以及@DSTransactional如何保证多数据源数据库的数据一致性

## 回答
这个问题可以分成两部分：

1. **多数据源原理是什么**
2. **`@DSTransactional` 怎么保证多数据源下的数据一致性**

下面按通用原理讲，不结合具体项目。

---

# 一、多数据源原理

## 1. 单数据源时 Spring 是怎么工作的

默认情况下，Spring Boot 只会围绕一个数据源工作：

- 一个 `DataSource`
- 一个事务管理器 `PlatformTransactionManager`
- 一套 MyBatis / JPA 配置
- 当前线程绑定一个数据库连接

也就是说，默认模型是：

```text
请求线程
  -> DataSource
  -> Connection
  -> TransactionManager
```

---

## 2. 多数据源要解决什么问题

如果系统里有多个库，比如：

- `master`
- `slave`
- `order`
- `user`

那框架必须回答几个问题：

1. 当前 SQL 应该走哪个数据源？
2. 当前事务应该绑定哪个连接？
3. 同一个方法里如果切换了多个数据源，怎么处理事务？
4. Mapper / Repository 最终怎么拿到正确的数据源？

---

## 3. 多数据源的两种主流实现方式

### 方式一：静态多数据源
也叫“多套数据源配置”。

做法通常是：

- 配多个 `DataSource`
- 配多个 `SqlSessionFactory`
- 配多个 `TransactionManager`
- 不同包下的 Mapper 绑定到不同数据源

例如：

- `userMapper` 永远走 user 库
- `orderMapper` 永远走 order 库

这种方式特点是：

- 配置明确
- 路由固定
- 适合库边界很清晰的系统

### 方式二：动态多数据源
也叫“路由数据源”。

做法通常是：

- 对外暴露一个“总数据源”
- 底层维护多个真实数据源
- 执行 SQL 时再决定当前使用哪个数据源

这通常基于：

- `AbstractRoutingDataSource`
- `ThreadLocal`
- AOP
- 注解，例如 `@DS("master")`

这是现在更常见的方案。

---

# 二、动态多数据源底层原理

## 1. 核心：路由数据源

Spring 提供了一个类：

```java
AbstractRoutingDataSource
```

它的设计思路就是：

> 你每次获取连接时，我先问一下“当前应该用哪个数据源 key”，然后再从目标数据源集合里取出对应真实数据源。

通常会重写：

```java
protected Object determineCurrentLookupKey()
```

比如返回：

- `"master"`
- `"slave"`
- `"order"`
- `"user"`

---

## 2. 数据源 key 放在哪

通常放在线程上下文里，也就是 `ThreadLocal`。

例如：

```java
private static final ThreadLocal<String> CONTEXT = new ThreadLocal<>();
```

业务调用前先设置：

```java
CONTEXT.set("slave");
```

执行结束后清理：

```java
CONTEXT.remove();
```

所以本质是：

**当前线程带着“我要用哪个数据源”的标记去执行 SQL。**

---

## 3. 谁来设置这个 key

一般是 AOP。

比如你写：

```java
@DS("slave")
public List<User> query() { ... }
```

框架在方法执行前拦截到这个注解，然后：

1. 把 `"slave"` 放进 ThreadLocal
2. 执行方法
3. 方法结束后清理上下文

这样方法里面的 Mapper 查询，最终拿连接时就会走 `slave`。

---

## 4. 整个流程长什么样

```text
调用 Service 方法
   ↓
AOP 拦截 @DS("slave")
   ↓
ThreadLocal 写入 slave
   ↓
Mapper 执行 SQL
   ↓
AbstractRoutingDataSource.determineCurrentLookupKey()
   ↓
返回 slave
   ↓
从 slave DataSource 获取 Connection
   ↓
执行 SQL
   ↓
清理 ThreadLocal
```

---

# 三、多数据源和事务为什么会冲突

## 1. Spring 事务本来是怎么工作的

Spring 的 `@Transactional` 本质上会：

- 在方法开始时开启事务
- 从某个 `DataSource` 获取连接
- 把这个连接绑定到当前线程
- 后续数据库操作尽量复用这个连接
- 最后统一提交或回滚

也就是说：

**Spring 默认事务模型更适合“一个事务对应一个数据源连接”。**

---

## 2. 多数据源下问题来了

假设一个方法中：

1. 先操作 `master`
2. 再切换到 `order`
3. 再切换到 `user`

这时实际上可能拿到了多个连接：

- `master` 的连接
- `order` 的连接
- `user` 的连接

而普通 `@Transactional` 的核心假设不是这种“一个方法多个数据源连接”的场景。

所以会出现问题：

- 普通事务管理器只认一个资源
- 动态切换后不一定能统一管理多个连接
- 可能 A 库提交了，B 库没提交

---

# 四、`@DSTransactional` 的原理

## 1. 它是干什么的

`@DSTransactional` 是动态多数据源框架在事务层做的增强。

它解决的问题是：

> 同一个业务方法里，如果实际访问了多个数据源，如何把这些数据源的本地事务一起管理起来。

也就是说，它不是只管一个连接，而是想办法管理：

- 当前调用链中涉及到的多个数据源连接
- 每个连接的提交 / 回滚
- 数据源切换上下文

---

## 2. 大致执行过程

虽然实现细节因框架不同而不同，但核心逻辑通常是这样的：

### 第一步：AOP 拦截 `@DSTransactional`
方法执行前，先建立一个“分布式数据源事务上下文”。

这个上下文一般也放在线程变量里，记录：

- 当前事务 ID
- 已参与事务的数据源集合
- 每个数据源拿到的连接
- 事务嵌套层次
- 是否出现异常

### 第二步：业务执行时按需获取多个连接
业务执行过程中，如果代码切换到了不同数据源，比如：

- 先 `master`
- 再 `order`
- 再 `user`

那么每次获取连接时，框架会：

1. 判断当前是不是处于 `@DSTransactional` 上下文
2. 如果是，就为该数据源创建或复用一个连接
3. 把该连接注册到当前事务上下文里
4. 把连接设为手动提交

于是一个方法里就可能维护一组连接：

```text
事务上下文
  ├─ master -> ConnectionA
  ├─ order  -> ConnectionB
  └─ user   -> ConnectionC
```

### 第三步：正常结束时统一提交
如果业务方法执行成功，框架会遍历当前事务上下文里的所有连接，然后依次提交。

比如：

1. commit ConnectionA
2. commit ConnectionB
3. commit ConnectionC

### 第四步：异常时统一回滚
如果业务方法抛出异常，框架会遍历所有已参与事务的连接，然后依次回滚：

1. rollback ConnectionA
2. rollback ConnectionB
3. rollback ConnectionC

### 第五步：清理上下文
最终清理：

- ThreadLocal 中的数据源上下文
- ThreadLocal 中的事务资源
- 连接对象
- 嵌套计数

避免线程复用导致串库。

---

# 五、它为什么能“保证一致性”

这里要特别严谨。

## 1. 它能保证什么

`@DSTransactional` 能保证的是：

### 在一个应用进程内
### 在一个方法调用链内
### 对多个数据源的本地事务进行“统一协调提交或统一协调回滚”

也就是说，它提升的是：

**“同 JVM 内多数据源本地事务协调能力”**

---

## 2. 它不能绝对保证什么

它不能像真正的 XA/2PC 那样，提供严格数据库级全局强一致。

原因很简单：

它通常不是数据库层的两阶段提交协议，而是应用层做协调。

例如极端情况：

1. `master` 提交成功
2. `order` 提交成功
3. `user` 提交时网络异常

这时就可能出现：

- 前两个库已经提交
- 第三个库失败

这说明：

**它不是严格意义上的“原子性跨库事务”**

所以更准确地说：

> `@DSTransactional` 通过应用层统一管理多个数据源的本地事务，尽量做到一致提交/一致回滚，但不等于真正的分布式强一致事务。

---

# 六、那它和 `@Transactional` 的本质差异是什么

| 对比项 | `@Transactional` | `@DSTransactional` |
|---|---|---|
| 目标 | 单数据源本地事务 | 多数据源本地事务协调 |
| 依赖 | Spring 事务管理器 | 动态数据源框架扩展 |
| 事务资源 | 通常一个连接 | 可能多个连接 |
| 是否处理动态切换 | 弱 | 强 |
| 是否适合多数据源 | 一般不够 | 更适合 |
| 是否强一致分布式事务 | 否 | 也否 |

---

# 七、实际理解它最好的方式

你可以把 `@DSTransactional` 理解成：

> **“一个事务协调器” + “多个本地数据库连接”**

它做的事情不是把多个库变成一个真正的数据库事务，而是：

- 记录这个方法用到了哪些库
- 给每个库各拿一个连接
- 方法成功时一起提交
- 方法失败时一起回滚

这是“协调型事务”，不是“真正的跨库原子事务”。

---

# 八、什么时候它足够用

它适合这些场景：

1. **同一个服务内**
2. **多个数据源**
3. **业务复杂度不想上 XA**
4. **允许极小概率极端异常下人工补偿**
5. **大部分情况下希望统一提交/回滚**

例如：

- 同服务内读写多个业务库
- 主从 / 分库 / 租户库切换
- 一次业务方法跨多个数据源操作

---

# 九、什么时候它不够用

如果你是以下场景，就不能只靠 `@DSTransactional`：

1. **多个微服务**
2. **跨进程调用**
3. **金融级强一致**
4. **任何一个库失败都绝不允许部分成功**

这时要考虑：

- XA / 2PC
- Seata
- TCC
- Saga
- Outbox + 最终一致性
- 补偿机制

---

# 十、一个例子帮助理解

假设有三个数据源：

- `dsUser`
- `dsOrder`
- `dsPay`

业务方法：

```java
@DSTransactional
public void createOrder() {
    userService.freezeAmount();   // dsUser
    orderService.createOrder();   // dsOrder
    payService.createPayRecord(); // dsPay
}
```

执行逻辑大致是：

### 方法开始
建立事务上下文。

### 调 `freezeAmount`
切到 `dsUser`，拿连接 A，加入事务上下文。

### 调 `createOrder`
切到 `dsOrder`，拿连接 B，加入事务上下文。

### 调 `createPayRecord`
切到 `dsPay`，拿连接 C，加入事务上下文。

### 如果都成功
依次提交：

- A
- B
- C

### 如果中途抛异常
依次回滚：

- 已加入事务的连接全部回滚

---

# 十一、一句话总结

## 多数据源原理
多数据源的核心是：

**通过路由数据源 + ThreadLocal + AOP，在运行时为当前线程选择正确的数据源连接。**

## `@DSTransactional` 原理
它的核心是：

**在一个方法调用链内，把多个数据源对应的本地连接统一纳入事务上下文管理，尽量做到统一提交和统一回滚。**

## 但要记住
**它是“多数据源本地事务协调”，不是“真正跨库强一致分布式事务”。**
