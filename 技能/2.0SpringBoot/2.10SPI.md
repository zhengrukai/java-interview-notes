# SPI 原理及实现（结合 system-server 项目）

## 项目中的 SPI 文件

```
META-INF/services/
├── com.anji.captcha.service.CaptchaService
│   └── cn.zrkcoder...PictureWordCaptchaServiceImpl
└── com.anji.captcha.service.CaptchaCacheService
    └── cn.zrkcoder...RedisCaptchaServiceImpl
```

---

## 一、什么是 SPI

**SPI（Service Provider Interface）** = 服务发现机制，让接口的**实现方**来决定用哪个实现，而非调用方硬编码。

```
┌───────────────┐          ┌──────────────────┐
│  框架/库       │          │  你的项目          │
│  (anji-captcha)│          │  (system-server)  │
│               │          │                   │
│  定义接口：    │  ◀─SPI─  │  提供实现：         │
│  CaptchaService│         │  PictureWordCaptchaServiceImpl │
│  CaptchaCacheService│    │  RedisCaptchaServiceImpl       │
└───────────────┘          └──────────────────┘

框架不依赖你的代码，但运行时能找到并加载你的实现
```

核心思想：**面向接口编程 + 运行时动态发现实现类**。

## 二、Java 原生 SPI

### 1. 三步实现

```
第一步：框架定义接口
──────────────────
public interface CaptchaService {
    void get(CaptchaVO captchaVO);
    ResponseModel check(CaptchaVO captchaVO);
}

第二步：你的项目实现接口
──────────────────
public class PictureWordCaptchaServiceImpl implements CaptchaService {
    @Override
    public void get(CaptchaVO captchaVO) { /* 自定义实现 */ }
}

第三步：在 META-INF/services 下声明
──────────────────
文件名 = 接口全限定名
文件内容 = 实现类全限定名
```

### 2. 加载原理：ServiceLoader

```java
ServiceLoader<CaptchaService> loader = ServiceLoader.load(CaptchaService.class);
for (CaptchaService impl : loader) {
    impl.get(captchaVO);
}
```

**底层流程**：

```
ServiceLoader.load(接口.class)
        │
        ▼
读取 classpath 下所有 JAR 中的
META-INF/services/接口全限定名
        │
        ▼
逐行读取实现类全限定名
        │
        ▼
通过 Class.forName() 加载类
        │
        ▼
通过反射 newInstance() 创建实例
        │
        ▼
返回所有实现类的迭代器
```

### 3. 项目中的效果

anji-captcha 框架通过 SPI 发现你的实现：
- **CaptchaService** → `PictureWordCaptchaServiceImpl`（图片文字验证码）
- **CaptchaCacheService** → `RedisCaptchaServiceImpl`（Redis 缓存验证码）

## 三、Spring Boot 中的 SPI

### 1. Spring Factories（Spring Boot 2.x）

```properties
# META-INF/spring.factories
org.springframework.boot.autoconfigure.EnableAutoConfiguration=\
  com.example.MyAutoConfiguration,\
  com.example.AnotherAutoConfiguration
```

### 2. AutoConfiguration.imports（Spring Boot 3.x）

```
# META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports
com.example.MyAutoConfiguration
com.example.AnotherAutoConfiguration
```

### 3. 三种 SPI 对比

| 维度 | Java SPI | Spring Boot SPI |
|------|----------|-----------------|
| 配置文件位置 | META-INF/services | META-INF/spring.factories 或 spring/*.imports |
| 文件命名 | 接口全限定名 | 固定文件名 |
| 加载器 | ServiceLoader | SpringFactoriesLoader |
| 一个文件 | 只声明一个接口 | 可声明多组映射 |
| 条件过滤 | ❌ 全部加载 | ✅ @Conditional 按需加载 |

### 4. Spring Boot 自动配置的 SPI 流程

```
@SpringBootApplication 启动
        │
        ▼
@EnableAutoConfiguration
        │
        ▼
SpringFactoriesLoader.loadFactoryNames()
        │
        ▼
扫描所有 JAR 的 META-INF/spring.factories
        │
        ▼
读取 EnableAutoConfiguration 对应的所有配置类
        │
        ▼
@Conditional 条件过滤
  ├── @ConditionalOnClass         存在某个类？
  ├── @ConditionalOnMissingBean   用户没自定义？
  ├── @ConditionalOnProperty      配置文件有某属性？
  └── ...
        │
        ▼
符合条件的配置类生效，注册 Bean
```

## 四、Java SPI 的优缺点

| 优点 | 缺点 |
|------|------|
| 解耦，面向接口编程 | 无法按需加载，全部实例化 |
| 可插拔，替换实现无需改代码 | 获取不到具体某一个实现 |
| 框架扩展的标准方式 | 多线程使用 ServiceLoader 不安全 |
| | 无依赖注入，无法使用 Spring Bean |

## 五、总结

```
你的项目中：

META-INF/services/com.anji.captcha.service.CaptchaService
  → 使用 Java 原生 SPI
  → 让 anji-captcha 框架发现并使用你的 PictureWordCaptchaServiceImpl
  → 实现了验证码服务的可插拔替换

SPI 本质：
  接口在框架中定义 → 实现在你的项目中提供 → 配置文件声明映射 → 运行时动态发现

  Java SPI:   META-INF/services/      + ServiceLoader
  Spring SPI: META-INF/spring.factories + SpringFactoriesLoader + @Conditional
```
