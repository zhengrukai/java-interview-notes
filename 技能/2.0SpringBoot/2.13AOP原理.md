# AOP 原理

## 问题
AOP原理

## 回答
AOP 原理，核心一句话：

> **不改原有业务代码，在方法执行前后插入增强逻辑。**

比如：

- 调方法前打印日志
- 调方法后提交事务
- 出异常时回滚
- 方法执行前做权限校验
- 统计耗时

这些都属于 AOP 的典型用途。

---

# 一、AOP 是什么

AOP 全称：

**Aspect Oriented Programming，面向切面编程**

它解决的问题是：

> 有一些“横切关注点”会散落在很多业务方法里，如果每个地方都手写一遍，会很重复、很难维护。

比如：

- 日志
- 事务
- 权限
- 缓存
- 监控
- 限流

这些逻辑本身不是核心业务，但很多地方都要用。

所以 AOP 的思路就是：

**把这些公共逻辑从业务代码里抽出来，统一织入。**

---

# 二、先用一个最简单的例子理解

假设没有 AOP，你要统计方法耗时：

```java
public void saveOrder() {
    long start = System.currentTimeMillis();
    try {
        // 业务逻辑
    } finally {
        System.out.println(System.currentTimeMillis() - start);
    }
}
```

每个方法都写一遍会很烦。

用 AOP 后，你只写业务：

```java
public void saveOrder() {
    // 业务逻辑
}
```

然后统一写一个切面：

- 方法执行前记录开始时间
- 方法执行后打印耗时

业务代码不变，但运行效果一样。

---

# 三、AOP 的核心思想

AOP 本质上就是：

**代理 + 拦截 + 增强**

也就是说，调用的往往不是原始对象本身，而是一个“代理对象”。

调用流程从：

```text
调用方 -> 目标对象
```

变成：

```text
调用方 -> 代理对象 -> 目标对象
```

代理对象会在调用目标方法前后插入额外逻辑。

---

# 四、AOP 里的几个核心概念

## 1. Aspect（切面）
切面就是一组横切逻辑。

比如：

- 日志切面
- 事务切面
- 权限切面

Spring 里通常就是：

```java
@Aspect
@Component
public class LogAspect {
}
```

---

## 2. Join Point（连接点）
连接点可以理解成：

**程序执行过程中，可以被拦截的点。**

在 Spring AOP 里，连接点通常就是：

- 方法执行

注意：
Spring AOP 主要只支持**方法级别**连接点。

---

## 3. Pointcut（切点）
切点就是：

**你到底要拦哪些方法。**

比如：

- 拦截某个包下所有 Service 方法
- 拦截带某个注解的方法
- 拦截某个类的某个方法

例如：

```java
@Pointcut("execution(* com.demo.service.*.*(..))")
public void serviceMethods() {}
```

---

## 4. Advice（通知/增强）
通知就是：

**在切点命中的方法前后，到底要做什么。**

常见有：

- `@Before`：方法前
- `@After`：方法后
- `@AfterReturning`：返回后
- `@AfterThrowing`：抛异常后
- `@Around`：环绕，最强大

---

## 5. Target（目标对象）
也就是原始业务对象。

比如：

```java
OrderService
```

---

## 6. Proxy（代理对象）
AOP 最关键的东西。

Spring 实际放到容器里给你用的，很多时候不是原始 `OrderService`，而是它的代理对象。

调用代理对象的方法时，增强逻辑才有机会执行。

---

# 五、AOP 到底怎么工作的

## 1. Spring 启动时做了什么
Spring 在启动时会扫描：

- 哪些类是 Bean
- 哪些类带 `@Aspect`
- 哪些切点表达式匹配哪些 Bean 方法

然后决定：

> 哪些 Bean 需要被代理。

---

## 2. 创建代理对象
如果某个 Bean 被某个切面命中，Spring 就不会直接把原始对象丢给你，而是创建一个代理对象。

比如：

原始对象：

```text
OrderServiceImpl
```

Spring 最终注入给别人的可能是：

```text
OrderServiceImpl$$Proxy
```

---

## 3. 调用方法时发生什么
当你调用：

```java
orderService.createOrder();
```

实际上调用到的是代理对象。

代理对象内部大致逻辑像这样：

```java
before();
try {
    target.createOrder();
    afterReturning();
} catch (Throwable e) {
    afterThrowing();
    throw e;
} finally {
    after();
}
```

如果是 `@Around`，就更像：

```java
around(() -> target.createOrder());
```

---

# 六、`@Around` 为什么最强

因为它可以完整控制目标方法是否执行。

例如：

```java
@Around("pointcut()")
public Object around(ProceedingJoinPoint pjp) throws Throwable {
    System.out.println("before");
    Object result = pjp.proceed();
    System.out.println("after");
    return result;
}
```

这里：

- `pjp.proceed()` 前：前置逻辑
- `pjp.proceed()` 后：后置逻辑

如果你不调用 `proceed()`，目标方法甚至可以不执行。

这就是为什么：

- 权限校验
- 缓存拦截
- 限流
- 重试
- 事务

都很喜欢用 `@Around`

---

# 七、Spring AOP 底层是怎么生成代理的

Spring AOP 主要有两种代理方式。

## 1. JDK 动态代理
如果目标类实现了接口，Spring 默认优先可能用 JDK 动态代理。

特点：

- 基于接口
- 生成一个实现同样接口的代理类

例如：

```java
public interface OrderService {
    void createOrder();
}
```

Spring 会生成一个代理对象，也实现 `OrderService`。

### 优点
- JDK 原生支持
- 不依赖字节码子类继承

### 缺点
- 只能代理接口方法

---

## 2. CGLIB 代理
如果目标类没有接口，或者强制使用类代理，Spring 会用 CGLIB。

特点：

- 通过生成目标类的子类来做代理
- 重写父类方法，在方法前后插入增强逻辑

例如：

```java
class OrderServiceImpl {
    public void createOrder() {}
}
```

Spring 可能生成：

```java
class OrderServiceImpl$$EnhancerBySpringCGLIB extends OrderServiceImpl
```

然后重写 `createOrder()`。

### 优点
- 不要求接口
- 更灵活

### 缺点
- 不能代理 `final` 类
- 不能增强 `final` 方法
- 本质是继承，有限制

---

# 八、为什么 final / private 方法 AOP 往往不生效

因为 Spring AOP 是基于代理的。

## 1. `private` 方法
代理对象无法从外部覆盖或拦截 private 方法调用。

## 2. `final` 方法
CGLIB 通过“子类重写”实现增强，但 `final` 方法不能被重写，所以增强不了。

## 3. 同类内部调用失效
这是最经典的问题。

例如：

```java
@Service
public class OrderService {

    public void a() {
        b();
    }

    @Transactional
    public void b() {
    }
}
```

你以为 `a()` 调 `b()` 会触发事务，实际上往往不会。

原因是：

- `a()` 里调用 `b()` 是 `this.b()`
- 没经过代理对象
- AOP 根本没介入

所以 Spring AOP 有个经典限制：

> **只有“通过代理对象发起的方法调用”才能触发切面。**

---

# 九、AOP 和事务是什么关系

Spring 事务本质上就是 AOP 的一个典型应用。

比如：

```java
@Transactional
public void save() {
}
```

底层并不是注解自己 magically 生效，而是：

- Spring 为这个 Bean 创建代理
- 调用方法时先进入事务拦截器
- 开启事务
- 执行业务方法
- 成功则提交
- 异常则回滚

所以可以说：

**Spring 声明式事务 = AOP + 事务管理器**

---

# 十、AOP 和动态代理的关系

很多人会混。

关系是：

- **AOP 是思想**
- **动态代理是实现手段**

AOP 说的是：

> 我要在不改业务代码的前提下插入横切逻辑

动态代理说的是：

> 我可以通过代理对象来包一层实现这个目标

在 Spring 里，AOP 的落地主要就是靠动态代理。

# 十一、一个完整执行链路示意

假设有一个日志切面：

```java
@Aspect
@Component
public class LogAspect {

    @Around("execution(* com.demo.service.*.*(..))")
    public Object around(ProceedingJoinPoint pjp) throws Throwable {
        System.out.println("before");
        Object result = pjp.proceed();
        System.out.println("after");
        return result;
    }
}
```

业务类：

```java
@Service
public class OrderService {
    public void createOrder() {
        System.out.println("执行业务");
    }
}
```

实际调用链路：

```text
调用方
  -> OrderService代理对象
      -> around 前置逻辑
      -> 调用目标对象 createOrder()
      -> around 后置逻辑
  -> 返回结果
```

输出顺序：

```text
before
执行业务
after
```

# 十二、Spring AOP 的限制

Spring AOP 不是完整 AspectJ，它有边界。

### 1. 主要拦截方法执行
不是所有字节码级别连接点都支持。

### 2. 基于代理
所以受代理机制限制：

- 同类自调用失效
- private/final 方法受限
- final 类受限

### 3. Bean 外部调用才最稳定
必须经过 Spring 容器里的代理对象。

---

# 十三、如果一句话解释 AOP 原理

你可以这样说：

> AOP 的本质是通过代理对象在目标方法执行前后织入增强逻辑。Spring 在启动时识别切面和切点，为命中的 Bean 创建 JDK 动态代理或 CGLIB 代理。调用方法时实际先进入代理，再执行前置、后置、异常、环绕等通知逻辑，最后再调用目标方法，因此可以在不改业务代码的情况下统一实现日志、事务、权限、监控等横切能力。

---

# 十四、面试简答版

如果是面试，简短版可以这么答：

> AOP 是面向切面编程，用来把日志、事务、权限这类横切逻辑从业务代码中抽离出来。Spring AOP 的底层核心是动态代理，如果目标类有接口一般使用 JDK 动态代理，没有接口通常使用 CGLIB。Spring 在启动时会解析切面、切点和通知，为匹配到的 Bean 创建代理对象。业务方法被调用时，实际上先进入代理对象，再执行前置、后置、异常或环绕通知，最后调用目标方法。像 `@Transactional` 本质上就是基于 AOP 实现的。
