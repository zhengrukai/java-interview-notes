## 一句话结论

CompletableFuture 是 JDK 8 引入的异步编程工具，实现了 `Future` + `CompletionStage` 接口，支持**链式调用、组合编排、异常处理**，是对传统 Future 的全面升级，生产中配合自定义线程池使用。

---

### Future vs CompletableFuture

| 维度 | Future | CompletableFuture |
|------|--------|-------------------|
| 获取结果 | `get()` 阻塞等待 | 回调式（`thenApply` 等），**非阻塞** |
| 链式调用 | 不支持 | 支持任意深度链式编排 |
| 组合多任务 | 不支持 | `allOf` / `anyOf` / `thenCombine` |
| 异常处理 | `get()` 抛 ExecutionException | `exceptionally` / `handle` 优雅处理 |
| 手动完成 | 不支持 | `complete()` / `completeExceptionally()` |
| 回调通知 | 不支持 | `thenAccept` / `thenRun` / `whenComplete` |

---

### 核心 API 分类

#### 1. 创建异步任务

```java
// 无返回值
CompletableFuture<Void> cf1 = CompletableFuture.runAsync(() -> {
    System.out.println("异步任务执行");
});

// 有返回值
CompletableFuture<String> cf2 = CompletableFuture.supplyAsync(() -> {
    return "查询结果";
});

// 指定线程池（生产必须指定！默认用 ForkJoinPool.commonPool，线程数有限）
ExecutorService pool = Executors.newFixedThreadPool(10);
CompletableFuture<String> cf3 = CompletableFuture.supplyAsync(() -> "result", pool);
```

> **重要**：生产环境**必须指定自定义线程池**，不要用默认的 `ForkJoinPool.commonPool()`（全局共享、线程数 = CPU 核心数 - 1，IO 密集任务会被饿死）

#### 2. 链式转换（结果加工）

| 方法 | 入参 | 返回值 | 说明 |
|------|------|--------|------|
| `thenApply(Function)` | 上一步结果 | 新结果 | 类似 Stream 的 map |
| `thenAccept(Consumer)` | 上一步结果 | void | 消费结果，无返回 |
| `thenRun(Runnable)` | 无 | void | 不关心上一步结果，只做后续动作 |

```java
CompletableFuture.supplyAsync(() -> "Hello")
    .thenApply(s -> s + " World")       // "Hello World"
    .thenApply(String::toUpperCase)     // "HELLO WORLD"
    .thenAccept(System.out::println);   // 输出：HELLO WORLD
```

**Async 后缀版本**：`thenApplyAsync` 等会在**另一个线程**中执行回调（默认 ForkJoinPool 或指定线程池），无 Async 后缀的在**上一步的线程**中执行。

#### 3. 组合多个任务

**两个任务都完成后合并**：
```java
CompletableFuture<String> userFuture = CompletableFuture.supplyAsync(() -> queryUser());
CompletableFuture<String> orderFuture = CompletableFuture.supplyAsync(() -> queryOrder());

// thenCombine：两个都完成后，合并结果
CompletableFuture<String> result = userFuture.thenCombine(orderFuture,
    (user, order) -> user + " 的订单：" + order);
```

**多个任务全部完成**：
```java
CompletableFuture<Void> all = CompletableFuture.allOf(cf1, cf2, cf3);
all.join(); // 等待全部完成
// 逐个获取结果
String r1 = cf1.join();
String r2 = cf2.join();
```

**多个任务任一完成**：
```java
CompletableFuture<Object> any = CompletableFuture.anyOf(cf1, cf2, cf3);
Object firstResult = any.join(); // 返回最先完成的结果
```

| 方法 | 语义 | 典型场景 |
|------|------|---------|
| `thenCombine` | 两个都完成，合并结果 | 聚合查询 |
| `thenCompose` | 串行依赖（flatMap） | 先查用户再查订单 |
| `allOf` | 全部完成 | 并发调多个接口，全部返回后汇总 |
| `anyOf` | 任一完成 | 多数据源竞速，取最快的 |

#### 4. 异常处理

```java
CompletableFuture.supplyAsync(() -> {
        if (true) throw new RuntimeException("查询失败");
        return "result";
    })
    // 方式1：exceptionally（类似 catch，只处理异常）
    .exceptionally(e -> {
        System.out.println("异常：" + e.getMessage());
        return "默认值";
    })
    // 方式2：handle（无论正常/异常都处理，类似 finally）
    .handle((result, ex) -> {
        if (ex != null) return "降级结果";
        return result;
    })
    // 方式3：whenComplete（不改变结果，只做日志/监控）
    .whenComplete((result, ex) -> {
        if (ex != null) log.error("任务失败", ex);
        else log.info("任务成功：{}", result);
    });
```

| 方法 | 能否修改返回值 | 类比 |
|------|-------------|------|
| `exceptionally` | 能（提供降级值） | catch |
| `handle` | 能（正常/异常都处理） | try-catch-finally |
| `whenComplete` | 不能（只观察） | finally 中的日志 |

### 生产最佳实践

#### 1. 并发调用多个接口并汇总（最常见场景）

```java
// 商品详情页：并发查询商品、库存、评价
ExecutorService pool = new ThreadPoolExecutor(8, 16, 60, TimeUnit.SECONDS,
    new ArrayBlockingQueue<>(200));

CompletableFuture<Product> productFuture = CompletableFuture.supplyAsync(
    () -> productService.getById(id), pool);
CompletableFuture<Stock> stockFuture = CompletableFuture.supplyAsync(
    () -> stockService.getStock(id), pool);
CompletableFuture<List<Review>> reviewFuture = CompletableFuture.supplyAsync(
    () -> reviewService.getReviews(id), pool);

// 等待全部完成
CompletableFuture.allOf(productFuture, stockFuture, reviewFuture).join();

// 组装结果
ProductDetailVO vo = new ProductDetailVO();
vo.setProduct(productFuture.join());
vo.setStock(stockFuture.join());
vo.setReviews(reviewFuture.join());
```

**性能对比**：串行调用 300ms + 200ms + 150ms = 650ms；并发调用 max(300, 200, 150) = **300ms**

#### 2. 超时控制（JDK 9+）

```java
CompletableFuture<String> cf = CompletableFuture.supplyAsync(() -> slowQuery())
    .orTimeout(3, TimeUnit.SECONDS)           // 超时抛 TimeoutException
    .completeOnTimeout("默认值", 3, TimeUnit.SECONDS); // 超时返回默认值
```

JDK 8 下的超时控制：
```java
CompletableFuture<String> cf = CompletableFuture.supplyAsync(() -> slowQuery());
try {
    String result = cf.get(3, TimeUnit.SECONDS); // get 的超时版本
} catch (TimeoutException e) {
    cf.cancel(true);
    return "默认值";
}
```

### 常见陷阱

| 陷阱 | 后果 | 正确做法 |
|------|------|---------|
| 不指定线程池 | 默认 ForkJoinPool 线程耗尽，影响全局 | 必须传入自定义线程池 |
| 忘记处理异常 | 异常被静默吞掉，业务无感知 | 链尾加 `exceptionally` 或 `whenComplete` |
| `join()` 放主线程 | 阻塞主线程，失去异步意义 | 尽量用回调链，避免在请求线程 join |
| 回调链太长 | 调试困难，堆栈不直观 | 适度拆分，关键步骤加日志 |
| `allOf` 后忘记获取结果 | `allOf` 返回 `Void`，需逐个 join 获取 | 明确逐个获取每个 Future 的结果 |

### thenApply vs thenCompose（高频对比）

```java
// thenApply：同步转换（类似 map）
// Function<T, U> → CompletableFuture<U>
cf.thenApply(s -> s.toUpperCase());

// thenCompose：异步链式（类似 flatMap）
// Function<T, CompletableFuture<U>> → CompletableFuture<U>
cf.thenCompose(userId -> CompletableFuture.supplyAsync(() -> queryOrder(userId)));
```

| | thenApply | thenCompose |
|---|----------|-------------|
| 入参返回值 | `T → U` | `T → CompletableFuture<U>` |
| 类比 Stream | `map()` | `flatMap()` |
| 使用场景 | 简单同步转换 | 需要再发起异步调用 |

### 高频追问

1. **CompletableFuture 默认线程池是什么？有什么问题？** `ForkJoinPool.commonPool()`，线程数 = CPU 核心数 - 1，全局共享。IO 密集型任务会阻塞线程，导致其他任务饿死
2. **join() 和 get() 的区别？** `join()` 抛未检查异常（CompletionException）；`get()` 抛受检异常（ExecutionException），需 try-catch
3. **如何实现任务编排：A 和 B 并行，都完成后执行 C？** `CompletableFuture.allOf(A, B).thenRun(C)`
4. **CompletableFuture 是线程安全的吗？** 是。内部通过 CAS + volatile 保证状态转换的线程安全
5. **生产环境怎么选：CompletableFuture vs RxJava vs Project Reactor？** 简单异步编排用 CompletableFuture 够用；响应式流（背压、无限流）用 Reactor/RxJava；Spring WebFlux 项目用 Reactor
