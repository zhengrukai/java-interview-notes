CAS（Compare-And-Swap，比较并交换）是硬件级别的原子操作（依赖 CPU 指令集，如 x86 的 cmpxchg），通过无锁方式实现共享变量的原子更新。

它包含三个操作数：

*   V：内存地址中的**实际值**
*   A：更新前的**预期值**
*   B：要更新的**新值**

**核心逻辑**：当且仅当 V 的值等于 A 时，原子地将 V 更新为 B；否则重试（自旋）或放弃。

### CAS的三大痛点与解决方案

#### 1. ABA问题

*   **问题描述**：线程 1 读取变量值为 A，此时线程 2 将 A 改为 B 再改回 A，线程 1 误以为值未变，仍执行更新。
*   **危害**：在某些场景下（如链表操作），可能导致数据结构被破坏。
*   **解决方案**：

    *   **版本号 / 时间戳**：为变量添加版本号（如 AtomicStampedReference），每次更新同时递增版本号，只有 “值 + 版本号” 都匹配才更新。

    // 初始值：100，初始版本号：0
    AtomicStampedReference\<Integer> stampedRef =
    new AtomicStampedReference<>(100, 0);

    // 预期值100，新值200，预期版本号0，新版本号1
    stampedRef.compareAndSet(100, 200, 0, 1);

#### 2. 循环时间长，CPU 开销大

*   **问题描述**：高并发下，CAS 长时间自旋（循环重试）会占用大量 CPU 资源。
*   **解决方案**:
    *   限制自旋次数（如 java.util.concurrent.locks.LockSupport）；
    *   自适应自旋（JVM 中根据历史自旋成功率动态调整）。

#### 3. 只能保证一个共享变量的原子操作

*   **问题描述**：CAS 仅对单个变量有效，无法同时保证多个变量的原子性。
*   **解决方案**：
    *   使用 AtomicReference 包装多个变量（将多个变量封装成一个对象）；
    *   使用锁（如 synchronized）处理多变量原子性。

### CAS 与锁的优劣势

| 维度   | CAS             | 锁（synchronized/Lock） |
| ---- | --------------- | -------------------- |
| 并发思想 | 乐观锁（假设冲突少）      | 悲观锁（假设冲突多）           |
| 性能   | 低冲突下性能高（无上下文切换） | 高冲突下性能稳定（避免自旋）       |
| 适用场景 | 细粒度、低冲突的原子操作    | 粗粒度、高冲突的同步块          |
| 复杂度  | 需处理 ABA、自旋等问题   | 语义简单，由 JVM 托管        |

### Unsafe 类（CAS 的底层依赖）

Unsafe 是 `sun.misc` 包下的类，是 Java 访问底层 CPU 指令的桥梁，CAS 操作的真正执行者。

**核心能力**：
- **CAS 操作**：`compareAndSwapInt()`、`compareAndSwapLong()`、`compareAndSwapObject()`
- **内存操作**：直接操作内存地址（`allocateMemory`、`freeMemory`）
- **线程调度**：`park()`/`unpark()` 挂起/唤醒线程（LockSupport 的底层）

**为什么叫 Unsafe？** 它绕过了 JVM 的安全检查，直接操作内存，使用不当会导致内存泄漏、JVM 崩溃。正常业务代码不应直接使用，但 JUC 包大量依赖它。

```java
// AtomicInteger.incrementAndGet() 底层实现（JDK 8）
public final int getAndAddInt(Object o, long offset, int delta) {
    int v;
    do {
        v = getIntVolatile(o, offset);  // 读取当前值
    } while (!compareAndSwapInt(o, offset, v, v + delta)); // CAS 自旋
    return v;
}
```

### LongAdder vs AtomicLong（高并发计数首选）

**AtomicLong 的瓶颈**：高并发下所有线程 CAS 竞争同一个 `value`，大量自旋浪费 CPU。

**LongAdder 的分段累加思想**：
```
                    LongAdder
                   ┌─────────┐
                   │  base    │  ← 无竞争时直接 CAS 累加
                   ├─────────┤
                   │ Cell[0]  │  ← 线程 hash 映射到不同 Cell
                   │ Cell[1]  │  ← 每个 Cell 独立 CAS，减少竞争
                   │ Cell[2]  │
                   │  ...     │
                   └─────────┘
                sum() = base + ΣCell[i]
```

**核心原理**：
- 内部维护一个 `base` 值 + `Cell[]` 数组
- 无竞争时，直接 CAS 更新 base
- 有竞争时，线程根据 hash 值映射到不同的 Cell，各自 CAS 累加
- 调用 `sum()` 时，汇总 base + 所有 Cell 的值

| 对比 | AtomicLong | LongAdder |
|------|-----------|-----------|
| 竞争策略 | 所有线程 CAS 同一个 value | 分散到多个 Cell |
| 高并发写性能 | 差（自旋严重） | 优（竞争分散） |
| 读取精确性 | 实时精确 | sum() 非原子操作，统计时可能有误差 |
| 适用场景 | 需精确值的场景（如序列号） | 高并发统计计数（如 QPS 统计） |

### 高频追问

1. **CAS 底层是怎么保证原子性的？** CPU 的 `cmpxchg` 指令 + 总线锁/缓存锁，硬件级别保证比较和交换是一个原子操作
2. **Java 中哪些地方用到了 CAS？** AtomicXxx 系列、AQS 的 state 修改、ConcurrentHashMap 的 put、LongAdder
3. **CAS 适合什么场景？** 读多写少、竞争不激烈的细粒度操作。高竞争场景应考虑 LongAdder 或加锁
4. **为什么 LongAdder 不能替代 AtomicLong？** LongAdder 的 sum() 不是原子操作，不适合需要精确实时值的场景（如全局序列号生成）
