ThreadLocal 是线程级别的数据隔离工具，它让每个线程拥有自己独立的变量副本，线程之间互不干扰，主要用于避免参数层层传递和解决线程不安全工具类的复用问题。

### 核心原理(Thread、ThreadLocal、ThreadLocalMap关系)

*   **归属关系**：每个 Thread 线程对象内部都维护了一个 ThreadLocalMap（Thread.threadLocals），这是线程隔离的根本。
*   **存储结构**：ThreadLocalMap 是一个类似 HashMap 的结构，内部是 Entry 数组，Entry 的 key 是 ThreadLocal 实例（弱引用），value 是业务数据（强引用）。
*   **操作逻辑**：ThreadLocal 本身不存储数据，只是作为「钥匙」，通过 get()/set()/remove() 操作当前线程的 ThreadLocalMap。

### 内存泄露问题

*   **为什么会内存泄漏？**

当 ThreadLocal 实例被回收（比如局部变量出栈），由于 key 是弱引用，GC 时会回收 key，导致 ThreadLocalMap 中出现「key 为 null，value 不为 null」的无效条目。如果线程是长生命周期的（比如线程池核心线程），value 会被 Thread 持有的强引用一直指向，无法被 GC 回收，最终堆积导致 OOM。

*   **为什么 key 用弱引用？**

这是 ThreadLocal 的「兜底设计」：如果 key 用强引用，即使 ThreadLocal 实例被销毁，key 仍会强引用它，导致 ThreadLocal 实例无法被回收，内存泄漏更严重。用弱引用至少能保证 ThreadLocal 实例被回收，只是 value 还需手动清理。

*   **如何避免内存泄漏？**

**核心原则**：使用完 ThreadLocal 后，主动调用 remove() 方法（最好放在 finally 块中，确保无论是否抛异常都能执行）。
补充：ThreadLocalMap 在 get()/set() 时会自动清理部分 key 为 null 的无效条目，但这是「兜底机制」，不能依赖，必须手动 remove()。

### 高频问题

**1. ThreadLocal 如何解决哈希冲突？**

> ThreadLocalMap 不像 HashMap 用「链表 + 红黑树」解决冲突，而是用线性探测法：如果当前哈希位置已被占用，就往后找下一个空位置，直到找到为止。删除时也会用「重新哈希」的方式调整后续元素的位置，避免出现空洞

**2. 父子线程间如何传递 ThreadLocal 数据？**

> 可以用 InheritableThreadLocal，它是 ThreadLocal 的子类，在创建子线程时，会将父线程的 InheritableThreadLocal 数据复制到子线程中。
>
> 但要注意：**线程池场景下** InheritableThreadLocal **会失效**（因为线程池线程是复用的，不是每次都创建新线程），此时可以用阿里开源的 TransmittableThreadLocal 来解决，它能在线程池任务执行前后传递数据。

**3. ThreadLocal 和 synchronized 的区别？**

| 维度   | ThreadLocal    | synchronized  |
| ---- | -------------- | ------------- |
| 核心思想 | 线程隔离（每个线程独立副本） | 线程同步（多线程轮流访问） |
| 资源消耗 | 空间换时间（每个线程存一份） | 时间换空间（加锁阻塞）   |
| 适用场景 | 避免参数传递、工具类复用   | 多线程共享资源的同步访问  |
| 线程安全 | 隔离实现安全         | 加锁实现安全        |
