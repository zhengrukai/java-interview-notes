## 一句话结论

死锁是指**两个或多个线程互相持有对方需要的锁，导致所有线程永久阻塞**。死锁的产生需要同时满足四个必要条件，破坏任意一个即可预防。

---

### 死锁的四个必要条件（面试必背）

| 条件 | 含义 | 破坏方式 |
|------|------|---------|
| **互斥** | 资源同一时刻只能被一个线程占用 | 无法破坏（锁的本质） |
| **持有并等待** | 线程持有至少一个资源，同时等待获取其他资源 | 一次性申请所有资源 |
| **不可剥夺** | 线程持有的资源不能被其他线程强制夺走 | 使用 `tryLock()` 超时主动释放 |
| **循环等待** | 线程之间形成环形等待链 | **锁排序**（按固定顺序加锁） |

> **记忆口诀**：互持不循（互斥、持有并等待、不可剥夺、循环等待）

---

### 死锁代码示例

```java
public class DeadLockDemo {
    private static final Object lockA = new Object();
    private static final Object lockB = new Object();

    public static void main(String[] args) {
        // 线程1：先拿 lockA，再拿 lockB
        new Thread(() -> {
            synchronized (lockA) {
                System.out.println("线程1：持有 lockA，等待 lockB...");
                try { Thread.sleep(100); } catch (InterruptedException e) {}
                synchronized (lockB) {
                    System.out.println("线程1：拿到 lockB");
                }
            }
        }, "Thread-1").start();

        // 线程2：先拿 lockB，再拿 lockA（顺序相反 → 死锁！）
        new Thread(() -> {
            synchronized (lockB) {
                System.out.println("线程2：持有 lockB，等待 lockA...");
                try { Thread.sleep(100); } catch (InterruptedException e) {}
                synchronized (lockA) {
                    System.out.println("线程2：拿到 lockA");
                }
            }
        }, "Thread-2").start();
    }
}
```

**结果**：程序永远卡住，两个线程互相等待，永远不会输出 "拿到" 的日志。

### 死锁预防策略

#### 1. 锁排序（破坏循环等待，最常用）

```java
// 错误：不同线程以不同顺序加锁
// 正确：所有线程按相同顺序加锁
private void transfer(Account from, Account to, int amount) {
    // 按账户 ID 排序，保证全局统一的加锁顺序
    Account first = from.getId() < to.getId() ? from : to;
    Account second = from.getId() < to.getId() ? to : from;

    synchronized (first) {
        synchronized (second) {
            from.debit(amount);
            to.credit(amount);
        }
    }
}
```

#### 2. 超时机制（破坏不可剥夺）

```java
ReentrantLock lockA = new ReentrantLock();
ReentrantLock lockB = new ReentrantLock();

// 尝试获取锁，超时则放弃并释放已持有的锁
if (lockA.tryLock(1, TimeUnit.SECONDS)) {
    try {
        if (lockB.tryLock(1, TimeUnit.SECONDS)) {
            try {
                // 业务逻辑
            } finally {
                lockB.unlock();
            }
        } else {
            // 获取 lockB 超时，释放 lockA，避免死锁
        }
    } finally {
        lockA.unlock();
    }
}
```

#### 3. 一次性申请所有资源（破坏持有并等待）

```java
// 使用一个全局锁来保护资源申请过程
synchronized (resourceAllocator) {
    // 一次性申请所有需要的资源
    if (canAllocate(resourceA, resourceB)) {
        allocate(resourceA, resourceB);
    }
}
```

#### 4. 使用并发工具替代手动加锁

| 方案 | 适用场景 |
|------|---------|
| `ConcurrentHashMap` | 替代 synchronized + HashMap |
| `AtomicXxx` | 替代 synchronized + 基本类型 |
| `StampedLock` | 读多写少场景 |
| 消息队列 / Actor 模型 | 避免共享状态 |

### 活锁与饥饿（死锁的兄弟问题）

| 问题 | 表现 | 原因 | 解决 |
|------|------|------|------|
| **死锁** | 线程永久阻塞 | 互相等待对方释放锁 | 锁排序、超时 |
| **活锁** | 线程不阻塞，但不断重试，无实际进展 | 两个线程互相谦让（类似两人窄路相让） | 加入**随机退避**时间 |
| **饥饿** | 某些线程长期得不到执行 | 高优先级线程持续抢占 | 使用**公平锁** |

**活锁示例**：两个线程都检测到死锁风险，同时释放锁并重试，结果又同时获取，反复循环。
**解决**：`tryLock()` 失败后 `Thread.sleep(随机时间)` 再重试。

### 高频追问

1. **如何在代码层面避免死锁？** 锁排序（全局统一加锁顺序）+ tryLock 超时 + 减小锁粒度
2. **生产环境出现死锁怎么排查？** jstack 导出线程栈 → 搜索 "deadlock" → 分析锁持有关系 → 找到加锁顺序不一致的代码
3. **数据库死锁和 Java 死锁一样吗？** 原理相同（循环等待），但数据库有**死锁检测机制**（如 MySQL InnoDB 的 wait-for graph），会主动回滚代价较小的事务
4. **synchronized 会死锁，volatile 会死锁吗？** 不会。volatile 不涉及锁，不存在互斥和等待
5. **线程池会导致死锁吗？** 会。如果任务 A 提交到线程池后，在任务内部又提交任务 B 并等待 B 完成（`future.get()`），当线程池满时，B 无法被执行，A 等 B、B 等线程 → 死锁。解决：使用不同的线程池或增加线程数
