## volatile

> 保证**可见性**（强制刷新主存）和**有序性**（禁止指令重排），但不保证**原子性**
> **可见性**：1、JMM 强制读写直接操作主内存；2、硬件 MESI 协议保证多 CPU 缓存一致性。
> **有序性**：插入内存屏障，禁止编译器和 CPU 对 volatile 变量的读写进行指令重排序。

### DCL 单例中 volatile 的作用（高频⭐⭐⭐⭐⭐）

**错误写法（无 volatile）**：
```java
public class Singleton {
    // 【关键】这里没有 volatile！
    private static Singleton instance;

    private Singleton() {}

    public static Singleton getInstance() {
        if (instance == null) {                    // 第一次检查
            synchronized (Singleton.class) {
                if (instance == null) {             // 第二次检查
                    instance = new Singleton();     // 【核心雷区】
                }
            }
        }
        return instance;
    }
}
```

**问题分析**：`new Singleton()` 分三步执行：
1. 分配内存空间
2. 初始化对象
3. 将引用指向内存地址

JIT 可能将步骤重排为 1→3→2，导致线程 B 在第一次检查时拿到**未初始化的对象**（引用不为 null，但字段还是默认值）。

**正确写法（加 volatile）**：
```java
public class Singleton {
    // volatile 禁止 new 操作的指令重排序
    private static volatile Singleton instance;

    private Singleton() {}

    public static Singleton getInstance() {
        if (instance == null) {                    // 第一次检查（无锁，快速返回）
            synchronized (Singleton.class) {
                if (instance == null) {             // 第二次检查（有锁，防并发）
                    instance = new Singleton();     // volatile 写：1→2→3 严格有序
                }
            }
        }
        return instance;
    }
}
```

**volatile 在此的两个作用**：
1. **禁止重排序**：确保 new 操作的三步严格按 1→2→3 执行
2. **可见性**：确保 instance 赋值后，其他线程立即可见

## synchronized(悲观锁)           &#x20;

> 保证**原子性**、**可见性**、**有序性**
> **原子性**：通过 monitor 互斥锁，保证同步块内的代码同一时间只能被一个线程执行。
> **可见性**：JMM 规定：unlock 前刷新回主内存，lock 时从主内存重新加载。
> **有序性**：不直接禁止重排序，但通过互斥锁让重排序对其他线程不可见（串行化执行）。
>
> **锁升级过程**（单向升级，不可降级）：
>
> ```
> 无锁 → 偏向锁 → 轻量级锁（自旋CAS）→ 重量级锁（OS互斥量）
> ```
>
> | 锁状态 | 适用场景 | 实现原理 | Mark Word 存储 |
> |--------|---------|---------|---------------|
> | 偏向锁 | 只有一个线程访问 | Mark Word 记录线程 ID，后续同线程进入免 CAS | 线程 ID + epoch |
> | 轻量级锁 | 多线程交替执行（无真正竞争） | CAS 将 Mark Word 指向栈中 Lock Record | Lock Record 指针 |
> | 重量级锁 | 多线程真实竞争 | 操作系统 mutex，线程进入内核态阻塞 | Monitor 指针 |
>
> **升级触发条件**：偏向锁遇到第二个线程 → 升级轻量级锁；轻量级锁自旋超过阈值（默认 10 次/自适应）→ 升级重量级锁

**作用范围**：修饰实例方法（锁this）、静态方法(锁 类.class)、代码块（锁 自定义对象），核心是锁定同一个对象才会互斥

**字节码层面**：
- 同步方法：编译后方法标志位增加 `ACC_SYNCHRONIZED`
- 同步代码块：`monitorenter` + `monitorexit`（两个 monitorexit：一个正常退出，一个异常退出）

    public class Counter {
        private int count = 0;

        public synchronized void increment() {
            count++;
        }

        public synchronized int getCount() {
            return count;
        }

    }

### synchronized 与 ReentrantLock 对比（高频⭐⭐⭐⭐⭐）

| 对比维度 | synchronized | ReentrantLock |
|---------|-------------|---------------|
| 实现层级 | JVM 内置（字节码指令） | Java 层（AQS） |
| 锁释放 | 自动释放（代码块结束/异常） | 手动 `unlock()`（必须 finally） |
| 可中断 | 不支持 | `lockInterruptibly()` |
| 公平锁 | 不支持 | 构造器指定 `new ReentrantLock(true)` |
| 条件变量 | 1 个（wait/notify） | 多个（Condition） |
| 锁超时 | 不支持 | `tryLock(timeout)` |
| 锁状态查询 | 不支持 | `isLocked()`、`getHoldCount()` |

**选型建议**：简单同步用 synchronized（JVM 优化好、不会忘记释放）；需要高级特性（可中断、超时、公平、多条件）用 ReentrantLock

## Atomic 变量类(乐观锁)

> 利用底层硬件的 CAS 指令保证特定操作的原子性
> 封装了 volatile 变量，保证可见性。
> java.util.concurrent.atomic 包下的原子变量类（如AtomicInteger、AtomicLong、AtomicReference），核心就是基于 CAS 指令 + 自旋实现线程安全的变量操作，底层依赖Unsafe类（Java 提供的访问底层 CPU 指令的工具类）。

    import java.util.concurrent.atomic.AtomicInteger;

    public class Counter {
        private AtomicInteger count = new AtomicInteger(0);

        public void increment() {
            count.incrementAndGet();
        }

        public int getCount() {
            return count.get();
        }

    }

## 8 条 Happens-Before 规则

1. **程序次序规则**

> 在同一个线程内，按照代码的书写顺序，前面的操作 Happens-Before 后面的操作。

2. **管程锁定规则**

> 对同一个锁的解锁操作 Happens-Before 后续对这个锁的加锁操作。
> 管程就是指 synchronized，这也是 synchronized 能保证可见性的核心原因。

3. **volatile 变量规则**

> 对 volatile 变量的写操作 Happens-Before 后续对这个 volatile 变量的读操作。
> 这是 volatile 能保证可见性的核心原因，也是禁止指令重排序的底层依据。

4. **线程启动规则**

> 调用 Thread.start() 方法的操作 Happens-Before 该线程内部的所有操作。

5. **线程终止规则**

> 线程的所有操作 Happens-Before 检测到该线程终止的操作（如 Thread.join() 返回、Thread.isAlive() 返回 false）。

6. **线程中断规则**

> 调用 Thread.interrupt() 方法的操作 Happens-Before 被中断线程检测到中断的操作（如 Thread.interrupted()、Thread.isInterrupted() 返回 true）。

7. **对象终结规则**

> 对象的构造函数执行完成（对象初始化完毕） Happens-Before 该对象的 finalize() 方法执行。
> 保证对象在被垃圾回收前，已经完成初始化，避免访问未初始化的对象。

8. **传递性规则**

> 如果 A Happens-Before B，且 B Happens-Before C，那么 A Happens-Before C。

> **总结**：Happens-Before 是 JMM 提供的内存可见性保证，而非 “时间执行顺序”；
