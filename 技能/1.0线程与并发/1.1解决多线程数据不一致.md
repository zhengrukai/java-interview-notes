## volatile

> 保证**可见性**（强制刷新主存）和**有序性**（禁止指令重排），但不保证**原子性**
> **可见性**：1、JMM 强制读写直接操作主内存；2、硬件 MESI 协议保证多 CPU 缓存一致性。
> **有序性**：插入内存屏障，禁止编译器和 CPU 对 volatile 变量的读写进行指令重排序。

修饰变量

    public class Singleton {
        // 【关键】这里没有 volatile！
        private static Singleton instance;

        private Singleton() {} // 私有构造

        public static Singleton getInstance() {
            // 第一次检查：避免不必要的加锁
            if (instance == null) { 
                synchronized (Singleton.class) {
                    // 第二次检查：确保只有一个线程能进入创建
                    if (instance == null) { 
                        // 【核心雷区】这行代码有问题！
                        instance = new Singleton(); 
                    }
                }
            }
            return instance;
        }
    }

## synchronized(悲观锁)           &#x20;

> 保证**原子性**、**可见性**、**有序性**
> **原子性**：通过 monitor 互斥锁，保证同步块内的代码同一时间只能被一个线程执行。
> **可见性**：JMM 规定：unlock 前刷新回主内存，lock 时从主内存重新加载。
> **有序性**：不直接禁止重排序，但通过互斥锁让重排序对其他线程不可见（串行化执行）。
>
> **锁升级过程**(无锁→偏向锁→轻量级锁(自旋)→重量级锁(\_EntryList,存放阻塞线程))

修饰实例方法（锁this）、静态方法(锁 类.class)、代码块（锁 自定义对象），核心是锁定同一个对象才会互斥

    public class Counter {
        private int count = 0;

        public synchronized void increment() {
            count++;
        }

        public synchronized int getCount() {
            return count;
        }

    }

## Atomic 变量类(乐观锁)

> 利用底层硬件的 CAS 指令保证特定操作的原子性
> 封装了 volatile 变量，保证可见性。
> java.util.concurrent.atomic 包下的原子变量类（如AtomicInteger、AtomicLong、AtomicReference），核心就是基于 CAS 指令 + 自旋实现线程安全的变量操作，底层依赖Unsafe类（Java 提供的访问底层 CPU 指令的工具类）。

    import java.util.concurrent.atomic.AtomicInteger;

    public class Counter {
        private AtomicInteger count = new AtomicInteger(0);

        public void increment() {
            count.incrementAndGet();
        }

        public int getCount() {
            return count.get();
        }

    }

## 8 条 Happens-Before 规则

1.  程序次序规则

> 在同一个线程内，按照代码的书写顺序

1.  管程锁定规则

> 对同一个锁的解锁操作 Happens-Before 后续对这个锁的加锁操作。
> 管程就是指 synchronized，这也是 synchronized 能保证可见性的核心原因

1.  volatile 变量规则

> 对 volatile 变量的写操作 Happens-Before 后续对这个 volatile 变量的读操作。
> 这是 volatile 能保证可见性的核心原因，也是禁止指令重排序的底层依据。

1.  线程启动规则

> 调用Thread.start()方法的操作 Happens-Before 该线程内部的所有操作

1.  线程终止规则

> 线程的所有操作 Happens-Before 检测到该线程终止的操作（如Thread.join()返回、Thread.isAlive()返回 false）。

1.  线程中断规则

> 调用Thread.interrupt()方法的操作 Happens-Before 被中断线程检测到中断的操作（如Thread.interrupted()、Thread.isInterrupted()返回 true）。

1.  对象终结规则

> 对象的构造函数执行完成（对象初始化完毕） Happens-Before 该对象的finalize()方法执行。
> 保证对象在被垃圾回收前，已经完成初始化，避免访问未初始化的对象

1.  传递性规则

> 如果 A Happens-Before B，且 B Happens-Before C，那么 A Happens-Before C。

> Happens-Before 是 JMM 提供的内存可见性保证，而非 “时间执行顺序”；
