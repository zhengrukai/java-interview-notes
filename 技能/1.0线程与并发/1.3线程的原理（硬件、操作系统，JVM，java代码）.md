Java 线程的原理是一个从**语言层面**深入到**操作系统（OS）层面**的话题。要理解它，我们需要剥开 JVM 的外壳，看它如何与底层硬件和操作系统交互。

以下是 Java 线程原理的核心解析，分为五个维度：

### 1. 线程模型：从 "Green Threads" 到 "1:1 模型"

这是理解 Java 线程最关键的一点：**现代 Java 线程本质上就是操作系统线程。**

*   **早期（JDK 1.2 之前）：** Java 使用“绿色线程（Green Threads）”。这意味着 JVM 自己模拟多线程，操作系统只看到一个进程。这种模式下，多核 CPU 无法被利用，因为 OS 不知道 JVM 内部有多个线程。
*   **现代（JDK 1.2 及以后）：** Java 采用了 **1:1 线程模型**（基于主流 OS 如 Linux 和 Windows）。
    *   **这意味着：** 当你 `new Thread()` 并调用 `start()` 时，JVM 会通过 JNI (Java Native Interface) 调用操作系统的内核接口，创建一个**内核级线程（Kernel-Level Thread, KLT）**。
    *   **映射关系：** 1 个 Java 线程 = 1 个操作系统内核线程。
    *   **调度者：** 线程的调度（谁先执行，谁后执行，执行多久）完全交给**操作系统**的调度器处理，JVM 不再负责 CPU 时间片的分配。

### 2. 启动流程：`start()` 的底层逻辑

为什么我们必须调用 `start()` 而不是直接调用 `run()`？

1.  **Java 层面：** `thread.start()` 会改变线程状态，并检查线程是否重复启动。
2.  **Native 层面：** 接着调用 `private native void start0()`。
3.  **JVM 层面（HotSpot 源码）：**
    *   JVM 创建一个 C++ 对象 `JavaThread`。
    *   调用操作系统的创建线程 API。
        *   **Linux:** 调用 `pthread_create`（通常对应 `clone` 系统调用）。
        *   **Windows:** 调用 `CreateThread`。
4.  **回调：** 新创建的 OS 线程启动后，会回调 JVM 的入口方法，最终调用 Java 代码中的 `run()` 方法。

### 3. 内存模型（JMM）：私有 vs 共享

在 JVM 运行时数据区中，线程的内存分配如下：

*   **线程私有（Thread Local）：** 这一部分不需要同步，是线程安全的。
    *   **程序计数器（PC Register）：** 记录当前线程执行到哪一行字节码指令了（为了上下文切换后能恢复）。
    *   **虚拟机栈（JVM Stack）：** 每个方法调用创建一个栈帧，存储局部变量、操作数栈等。
    *   **本地方法栈（Native Method Stack）：** 服务于 Native 方法。
*   **线程共享（Shared）：** 需要同步机制（锁、CAS 等）来保证安全。
    *   **堆（Heap）：** 存储对象实例（`new` 出来的东西）。
    *   **方法区（Method Area）：** 存储类信息、常量、静态变量。

**原理关键点：** 每个线程在 CPU 执行时，会将共享内存中的变量拷贝一份到自己的\*\*工作内存（CPU Cache/寄存器）\*\*中。这就引出了多线程的三大问题：可见性、原子性、有序性。

### 4. 线程调度与上下文切换

由于 Java 使用 1:1 模型，线程调度依赖 OS。

*   **调度策略：**
    *   **抢占式调度（Preemptive）：** 现代 OS（Linux/Windows）的主要策略。系统给每个线程分配时间片，时间到了就强制暂停，切换给下一个线程。
    *   **优先级：** Java 有 `Thread.setPriority()`，但这只是给 OS 一个“建议”。OS 可能会忽略它，或者处理方式不同，所以不能依赖优先级来保证执行顺序。
*   **上下文切换（Context Switch）：**
    *   当一个线程的时间片用完，或者它主动阻塞（IO、sleep、wait），CPU 需要从当前线程切换到另一个线程。
    *   **代价高昂：** CPU 需要保存当前线程的上下文（寄存器状态、PC 指针），加载新线程的上下文。这涉及从\*\*用户态（User Mode）**到**内核态（Kernel Mode）\*\*的切换，会消耗大量的 CPU 周期，并导致 CPU 缓存失效。

### 5. 线程状态（生命周期）

Java 定义了 6 种状态（`Thread.State` 枚举），这与 OS 的状态略有不同：

1.  **NEW：** 线程被创建，但未 start。
2.  **RUNNABLE：** 包含 OS 的 **Ready（就绪）** 和 **Running（运行）**。只要线程在 Java 看来是“活的”且没被阻塞，就是 Runnable。
    *   *注意：* 即使线程在排队等 CPU 时间片，Java 也认为它是 Runnable。
3.  **BLOCKED：** 等待 `synchronized` 锁。
4.  **WAITING：** 无限期等待（`Object.wait()`, `Thread.join()`），需要被唤醒。
5.  **TIMED\_WAITING：** 限时等待（`Thread.sleep(1000)`）。
6.  **TERMINATED：** 执行结束。

### 6. 新时代的变革：虚拟线程 (Virtual Threads / Project Loom)

从 JDK 21 开始，Java 引入了**虚拟线程**（协程），这是对传统 1:1 模型的颠覆。

*   **原理（M\:N 模型）：** 多个虚拟线程（M）映射到少量的操作系统线程（N）。
*   **区别：**
    *   传统线程：阻塞时（如读文件、网络请求），OS 线程被挂起，CPU 资源浪费，且上下文切换重。
    *   虚拟线程：阻塞时，JVM 自动把该虚拟线程卸载（unmount）到堆内存中，底层的 OS 线程（Carrier Thread）立刻去执行其他虚拟线程。
*   **优势：** 可以轻松创建百万级线程，极大地提高了高并发 IO 密集型应用的吞吐量。

### 总结

Java 线程的原理可以用一句话概括：
**Java 线程是对操作系统内核线程的 1:1 封装，它拥有独立的栈空间和程序计数器，通过共享堆内存进行通信，其调度和上下文切换由操作系统内核控制。**
