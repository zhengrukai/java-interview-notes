## 一句话结论

synchronized 的底层是**对象头 Mark Word + Monitor 监视器**，JDK 6 引入锁升级优化：**无锁 → 偏向锁 → 轻量级锁 → 重量级锁**，单向升级不可降级，根据竞争激烈程度自动选择最优策略。

---

### 对象头结构（Mark Word）

Java 对象在内存中的布局：**对象头 + 实例数据 + 对齐填充**

对象头由两部分组成：
- **Mark Word**（32/64 位）：存储锁状态、HashCode、GC 年龄等
- **Klass Pointer**：指向类元数据的指针

**Mark Word 在不同锁状态下的存储内容（64 位 JVM）**：

```
┌──────────────────────────────────────────────────────────┬──────────┬───────┐
│                     Mark Word (64 bit)                    │  锁标志位  │ 状态   │
├──────────────────────────────────────────────────────────┼──────────┼───────┤
│ unused:25 | hashcode:31 | unused:1 | age:4 | biased:0   │   01     │ 无锁   │
│ thread:54 | epoch:2     | unused:1 | age:4 | biased:1   │   01     │ 偏向锁  │
│ ptr_to_lock_record:62                                    │   00     │ 轻量级锁│
│ ptr_to_heavyweight_monitor:62                            │   10     │ 重量级锁│
│ (空)                                                     │   11     │ GC标记 │
└──────────────────────────────────────────────────────────┴──────────┴───────┘
```

**关键点**：锁标志位只有 2 bit，偏向锁和无锁都是 `01`，靠 biased 位区分。

### 锁升级全流程

```
                        只有一个线程访问
    ┌──────┐           ┌──────────┐
    │ 无锁  │ ────────► │  偏向锁   │  Mark Word 记录线程 ID，后续同线程进入零开销
    └──────┘           └─────┬────┘
                             │ 第二个线程尝试获取
                             ▼
                       ┌──────────┐
                       │ 轻量级锁  │  CAS 自旋，不阻塞线程
                       └─────┬────┘
                             │ 自旋超过阈值 / 竞争激烈
                             ▼
                       ┌──────────┐
                       │ 重量级锁  │  OS mutex，线程内核态阻塞
                       └──────────┘
```

---

### 1. 偏向锁（Biased Locking）

**适用场景**：始终只有一个线程访问同步块（实际场景中占大多数）

**工作原理**：
1. 线程 A 首次进入同步块 → CAS 将自己的线程 ID 写入 Mark Word
2. 线程 A 再次进入 → 检查 Mark Word 中线程 ID 是否是自己，是则**直接进入，无需任何同步操作**
3. 线程 B 尝试获取 → 偏向锁撤销，升级为轻量级锁

**偏向锁撤销**：
- 需要等到**全局安全点（Safe Point）**，暂停持有偏向锁的线程
- 检查持有线程是否存活、是否仍在同步块中
- 撤销后恢复为无锁或升级为轻量级锁

**注意**：
- JDK 15 起偏向锁**默认关闭**（`-XX:-UseBiasedLocking`），因为撤销开销在高并发场景不划算
- JVM 启动后有 4 秒延迟才启用偏向锁（`-XX:BiasedLockingStartupDelay=0` 可关闭延迟）

---

### 2. 轻量级锁（Lightweight Lock）

**适用场景**：多个线程**交替执行**同步块，没有真正的竞争

**工作原理**：
1. 线程进入同步块前，JVM 在当前线程栈帧中创建 **Lock Record（锁记录）**
2. 将 Mark Word 拷贝到 Lock Record 中（称为 **Displaced Mark Word**）
3. CAS 尝试将 Mark Word 指向 Lock Record 的指针
4. 成功 → 获取轻量级锁；失败 → 自旋重试

```
  线程栈帧                      对象头
┌──────────────┐          ┌──────────────┐
│ Lock Record  │◄─────────│  Mark Word   │
│ ┌──────────┐ │  CAS替换  │  (ptr: 00)   │
│ │Displaced │ │          └──────────────┘
│ │Mark Word │ │  ← 保存原始 Mark Word，解锁时恢复
│ └──────────┘ │
└──────────────┘
```

**解锁过程**：CAS 将 Displaced Mark Word 替换回对象头，成功则释放；失败说明有竞争，膨胀为重量级锁。

---

### 3. 重量级锁（Heavyweight Lock）

**适用场景**：多线程真实竞争，自旋已无法解决

**工作原理**：
1. JVM 为对象分配一个 **ObjectMonitor**（C++ 实现）
2. Mark Word 指向 ObjectMonitor 的地址
3. 竞争失败的线程进入 **_EntryList**（阻塞队列），由操作系统 mutex 实现阻塞
4. 持有锁的线程调用 `wait()` → 进入 **_WaitSet**（等待队列）

**ObjectMonitor 核心结构**：
```
ObjectMonitor {
    _owner        → 持有锁的线程
    _count        → 重入计数器
    _EntryList    → 阻塞等待锁的线程队列
    _WaitSet      → 调用 wait() 后等待的线程队列
    _recursions   → 重入次数
}
```

**为什么重量级锁开销大？**
- 线程阻塞/唤醒需要 **用户态 ↔ 内核态切换**
- 一次切换约消耗 **1~10 微秒**（相当于几千个 CPU 周期）
- 涉及线程上下文保存/恢复、CPU 缓存失效

---

### 自旋优化

轻量级锁升级为重量级锁前，JVM 会先尝试**自旋**（忙等待），避免立即进入内核态阻塞。

| 自旋类型 | 策略 | 说明 |
|---------|------|------|
| 固定自旋（JDK 6 前） | 默认 10 次 | `-XX:PreBlockSpin` 调整 |
| **自适应自旋**（JDK 6+） | 动态调整 | 上次自旋成功 → 增加次数；失败 → 减少甚至跳过 |

**自适应自旋的智能判断**：
- 如果某个锁对象上自旋经常成功 → JVM 认为下次也大概率成功，允许更长自旋
- 如果某个锁对象上自旋从来没成功 → JVM 直接跳过自旋，省 CPU

### 字节码层面实现

**同步代码块**：通过 `monitorenter` / `monitorexit` 指令实现
```java
synchronized (obj) {
    // 业务代码
}
```
编译后：
```
monitorenter          // 进入同步块，获取 monitor
// 业务代码字节码
monitorexit           // 正常退出，释放 monitor
goto finally_end
monitorexit           // 异常退出，释放 monitor（编译器自动生成，确保不会死锁）
```

**同步方法**：方法标志位增加 `ACC_SYNCHRONIZED`
```java
public synchronized void method() { }
```
JVM 执行时检测到 `ACC_SYNCHRONIZED` 标志，自动在方法入口/出口执行 monitorenter/monitorexit。

### 锁消除与锁粗化（JIT 编译优化）

**锁消除（Lock Elimination）**：JIT 编译器通过**逃逸分析**，发现锁对象不会被其他线程访问，直接消除锁操作。
```java
// StringBuffer.append() 内部有 synchronized，但 sb 是局部变量，不可能逃逸
public String concat(String s1, String s2) {
    StringBuffer sb = new StringBuffer();
    sb.append(s1);  // JIT 会消除 synchronized
    sb.append(s2);  // JIT 会消除 synchronized
    return sb.toString();
}
```

**锁粗化（Lock Coarsening）**：JIT 检测到连续多次对同一对象加锁/解锁，将锁范围扩大，减少加锁次数。
```java
// 优化前：循环内反复加锁解锁
for (int i = 0; i < 100; i++) {
    synchronized (lock) { list.add(i); }
}
// JIT 优化后：锁粗化为一次加锁
synchronized (lock) {
    for (int i = 0; i < 100; i++) { list.add(i); }
}
```

### 完整对比总结

| 维度 | 偏向锁 | 轻量级锁 | 重量级锁 |
|------|-------|---------|---------|
| 适用场景 | 单线程访问 | 多线程交替（低竞争） | 多线程真实竞争 |
| 加锁方式 | CAS 写线程 ID（仅一次） | CAS 替换 Mark Word | OS mutex 阻塞 |
| 解锁方式 | 无需操作 | CAS 恢复 Mark Word | 唤醒 _EntryList |
| 性能 | 最高（零同步开销） | 较高（CAS 自旋） | 最低（内核态切换） |
| 是否阻塞 | 否 | 否（自旋） | 是（内核态阻塞） |

### 高频追问

1. **偏向锁一定比轻量级锁快吗？** 不一定。如果频繁出现偏向锁撤销（多线程交替），撤销本身需要 STW，开销可能更大。所以 JDK 15 默认关闭偏向锁
2. **为什么锁只能升级不能降级？** 降级需要在所有线程间同步状态，开销极大且实现复杂。升级是顺着竞争加剧的方向，自然演进
3. **synchronized 和 volatile 能一起用吗？** 可以但多余。synchronized 已保证可见性和有序性，额外加 volatile 无实际意义（除非有无锁读路径）
4. **如何查看对象的锁状态？** 使用 JOL（Java Object Layout）工具：`ClassLayout.parseInstance(obj).toPrintable()`
5. **为什么 JDK 6 之后 synchronized 性能大幅提升？** 引入了偏向锁、轻量级锁、自适应自旋、锁消除、锁粗化等一系列 JIT 优化
