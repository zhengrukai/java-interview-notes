## 一句话结论

多线程并发数据不一致的根本原因是**原子性、可见性、有序性**三大问题，本质上都源于 JMM（Java 内存模型）中**主内存与工作内存的交互机制**以及**CPU/编译器的优化行为**。

### JMM 内存模型（面试必画图）

```
┌─────────────────────────────────────────────┐
│                   主内存                      │
│          （堆中的共享变量存储区）                 │
│      ┌─────────┐  ┌─────────┐               │
│      │ count=10 │  │ flag=F  │  ...          │
│      └────┬─────┘  └────┬────┘               │
└───────────┼─────────────┼────────────────────┘
       read/load      read/load
            │              │
   ┌────────┴───┐   ┌─────┴──────┐
   │  工作内存A   │   │  工作内存B   │
   │ (线程A的    │   │ (线程B的    │
   │  CPU缓存)   │   │  CPU缓存)   │
   │ count=10   │   │ count=10   │
   └────────────┘   └────────────┘
```

**JMM 8 种内存交互操作**：lock → read → load → use → assign → store → write → unlock

---

### 1. 原子性问题

> 原子性是指一个操作是不可中断的，要么全部执行成功，要么全部执行失败。在多线程环境下，某些操作虽然看起来是单个操作，但实际上可能由多个步骤组成，不是原子操作。

    // 非原子操作
    count++; // 实际上包含三个步骤：读取count值、加1、写回count

    # 当两个线程同时执行count++时，可能会发生以下情况：

    # 线程A读取count值为10
    # 线程B读取count值为10
    # 线程A执行加1，得到11
    # 线程B执行加1，得到11
    # 线程A写回count值为11
    # 线程B写回count值为11
    # 最终count值为11，而不是期望的12。

### 2. 可见性问题

> 可见性是指当一个线程修改了共享变量的值后，其他线程能够立即看到这个修改。在多线程环境下，由于CPU缓存的存在，可能会导致可见性问题。

**原因：**

*   每个线程都有自己的工作内存（CPU缓存）
*   线程操作共享变量时，会先将变量从主内存复制到工作内存
*   修改后，线程会将修改后的值写回主内存
*   但这个写回操作可能不是立即执行的，导致其他线程读取到的还是旧值

### 3. 有序性问题

> 有序性是指程序执行的顺序按照代码的先后顺序执行。在多线程环境下，由于编译器优化、CPU指令重排序等原因，可能会导致有序性问题

    // 线程1
    a = 1; // 操作1
    flag = true; // 操作2

    // 线程2
    while (!flag) {
        // 等待
    }
    int b = a; // 期望b=1
    # 由于指令重排序，操作1和操作2的执行顺序可能被交换，导致线程2读取到a的值为0（初始值）

**指令重排序的三个层次**：
1. **编译器重排序**：编译器在不改变单线程语义的前提下调整指令顺序
2. **CPU 指令级并行重排序**：处理器采用指令级并行技术将多条指令重叠执行
3. **内存系统重排序**：处理器使用写缓冲区，读/写操作可能看起来乱序

---

### 三大问题对照表

| 问题 | 本质原因 | 解决方案 | 关键字/工具 |
|------|---------|---------|-----------|
| 原子性 | 操作被线程切换打断 | 加锁或原子指令 | synchronized、Lock、Atomic 类 |
| 可见性 | CPU 缓存导致读取旧值 | 强制刷新主内存 | volatile、synchronized、Lock |
| 有序性 | 编译器/CPU 指令重排序 | 内存屏障禁止重排 | volatile、Happens-Before 规则 |

### 高频追问

1. **volatile 能保证原子性吗？** 不能。volatile 只保证可见性和有序性，count++ 仍需 Atomic 类或加锁
2. **synchronized 能保证可见性吗？** 能。JMM 规定 unlock 前必须将工作内存刷回主内存，lock 时从主内存重新加载
3. **JMM 和 JVM 内存结构（堆/栈）是一回事吗？** 不是。JMM 是抽象的并发访问规范，JVM 内存结构是运行时数据区的物理划分
4. **什么是 as-if-serial 语义？** 不管怎么重排序，单线程程序的执行结果不能改变（编译器和处理器必须遵守）
