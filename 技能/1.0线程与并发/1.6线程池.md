### 核心架构
##### ThreadPoolExecutor 继承体系
> Executor -> ExecutorService -> AbstractExecutorService -> ThreadPoolExecutor 继承此类
##### 核心组成
线程池内部由三大部分协作：corePool 核心线程、BlockingQueue 任务队列、非核心线程 (按需)。还有RejectedExecutionHandler、ThreadFactory
##### ctl 状态字段
用一个 AtomicInteger ctl 同时存储两个信息：
1. 高3位 → 线程池状态
2. 低29位 → 工作线程数

5个状态：RUNNING → SHUTDOWN → STOP → TIDYING → TERMINATED

### 七大参数
ThreadPoolExecutor 构造参数
```
new ThreadPoolExecutor(
    corePoolSize,       // ① 核心线程数
    maximumPoolSize,    // ② 最大线程数
    keepAliveTime,      // ③ 非核心线程空闲存活时间
    TimeUnit.SECONDS,   // ④ 时间单位
    workQueue,          // ⑤ 任务阻塞队列
    threadFactory,      // ⑥ 线程工厂
    handler             // ⑦ 拒绝策略
);
```

| 参数 | 作用 | 面试关键点 |
| --- | --- | --- |
| corePoolSize | 常驻核心线程数 | 	即使空闲也不销毁（除非 allowCoreThreadTimeOut=true） |
| maximumPoolSize | 线程池允许的最大线程数 | 	队列满后才创建非核心线程；≥ corePoolSize |
| keepAliveTime | 非核心线程空闲存活时长 | 超时后被回收；不影响核心线程 |
| unit | keepAliveTime 的单位 | TimeUnit 枚举：NANOSECONDS ~ DAYS |
| workQueue | 存放待执行任务的队列 | 核心线程满后，先入队；常用3种见下表 |
| threadFactory | 创建线程的工厂 | 可自定义线程名/优先级/daemon；建议用 Guava ThreadFactoryBuilder |
| handler | 队列满且线程达上限时的策略 | 4种内置策略，见"拒绝策略"面板 |

常用工作队列

| 队列类型 | 特点 | 适用场景 |
| --- | --- | --- |
| LinkedBlockingQueue | 无界（默认 Integer.MAX_VALUE） | FixedThreadPool / SingleThreadExecutor 使用，⚠ OOM 风险 |
| SynchronousQueue | 不存储任务，直接移交 | CachedThreadPool，无任务积压 |
| ArrayBlockingQueue | 有界队列 | 生产环境推荐，能控制队列大小 |
| PriorityBlockingQueue | 按优先级排序 | 	任务有优先级要求时 |
| DelayedWorkQueue | 延迟执行 | 	ScheduledThreadPoolExecutor 专用 |

### 执行流程
execute() 任务分配流程：
提交任务 -> 核心线程数未满？
是 -> 直接创建核心线程执行
否↓
工作队列未满？
是 -> 任务加入队列等待
否↓
线程数 < maxPoolSize？
是 -> 创建非核心线程执行
否 -> 触发拒绝策略

##### 线程生命周期
NEW -> RUNNABLE -> BLOCKED -> WAITING -> TERMINATED

### 四种线程池
⚠️ 阿里规范禁止使用 Executors 创建线程池，了解其缺陷是面试重点
1. **FixedThreadPool** ⚠ OOM风险
```
Executors.newFixedThreadPool(n)
// corePoolSize = maximumPoolSize = n
// 队列：LinkedBlockingQueue（无界！）
// ❌ 任务积压可能导致内存溢出
```
适合：负载稳定、任务量可控的场景
2. **CachedThreadPool** ⚠ 线程暴增
```
Executors.newCachedThreadPool()
// corePoolSize = 0，maximumPoolSize = Integer.MAX_VALUE
// 队列：SynchronousQueue（不存储）
// ❌ 任务峰值时线程数量无上限
```
适合：短期异步、任务执行时间短
3. **SingleThreadExecutor** ⚠ OOM风险
```
Executors.newSingleThreadExecutor()
// corePoolSize = maximumPoolSize = 1
// 队列：LinkedBlockingQueue（无界！）
// 保证任务串行执行
```
适合：需要顺序执行任务的场景
4. **ScheduledThreadPool** ⚠ 注意使用
```
Executors.newScheduledThreadPool(n)
// 支持定时/周期性任务
// 队列：DelayedWorkQueue
// scheduleAtFixedRate vs scheduleWithFixedDelay
```
适合：定时调度场景，注意异常会吞掉后续任务
### 拒绝策略
1. AbortPolicy 默认
直接抛出 RejectedExecutionException，调用者需处理异常
2. CallerRunsPolicy 推荐
由调用线程（提交任务的线程）自己执行任务，起到降速反压效果
3. DiscardPolicy
直接丢弃任务，不抛异常，不通知调用者，慎用
4. DiscardOldestPolicy
丢弃队列中最旧的任务，然后重试提交当前任务
##### 自定义拒绝策略（生产推荐）
```
new RejectedExecutionHandler() {
    public void rejectedExecution(Runnable r, ThreadPoolExecutor e) {
        // 1. 记录告警日志
        log.error("Task rejected: {}", r.toString());
        // 2. 降级处理（写 MQ / 持久化）
        fallbackQueue.offer(r);
    }
}
```
### 高频面试题
#### 1. 线程池核心线程数如何设置？
CPU密集型：核心数 + 1（多一个防止偶发暂停）
IO密集型：核心数 × 2 或用公式：线程数 = 核心数 / (1 - 阻塞系数)，阻塞系数一般取 0.8~0.9
最终：以压测数据为准，结合监控动态调整，可用 setCorePoolSize() 热更新
#### 2. 线程池的 shutdown() 和 shutdownNow() 区别？
shutdown()：不再接收新任务，等待队列中任务和正在执行的任务完成后关闭
shutdownNow()：立即中断所有线程，返回未执行的任务列表，通过 interrupt() 通知
生产建议：先 shutdown()，超时未完成再 shutdownNow()
#### 3. 为什么不推荐用 Executors 创建线程池？
① FixedThreadPool / SingleThreadExecutor：使用无界 LinkedBlockingQueue，高并发下任务无限堆积 → OOM
② CachedThreadPool：maximumPoolSize = Integer.MAX_VALUE，可创建大量线程 → CPU飙升 / OOM
应使用 ThreadPoolExecutor 显式指定参数，明确每个参数含义
#### 4. 线程池中线程异常了怎么处理？
execute() 提交：未捕获异常会打印到 stderr，线程被销毁并创建新线程
submit() 提交：异常被封装在 Future 中，需调用 future.get() 才能感知
最佳实践：① 任务内 try-catch；② 自定义 ThreadFactory 设置 UncaughtExceptionHandler；③ 重写 afterExecute()
#### 5. 如何实现线程池的监控？
ThreadPoolExecutor 提供的监控 API：
getPoolSize() 当前线程数 | getActiveCount() 活跃线程数
getQueue().size() 队列积压数 | getCompletedTaskCount() 完成任务数
进阶：重写 beforeExecute/afterExecute/terminated 钩子；接入 Micrometer/Prometheus
#### 6. execute() 和 submit() 的区别？
execute(Runnable)：无返回值，异常直接抛出
submit(Callable/Runnable)：返回 Future，可获取结果和异常
submit 底层也是调用 execute，只是把任务包装成 FutureTask
#### 7. 线程池如何实现优雅关闭？
```
pool.shutdown();
try {
  if (!pool.awaitTermination(60, TimeUnit.SECONDS)) {
    pool.shutdownNow();
    if (!pool.awaitTermination(60, TimeUnit.SECONDS))
      log.error("线程池未能正常关闭");
  }
} catch (InterruptedException e) {
  pool.shutdownNow();
  Thread.currentThread().interrupt();
}
```
### 最佳实践
✅ 生产环境推荐配置
```
int core = Runtime.getRuntime().availableProcessors();

ThreadPoolExecutor pool = new ThreadPoolExecutor(
    core,                            // corePoolSize
    core * 2,                        // maximumPoolSize
    60L, TimeUnit.SECONDS,           // keepAliveTime
    new ArrayBlockingQueue<>(1000),  // 有界队列，防OOM
    new ThreadFactoryBuilder()
        .setNameFormat("biz-pool-%d")
        .setDaemon(false)
        .build(),
    new ThreadPoolExecutor.CallerRunsPolicy() // 反压
);
pool.allowCoreThreadTimeOut(false); // 核心线程不超时
```
动态线程池（美团方案）
```
// 运行时调整参数，无需重启
pool.setCorePoolSize(newCore);
pool.setMaximumPoolSize(newMax);
// 配合配置中心（Apollo/Nacos）
// 实现线程池参数热更新
```
CompletableFuture 配合
```
CompletableFuture
  .supplyAsync(() -> fetchData(), pool)
  .thenApplyAsync(data -> process(data), pool)
  .exceptionally(e -> {
    log.error("error", e);
    return fallback();
  });
```
❌ 常见错误写法

| 错误 | 后果 | 正确做法 |
| --- | --- | --- |
| 使用 Executors.newFixedThreadPool | 无界队列 OOM | 显式指定 ArrayBlockingQueue |
| 线程池定义在方法内 | 每次调用创建新池，资源泄漏 | 设为 static final 单例 |
| 忽略 Future.get() 的异常 | 任务失败无感知 | try-catch ExecutionException |
| 不关闭线程池 | JVM 无法正常退出 | 注册 shutdown hook |
